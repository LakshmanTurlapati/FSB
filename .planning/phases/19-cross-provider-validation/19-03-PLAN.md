Add a message handler in background.js for `CLI_VALIDATION_LIVE_TEST` messages from the options page. This handler delegates to the existing UniversalProvider to make real API calls to AI providers and returns the raw text response for validation.

**In the existing chrome.runtime.onMessage listener in background.js, add a new case:**

```javascript
case 'CLI_VALIDATION_LIVE_TEST': {
  // Live mode: call real AI provider and return raw text response
  const { provider, systemPrompt, userMessage, domSnapshot } = request;

  try {
    // Load current settings to get API keys and model config
    const settings = await chrome.storage.local.get(null);

    // Build the messages array matching the normal AI call format
    const messages = [
      { role: 'system', content: systemPrompt || '' },
      { role: 'user', content: userMessage + (domSnapshot ? '\n\n' + domSnapshot : '') }
    ];

    // Use the existing AI integration to make the call
    // The AIIntegration class is already available in service worker context
    const aiIntegration = new AIIntegration();
    await aiIntegration.initialize();

    // Make a direct provider call (bypass the full automation queue)
    const response = await aiIntegration.provider.call(messages, {
      temperature: 0.3,
      maxTokens: 2000
    });

    // Return raw text response (the content field from provider response)
    const rawText = typeof response === 'string' ? response : (response?.content || response?.choices?.[0]?.message?.content || String(response));

    sendResponse({ success: true, response: rawText, provider });
  } catch (error) {
    sendResponse({ success: false, error: error.message, provider });
  }

  return true; // Async response
}
```

**Important implementation details:**
- The handler must return `true` to indicate async response (sendResponse called later)
- Use the existing AIIntegration class which is already loaded in the service worker via importScripts
- The provider setting determines which API is called (xai, openai, anthropic, gemini)
- For testing a SPECIFIC provider that is not the default, the options page should send a `provider` field. The handler should temporarily configure the provider before calling. If provider switching is complex, simplify: only test the currently configured provider in live mode, and show a message "Configure {provider} as active provider in API Configuration to test it in live mode."
- Add a rate limit guard: minimum 3 second delay between live test calls to avoid hitting provider rate limits
- Log the test call in console with `[CLI Validation]` prefix for debugging
