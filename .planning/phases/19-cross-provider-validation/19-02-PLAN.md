**Step 1: Bundle gpt-tokenizer**

Approach: Use npm to download, then extract the UMD bundle file:
```bash
cd /tmp && npm pack gpt-tokenizer && tar -xzf gpt-tokenizer-*.tgz
```
Then locate the UMD/browser bundle file and copy to lib/gpt-tokenizer.min.js.

If the UMD bundle is unavailable or too large (over 2MB), use this fallback approach:
- Create a lightweight token estimator in lib/gpt-tokenizer.min.js that provides the same API surface
- Use the character-based estimation (chars / 3.5) which is already used in ai-integration.js line 4527
- Label the estimator clearly: `_isEstimation: true` flag on the returned object
- The comparison is CLI vs JSON on the SAME estimator, so relative accuracy is maintained

If the real UMD bundle IS available and under 2MB, use it. It provides `GPTTokenizer_o200k_base` global with:
- `encode(text)` -> token array
- `countTokens(text)` -> number (if available) or use `encode(text).length`

**Step 2: Create utils/token-comparator.js**

Create the TokenComparator module (loaded via script tag in options.html, browser context):

**1. OLD_SYSTEM_PROMPT_TEMPLATE constant:**
Reconstruct the old JSON tool documentation system prompt. Based on the Phase 15 research and Phase 17 changes, the old system prompt contained ~400 lines of JSON tool documentation. Create a realistic approximation:
- Include the old tool documentation header ("You are an AI browser automation assistant...")
- Include 30+ tool definitions in JSON schema format, each with name, description, parameters, example JSON object
- Include the old task completion format: `"taskComplete": true, "result": "summary"`
- Include the old response format instructions: "Respond with a JSON object containing..."
- This does NOT need to be the exact old prompt -- it needs to be a fair representation of the size and structure. The comparison measures format overhead, not exact content.
- Target: ~2000-3000 tokens for the system prompt alone (based on research estimates)

**2. reconstructJSONSnapshot(yamlSnapshotText) function:**
Takes a YAML-format DOM snapshot string and reconstructs what the old JSON-format snapshot would have looked like. The old format was a JSON object with:
```json
{
  "elements": [
    {
      "elementId": "elem_123",
      "type": "button",
      "text": "Submit Form",
      "id": "submit-btn",
      "class": "btn btn-primary",
      "position": {"x": 150, "y": 250, "width": 120, "height": 40, "inViewport": true},
      "attributes": {"data-testid": "submit-button"},
      "visibility": {"display": "block", "visibility": "visible", "opacity": "1"},
      "interactionState": {"disabled": false, "focused": false},
      "selectors": ["#submit-btn", ".btn.submit-btn"],
      "formId": null,
      "labelText": null
    }
  ],
  "htmlContext": {
    "pageStructure": {
      "title": "Page Title",
      "url": "https://...",
      "forms": [],
      "navigation": [],
      "headings": []
    }
  },
  "scrollPosition": {"x": 0, "y": 0},
  "viewport": {"width": 1920, "height": 1080}
}
```

Parse the YAML snapshot line by line:
- Extract metadata (url, title, scroll, viewport) from the header lines
- Parse element lines (format: `eN: tag "text" #id .class [attrs]`) into full JSON objects with all the verbose fields
- Add synthetic position, visibility, interactionState, selectors for each element (these add token overhead in JSON but are absent in YAML)
- Wrap in the full htmlContext structure
- Return as a JSON string with 2-space indentation (the old format was pretty-printed for debugging)

**3. TokenComparator class:**
- `constructor()`: stores tokenizer reference (from window.GPTTokenizer_o200k_base or the estimator fallback), isEstimation flag
- `countTokens(text)`: uses tokenizer.encode(text).length if real tokenizer, or Math.ceil(text.length / 3.5) if estimation. Returns number.
- `async loadCLISystemPrompt()`: reads the REAL current system prompt from ai-integration.js. Since we cannot call the class method from options page, hardcode a reference to the CLI_COMMAND_TABLE and SYSTEM_PROMPT constants from the actual ai-integration.js code. Alternatively, fetch the ai-integration.js file via chrome.runtime.getURL and extract the system prompt string via regex. PREFERRED approach: store a snapshot of the current CLI system prompt as a string constant in this module (CLI_SYSTEM_PROMPT_SNAPSHOT) extracted from the actual ai-integration.js SYSTEM_PROMPT + CLI_COMMAND_TABLE. This is simpler and avoids runtime parsing.
- `async loadJSONBaseline(snapshotName)`: fetches from chrome.runtime.getURL(`test-data/json-baselines/${snapshotName}.json`)
- `compareTokenUsage(cliSystemPrompt, cliDOMSnapshot, cliUserMessage, jsonSystemPrompt, jsonDOMSnapshot, jsonUserMessage)`: counts tokens for each component, calculates total CLI tokens, total JSON tokens, reduction percentage, and per-component breakdown. Returns:
  ```javascript
  {
    cliTokens: { system, dom, user, total },
    jsonTokens: { system, dom, user, total },
    reduction: { system: number, dom: number, user: number, total: number },
    meetsTarget: boolean,  // total reduction >= 0.40
    isEstimation: boolean  // whether using BPE or character estimation
  }
  ```
- `async runComparison(snapshotName, yamlSnapshotText, taskDescription)`: orchestrated comparison for one snapshot:
  1. Load JSON baseline (or reconstruct from YAML if file not found)
  2. Use CLI system prompt snapshot
  3. Build CLI user message: "Task: {taskDescription}\n\n{yamlSnapshotText}"
  4. Build JSON user message: "Task: {taskDescription}\n\n{jsonSnapshot}"
  5. Call compareTokenUsage and return result
- `async runAllComparisons()`: runs comparison for all 6 snapshot types. Returns array of results + aggregate summary with average reduction.

**4. CLI_SYSTEM_PROMPT_SNAPSHOT constant:**
Extract the actual CLI system prompt from ai-integration.js. Read the file to find the SYSTEM_PROMPT constant and CLI_COMMAND_TABLE constant. Combine them into a single string that represents what the AI actually receives as the system prompt. This is the "CLI format" side of the comparison.

**5. Module export:**
```javascript
window.TokenComparator = TokenComparator;
window.OLD_SYSTEM_PROMPT_TEMPLATE = OLD_SYSTEM_PROMPT_TEMPLATE;
```

**Important notes:**
- The comparison MUST include the FULL prompt: system prompt + DOM snapshot + user message (per Pitfall 6 in research)
- The JSON baseline must be a realistic representation -- not minimal (would inflate CLI advantage) and not artificially verbose (would make comparison meaningless)
- The target is 40% AVERAGE reduction across all snapshots, not 40% on every individual one
- Token counts differ by 5-15% across provider tokenizers, but since we compare CLI vs JSON on the SAME tokenizer, the relative difference is accurate
