**Target Game Selection:**
Choose a publicly accessible HTML5 browser game that renders its ENTIRE UI on a canvas element (no DOM buttons). Good candidates:
- itch.io HTML5 games (many render everything on canvas via game engines like Phaser, Construct, Unity WebGL)
- Specific recommendation: Use a well-known, stable itch.io game OR a classic HTML5 game that embeds directly (e.g., a Phaser-based game with a canvas "Play" button on the title screen)
- The game MUST have a clearly visible "Play", "Start", or menu button that is painted on the canvas (not a DOM overlay)
- The game MUST be free, no auth required, and load without complex setup
- If itch.io is chosen, the game page typically has a DOM "Run game" button that launches an iframe containing the canvas game -- document both the DOM launcher click and the subsequent canvas button click

**Site Guide Structure:**
Create `site-guides/games/canvas-game.js` following the registerSiteGuide pattern from google-solitaire.js. The site guide MUST include:

1. **Header comment** explaining this is a canvas-rendered game where ALL UI is painted on canvas (contrast with Google Solitaire which uses DOM elements)

2. **patterns** array: URL patterns matching the chosen game site

3. **guidance** string covering:
   - GAME LAUNCH: How to navigate to the game and start it (including any DOM "Run game" iframe launcher)
   - CANVAS RENDERING: Explicit statement that the game UI (buttons, menus, text) is entirely canvas-rendered with NO clickable DOM elements inside the game frame
   - BUTTON LOCATIONS: Document expected pixel-coordinate regions for common canvas buttons (Play/Start on title screen, menu items). Express coordinates as percentages of canvas dimensions for viewport independence (e.g., "Play button typically centered horizontally at ~50% width, vertically at ~60-70% height")
   - INTERACTION PATTERN: All game interactions require CDP click_at at viewport-relative pixel coordinates. DOM click will NOT work on canvas-painted elements.
   - STATE VERIFICATION: How to verify a button click registered -- look for canvas content changes (DOM won't reflect game state changes), URL hash changes, or iframe postMessage events. Alternatively, use a short wait_for_stable after click_at and re-check the page.
   - COMMON OBSTACLES: Cookie consent popups on the host page, "Run game" iframe launchers, loading screens before the title screen, fullscreen prompts

4. **selectors** object:
   - `gameCanvas`: CSS selector for the main canvas element (e.g., `canvas`, `iframe canvas`, `#game-canvas`, `canvas.game`)
   - `gameIframe`: CSS selector for the game iframe if applicable (e.g., `iframe[src*="game"]`, `iframe.game_drop`)
   - `runGameButton`: DOM selector for the "Run game" / "Play" launcher button on the host page (before the canvas game loads)
   - `fullscreenButton`: DOM selector for fullscreen toggle if present
   - Provide 2-3 fallback selectors per element

5. **workflows** object with these workflows:
   - `launchGame`: Steps to navigate to game URL, dismiss popups, click "Run game" if iframe-hosted, wait for canvas to load
   - `clickCanvasButton`: Steps to identify canvas element bounds, calculate button pixel coordinates from canvas offset + percentage position, execute click_at at those coordinates
   - `verifyButtonClick`: Steps to verify the canvas button click registered (wait for stable, check if canvas content changed by looking for DOM changes around the canvas or URL hash changes)

6. **warnings** array:
   - "Game UI is fully canvas-rendered -- DOM selectors CANNOT target individual buttons, text, or game elements inside the canvas"
   - "Canvas button coordinates are approximate -- games may render at different sizes depending on viewport and iframe dimensions"
   - "Always get canvas element bounding rect from DOM snapshot before calculating click coordinates"
   - "Loading screens may take 3-10 seconds before the title screen with clickable buttons appears"
   - "If game is in an iframe, click_at coordinates are relative to the main viewport, not the iframe -- calculate the iframe offset first"

7. **toolPreferences** array:
   - "ALWAYS use click_at for canvas button interaction -- never use DOM click on canvas-painted elements"
   - "Use get_dom_snapshot to find canvas element position and dimensions before clicking"
   - "Use wait_for_stable after click_at to allow canvas to re-render after button press"
   - "For iframe-hosted games: first click DOM 'Run game' button, then use click_at for in-canvas interactions"
