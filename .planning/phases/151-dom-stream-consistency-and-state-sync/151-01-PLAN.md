<objective>
Prevent the website dashboard preview from drifting onto stale DOM state by giving DOM stream messages a stable stream identity, rejecting stale updates after reconnect/tab switches, and forcing a fresh snapshot when mutation application detects divergence.

Purpose: Phase 150 made transport recovery deterministic. Phase 151 now needs the preview itself to remain correct once messages arrive, especially across reconnects, resumed streams, and streaming-tab transitions.
Output: DOM stream messages include session/snapshot metadata, the dashboard rejects stale updates and requests resyncs, and stale mutation failures fail closed to a fresh snapshot instead of silently freezing the preview.
</objective>

  <name>Task 1: Add stream identity metadata to DOM stream payloads and reject stale dashboard updates</name>
  <files>content/dom-stream.js, background.js, ws/ws-client.js, showcase/js/dashboard.js</files>
  <read_first>content/dom-stream.js (serializeDOM, processMutationBatch, message listener), background.js (domStreamSnapshot/domStreamMutations/domStreamScroll/domStreamOverlay/domStreamDialog cases), showcase/js/dashboard.js (handleDOMSnapshot, handleDOMMutations, handleDOMScroll, handleDOMOverlay, handleDOMDialog)</read_first>
  <action>
Implement a stable stream identity contract for DOM preview updates.

1. In `content/dom-stream.js`, introduce a stream session identifier and snapshot counter that are regenerated on fresh stream starts and resumes. Include `streamSessionId`, `snapshotId`, and `tabUrl`/`viewport` metadata on snapshot payloads and attach `streamSessionId` plus the latest `snapshotId` to mutation, scroll, overlay, and dialog messages.
2. In `background.js`, preserve and forward that metadata on `ext:dom-snapshot`, `ext:dom-mutations`, `ext:dom-scroll`, `ext:dom-overlay`, and `ext:dom-dialog` instead of stripping it down to only the core payload.
3. In `showcase/js/dashboard.js`, track the current preview stream identity from the last accepted snapshot. Ignore `ext:dom-*` messages whose `streamSessionId` or `snapshotId` does not match the active preview state.
4. When a new snapshot from a different stream session or tab is accepted, reset stale overlay/dialog/progress state and update the current preview identity before showing the iframe.
5. Keep the contract backward-compatible: if a message arrives without identity metadata, treat it as legacy only until the next identified snapshot takes over.
  </action>

  <action>
Replace silent stale-DOM freezes with controlled preview resync.

1. In `showcase/js/dashboard.js`, count stale parent/target misses and mutation-apply failures for the current snapshot generation.
2. When a threshold is exceeded, or when a full mutation batch fails at the top level, stop applying further updates for that preview generation and trigger a single resync helper (for example `requestPreviewResync(reason)`).
3. The resync helper should send `dash:request-status` and/or `dash:dom-stream-start`, move the preview back to loading, and record a bounded diagnostic event explaining why the resync was requested.
4. Clear stale counters when a fresh accepted snapshot is rendered successfully so a recovered preview can resume normal incremental updates.
5. Do not spam the extension with repeated resync requests; dedupe repeated requests while a resync is already pending for the current preview generation.
  </action>
