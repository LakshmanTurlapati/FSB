  <name>Task 1: Cache recoverable dashboard task context in the extension and include it in reconnect snapshots</name>
  <files>background.js, ws/ws-client.js</files>
  <read_first>background.js (broadcastDashboardProgress, broadcastDashboardComplete, handleStopAutomation, startDashboardTask), ws/ws-client.js (_sendStateSnapshot)</read_first>
  <action>
Build a recoverable source of truth for the current and most recent dashboard task state.

1. In `background.js`, maintain a small cached dashboard-task snapshot object that is updated on progress, success, failure, and stop paths. Include fields such as `taskStatus`, `task`, `progress`, `phase`, `elapsed`, `action`, `lastAction`, `summary`, `error`, `stopped`, and `updatedAt`.
2. Update the cache from `broadcastDashboardProgress()`, `broadcastDashboardComplete()`, the stop path in `handleStopAutomation()`, and any direct fallback completion send path so reconnect recovery sees the last known truth.
3. In `ws/ws-client.js`, include that cached state in `ext:snapshot` whenever it exists, not only when a running session is currently found in `activeSessions`.
4. Preserve backward compatibility with existing payload fields, but add explicit `taskStatus` so the dashboard can distinguish `running`, `success`, `failed`, `stopped`, and `idle`.
5. Keep the cached object bounded and ephemeral. Do not persist unbounded history; only the current task or last relevant outcome should be recoverable.
  </action>

  </acceptance_criteria>
