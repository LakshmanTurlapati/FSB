## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-90):** MCP server process runs on port 7225 but returns HTTP 426 "Upgrade Required." Same blocker since Phase 55. The closePopupAd 8-step workflow requires: navigate to page with ad overlay, get_dom_snapshot to detect overlay and scan close button candidates by attributes, get_attribute to read close button aria-label/class/id/data-*, click or click_at to dismiss the overlay, get_dom_snapshot again to verify dismissal -- 5+ MCP tool invocations minimum.
- **No computed CSS style reading tool:** There is no MCP tool to read computed CSS properties (opacity, color, font-size, width, height, display, visibility, position) of elements. A `get_computed_style(selector, properties)` tool would enable detection of opacity-based camouflage (close button with opacity:0.1), color-matching camouflage (close button color matching background), tiny hit target camouflage (close button < 10px), and hidden state detection (display:none or visibility:hidden on delayed close buttons). This is a critical gap for DARK-04: DOM attribute analysis can find the close button element, but cannot determine how severely it is visually camouflaged without computed style values.
- **No iframe DOM access tool:** There is no MCP tool to read DOM content inside iframes. A `get_iframe_dom(iframe_selector)` or `get_dom_snapshot(target_iframe)` tool would enable detection of close buttons inside ad iframes -- the most challenging camouflage scenario where the close button is in a separate DOM tree from the parent page. Currently, the only option is coordinate-based click_at on the iframe's bounding rect + typical close button offset.
- **No delayed element appearance detection tool:** There is no MCP tool to wait for a specific element to appear in the DOM after a delay. A `wait_for_element(selector, timeout_ms)` tool would enable handling of close buttons that appear after a setTimeout/CSS animation delay. The current workaround is polling via repeated get_dom_snapshot calls with manual delays.
- **No Escape key simulation gap (covered):** The press_key MCP tool can simulate Escape key press. This fallback strategy is available once the WebSocket bridge is operational.
- **No element bounding rect tool:** There is no MCP tool to get an element's bounding rectangle (x, y, width, height) for coordinate-based click_at targeting. A `get_bounding_rect(selector)` tool would enable precision click_at on close buttons that are too small for CSS selector clicks, and coordinate calculation for iframe close button targeting.
- **WebSocket bridge disconnect (persistent, Phases 55-90):** Same root cause as all previous phases. MCP server runs, returns HTTP 426 for non-WebSocket requests. The Chrome extension WebSocket client cannot establish or maintain the connection. This blocks all live browser interaction testing.

### Recommendations for DOM-Only Close Button Detection Without Visual Perception

1. **aria-label is the single most reliable close button identifier.** BusinessInsider's aria-label="Close this ad" was found with zero ambiguity using `[aria-label*="close" i]`. Always check aria-label first in Tier 1 detection.
2. **Class name patterns provide high coverage.** The pattern `[class*="close"]` matched 6 different close-related elements on BusinessInsider alone (close-icon-wrapper, close-icon-circle, close-icon, close-button, close-button-background, notify-close). Use both `[class*="close"]` and `[class*="dismiss"]` for broad coverage.
3. **Multi-element close button hierarchies are common.** BusinessInsider wraps its close icon in 3 levels (wrapper > circle > SVG). The clickable target may be the outermost wrapper (close-icon-wrapper with aria-label), not the innermost SVG. Always try clicking the element with aria-label first, then walk up/down the DOM tree if the click does not dismiss.
4. **JavaScript-rendered close buttons require repeated DOM scanning.** Forbes and weather.com have zero close buttons in server HTML. After navigating to a page, always wait 3-5 seconds and rescan the DOM, as ad scripts take time to inject overlays and close buttons.
5. **Ad configuration JSON can reveal close button behavior.** BusinessInsider's fenrirClientData config exposed rollUpTimeout:5000 and closeButton:true. If accessible, parse embedded JSON config for close button timing and behavior hints.
6. **SVG close icons confirm visual X without needing vision.** The svg.close-icon class on BusinessInsider confirms the element renders an X shape. Check for SVG elements inside close button containers as Tier 2 confirmation.

## Autopilot Recommendations

1. **Always use get_dom_snapshot to scan for close button DOM attributes (aria-label, class, id, data-*) rather than looking for visible X elements.** The AI has no vision -- even a close button with opacity:0.1 or 4x4px dimensions is present in the DOM with identifiable attributes. On BusinessInsider, aria-label="Close this ad" is the definitive signal despite the close icon being visually minimal.

2. **Apply the 3-tier detection strategy: attributes first, then content/SVG, then positional/fallback.** Tier 1 (aria-label, data-dismiss, data-close, class/id containing "close") should be checked first and is sufficient in most cases. Only expand to Tier 2 (SVG paths, onclick handlers, small positioned elements) and Tier 3 (text content "X" or "Close", opacity < 0.5 elements) when Tier 1 returns no candidates.

3. **Check for decoy close buttons before clicking: verify onclick/href does not point to an external URL.** Before clicking any close button candidate, inspect its onclick handler and href attribute. If it navigates to an external URL, contains window.open(), or has target="_blank", it is a decoy. Click only candidates whose action would close/dismiss/hide the overlay. No decoys were found in the tested targets, but they are common in ad iframe content.

4. **Handle delayed close button appearance: if no close button found, wait 3-10 seconds and rescan.** BusinessInsider's ad config reveals a 5-second rollUpTimeout before the close button appears. After navigating to a page with an ad overlay, if no close button is found on the first DOM scan, wait 3 seconds and rescan. Repeat at 5 seconds and 7 seconds. After 10 seconds with no close button, switch to fallback strategies.

5. **Access iframe DOM content when the close button is inside an ad iframe.** All tested targets inject ad iframes via JavaScript at runtime. If no close button is found in the parent document, check for iframe elements within the overlay container and attempt to read their DOM content via get_dom_snapshot. If iframe DOM is inaccessible, calculate close button coordinates from the iframe's bounding rect (top-right corner, offset 10-15px inset from edge) and use click_at.

6. **Use click_at for extremely small close buttons (< 10px) where regular CSS selector click may miss.** Some close buttons have hit targets as small as 4x4px. For elements with width or height below 10px, use click_at with the element's center coordinates for precision rather than relying on CSS selector click which may target the wrong point on very small elements.

7. **Try Escape key as first fallback before more invasive approaches.** Press Escape (via press_key) as the first fallback when no close button is found or when all close button candidates fail to dismiss the overlay. Many modals and ad overlays listen for the Escape keydown event and close automatically. This is the simplest and least invasive fallback with no side effects.

8. **Use DOM removal (element.remove() or style.display="none") as last resort when no close mechanism works.** If Escape key and backdrop click both fail, identify the overlay container element (class="overlay", data-component-type="overlay", role="dialog") and force-remove it from the DOM. BusinessInsider's overlay has class="overlay" and data-component-type="overlay" -- both reliable selectors for force-removal. This bypasses the intended close mechanism but guarantees the overlay is dismissed.

9. **Track which ad platforms use which camouflage techniques for faster detection.** Google Ad Manager (GAM/DFP/GPT) uses rollUpTimeout for delayed close buttons on sticky ads, with close-button/close-icon class patterns. Ketch CMP uses icon-font close buttons (fs-icon--close). Native ad platforms (Taboola/Outbrain) use small text links. Knowing the ad platform (detectable from script URLs and config JSON) narrows the close button search to platform-specific patterns.

10. **Verify overlay dismissal after clicking -- check that the overlay container is removed or hidden.** After clicking any close button candidate, wait 500ms for animations, then use get_dom_snapshot to verify: (a) the overlay container has display:none or is removed from the DOM, (b) page content behind the overlay is now visible and interactive, (c) no NEW overlay appeared (some ad systems chain multiple overlays). If the overlay is still present, the clicked element may have been non-functional or the wrong target -- try the next candidate.
