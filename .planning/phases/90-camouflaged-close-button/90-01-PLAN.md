<objective>
Create a camouflaged-close.js site guide with a closePopupAd workflow and DARK-04 guidance for finding and clicking close buttons that are visually camouflaged against the pop-up ad background.

Purpose: DARK-04 requires closing a pop-up ad where the X button is camouflaged against the background. This is a "Visual Ambiguity & Dark Patterns" edge case -- pop-up ads deliberately style their close button to be nearly invisible: matching the background color, using very low opacity, making the hit target extremely small (4x4px or less), or positioning the X outside the visible ad boundary. The AI cannot rely on visual recognition since it has no vision -- it must use DOM attribute analysis to find close button candidates: aria-label="Close" or aria-label="Dismiss", role="button" with X/cross Unicode characters, class names containing close/dismiss/exit/shut, SVG paths drawing an X shape, and elements with onclick handlers that call close/dismiss/hide functions. The site guide must document: common pop-up ad platforms and their close button DOM signatures, DOM-based close button detection strategy, fallback approaches (Escape key, clicking overlay backdrop, removing overlay via DOM), and the full closePopupAd workflow.

Output: site-guides/utilities/camouflaged-close.js and background.js updated with importScripts entry

Create a new file site-guides/utilities/camouflaged-close.js. This site guide targets pop-up ad overlays where the close (X) button is deliberately styled to blend with the background for the DARK-04 dark pattern avoidance edge case.

**Pop-up ad targets with camouflaged close buttons:**

Primary approach is DOM-based detection since the AI has no vision. Even when the close button is invisible to a human user, it exists in the DOM with identifiable attributes.

Candidate targets for testing:
1. Sites with interstitial/overlay ads served by Google Ad Manager (GPT/DFP) -- close button is a small SVG X in an iframe with class="close-button" or id="dismiss-button"
2. Sites with newsletter pop-up overlays using common libraries (OptinMonster, Sumo, Privy, Mailchimp embedded) -- close button often has class containing "close" or aria-label="Close"
3. Sites with cookie consent pop-ups that overlay content (distinct from cookie-opt-out.js which handles the consent choice -- this handles the overlay dismissal)
4. Demo/test dark pattern sites: userinyerface.com (has overlay modals), popuptest.com, or similar sites with intentionally hostile close buttons

Real-world platforms known to use camouflaged close buttons:
1. Google Ad Manager interstitial overlays: close button is 15x15px SVG inside a div with opacity:0.3 or color matching ad background
2. Native ad platforms (Taboola, Outbrain): sponsored content widgets with tiny "X" or "close" link styled as 8px light gray text
3. Video pre-roll overlay ads: skip/close button appears after countdown, often transparent until hover
4. Newsletter signup modals: close button is a thin "X" or "No thanks" link with font-size:10px and color:#ccc on white background

4. `guidance`: Multi-line template string with these sections:
   - CAMOUFLAGED CLOSE BUTTON INTELLIGENCE (DARK-04)
   - DARK PATTERN CONTEXT section explaining: Pop-up ads and overlay modals deliberately camouflage their close button to maximize impression time and accidental ad clicks. Common camouflage techniques: (a) matching close button color to the ad background color so the X is invisible (white X on white background, gray X on gray background), (b) setting close button opacity to 0.1-0.3 so it is nearly transparent, (c) making the close button extremely small (4x4px to 8x8px hit target), (d) positioning the close button outside the visual boundary of the ad (above or to the right of the visible frame), (e) delaying the close button appearance by 3-10 seconds after the ad loads (not present in DOM initially), (f) using a countdown timer that must complete before the close button appears, (g) placing a decoy "X" that opens another ad instead of closing, (h) nesting the close button inside an iframe that has different z-index than the ad content. The AI must use DOM attribute analysis since it has no visual perception. Even a close button invisible to humans is present in the DOM with identifiable attributes.
   - DOM-BASED CLOSE BUTTON DETECTION STRATEGY section: The core approach is to scan all elements within the pop-up container and any associated iframes for close button indicators. Detection signals ranked by reliability:
     Tier 1 (high confidence): aria-label containing "close", "dismiss", "shut", or "exit". role="button" with textContent of "X", "x", unicode cross characters (U+2715, U+2716, U+2717, U+00D7, U+2573). data-dismiss, data-close, data-action="close" attributes. id or class containing "close", "dismiss", "exit", "shut".
     Tier 2 (medium confidence): SVG elements with path data drawing an X shape (two diagonal lines from corners). Elements with onclick/click handler containing "close", "dismiss", "hide", "remove". Button or anchor elements that are direct children of the overlay/modal container with very small dimensions (width < 30px, height < 30px). Elements with position:absolute in the top-right corner of the overlay container (typical close button placement).
     Tier 3 (low confidence / fallback): Any element with text content "X", "x", "Close", "No thanks", "Not now", "Maybe later", "Skip". Link elements with href="#" or href="javascript:void(0)" inside the overlay. Elements whose computed style has opacity < 0.5 and is inside the overlay container.
   - IFRAME CLOSE BUTTON DETECTION section: Many ad overlays render the ad content (including its close button) inside an iframe. The close button may be: (a) inside the ad iframe (requires iframe DOM access via CDP), (b) outside the iframe but overlaying it (in the parent document), (c) in a separate small iframe that only contains the close button. Strategy: First scan the parent document for close buttons near the overlay. If not found, use get_dom_snapshot on any iframes within the overlay to check for close buttons inside them. If CDP tools are available, use click_at with coordinates calculated from the iframe's bounding rect + typical close button position (top-right corner, offset 10-15px from edge).
   - FALLBACK DISMISSAL STRATEGIES section: If no close button can be found in the DOM, use these fallback approaches in order:
     Fallback 1: Press Escape key (many overlays listen for keydown Escape)
     Fallback 2: Click the dark backdrop/overlay behind the pop-up (many modals close on backdrop click)
     Fallback 3: Use get_dom_snapshot to find the overlay container element, then set its style.display='none' or remove it from the DOM (bypasses the intended close mechanism)
     Fallback 4: If the overlay has a specific cookie that controls its display (e.g., "popup_shown=1"), set that cookie and reload the page
     Fallback 5: Use click_at with coordinates just outside the overlay boundary (some overlays close when clicking outside)
   - DELAYED CLOSE BUTTON DETECTION section: Some close buttons appear only after a timer. Strategy: If no close button found on first scan, wait 3 seconds and rescan. If still not found, wait up to 10 seconds total, rescanning every 2 seconds. After 10 seconds, switch to fallback strategies. The site guide should document that some ads use setTimeout or CSS animation-delay to show the close button, so the DOM element may exist but have display:none or visibility:hidden until the timer fires.
   - DECOY CLOSE BUTTON DETECTION section: Some ads place a fake "X" that opens another ad instead of closing. Detection: if clicking an X element triggers a navigation or opens a new window instead of dismissing the overlay, it is a decoy. The real close button may be a smaller, less visible element nearby. Strategy: check the onclick handler or href of any X element before clicking. If it points to an external URL, it is a decoy. Look for another close element with an onclick that calls close/hide/dismiss/remove functions.

6. `workflows`: Object with closePopupAd array of 8 steps:
   Step 1: DETECT POP-UP OVERLAY -- Use get_dom_snapshot to identify overlay/modal/popup containers on the page. Look for elements matching overlayContainer selectors. Check for visibility (display not none, opacity > 0). If no overlay detected, the page may not have an active pop-up -- report and exit.
   Step 2: SCAN FOR CLOSE BUTTON BY ATTRIBUTES -- Within the overlay container (and its descendants), search for elements matching closeButtonByAttribute selectors: aria-label containing close/dismiss, data-dismiss/data-close attributes, id/class containing close/dismiss/exit. Record all candidates with their selector, text content, and computed dimensions.
   Step 3: SCAN FOR CLOSE BUTTON BY CONTENT -- If no attribute-based match found, search for elements containing X/cross characters (U+2715, U+2716, U+00D7, "X", "x") or text content "Close", "No thanks", "Dismiss". Check SVG elements for X-shaped paths. Record candidates.
   Step 4: FILTER DECOY CLOSE BUTTONS -- For each candidate close button, check: does its onclick/href point to an external URL? If yes, it is likely a decoy. Does it have target="_blank"? Decoy indicator. Is it inside an ad tracking iframe? Likely decoy. Filter out decoys, keeping only candidates whose action closes/dismisses/hides.
   Step 5: CHECK IFRAME CLOSE BUTTONS -- If no close button found in parent document, scan any iframes within the overlay. Use get_dom_snapshot on iframe contents if accessible. If CDP tools available, calculate close button coordinates from iframe bounding rect (top-right corner, 10-15px inset) and use click_at.
   Step 6: CLICK THE CLOSE BUTTON -- Click the highest-confidence close button candidate. Prefer: aria-label match > data-attribute match > class/id match > text content match > SVG match > coordinate-based click_at. If the element is very small (< 10px), use click_at with the element's center coordinates for precision.
   Step 7: VERIFY DISMISSAL -- After clicking, use get_dom_snapshot to verify the overlay is gone. Check: overlay container has display:none or is removed from DOM, page content behind the overlay is now interactive, no new overlay appeared. If the overlay is still present, the clicked element may have been a decoy -- go back to Step 4 and try the next candidate. If all candidates exhausted, move to fallback strategies.
   Step 8: FALLBACK AND REPORT -- If no close button succeeded: try Escape key, backdrop click, DOM removal in order. Report final outcome: which close button candidates were found, which was clicked, whether dismissal was verified, whether fallback was needed.
