## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-67):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55-66 and now Phase 67. Without it, no tool (DOM or CDP based) can execute against the live browser. This is the primary blocker for all live MCP testing in this milestone.
- **SPA-rendered content inaccessible via HTTP:** X/Twitter's fully client-side React architecture means get_dom_snapshot and read_page can only work in a live browser with JavaScript execution. Unlike Buzzsprout (Phase 66) which had server-rendered HTML elements, X/Twitter provides zero server-side tweet content. This makes HTTP-based validation impossible for X/Twitter.
- **count_elements tool (proposed):** A tool that runs `document.querySelectorAll(selector).length` and returns the count without the full DOM snapshot overhead would be valuable for the scroll loop. Instead of calling get_dom_snapshot on every scroll (returning the full element tree), calling `count_elements('[data-testid="tweet"]')` would quickly check whether new tweets have loaded. This would reduce per-scroll overhead from full snapshot serialization to a single integer response.
- **scroll_until tool (proposed):** A tool that scrolls until a condition is met (e.g., N elements of selector X found) would encapsulate the entire scroll loop in a single tool call. Parameters: `scroll_until(selector, targetCount, maxScrolls, scrollAmount, waitMs)`. This would handle: repeated scroll, wait, count cycles; permalink deduplication internally; breaking when targetCount reached or maxScrolls exceeded. Would dramatically simplify the autopilot prompt for infinite scroll tasks.
- **extract_elements tool (proposed):** A tool that runs `querySelectorAll(selector)` and returns specified attributes for all matching elements (e.g., text content and href for each tweet). This is lighter than get_dom_snapshot (which returns all page elements) and more targeted than read_page (which returns all text). Would enable efficient per-scroll tweet extraction: `extract_elements('[data-testid="tweet"]', ['tweetText.textContent', 'a[href*="/status/"].href'])`.
- **Auth wall detection during scroll (gap):** No tool currently detects whether an auth overlay has appeared during scrolling. The scroll tool would continue scrolling even if a login modal covers the timeline. A `detect_overlay` or `check_for_modal` tool that checks for fixed/sticky positioned elements with high z-index covering the primary content area would help detect mid-scroll auth walls.
- **No get_bounding_rect tool (carried from Phases 57-66):** While less critical for scroll-and-count than for click_at tasks, knowing the exact viewport position of tweet elements would help verify whether tweets are actually in-view after scrolling.

## Autopilot Recommendations

1. **Use public profile pages, not the home feed, for unauthenticated tweet access:** The X/Twitter home feed (x.com/home) requires authentication. Public profile pages (x.com/NASA, x.com/nytimes, x.com/BBCWorld) return HTTP 200 without redirect and render the user's tweet timeline in the live browser DOM. Prioritize high-volume accounts with 10K+ tweets to ensure enough content for the 150-post target. NASA (70K+ tweets), nytimes, and BBCWorld are reliable targets.

2. **Detect auth walls before and during scroll:** X/Twitter may show login overlays at various points: (a) immediately on page load for some browsers/IPs, (b) after scrolling past N tweets, (c) after N minutes of browsing. Detect auth walls by checking for `[data-testid="loginButton"]` visibility, redirect to `x.com/i/flow/login`, or fixed-position overlays covering `[data-testid="primaryColumn"]`. If auth wall detected, classify as SKIP-AUTH and document at which point (tweet count, scroll count) the wall appeared.

3. **Handle virtualized DOM recycling with permalink-based deduplication:** X/Twitter only keeps ~20-40 tweet elements in the DOM at any time. As the user scrolls down, earlier tweets are removed and new ones are added. Do NOT rely on element count in a single snapshot for total post count. Instead: maintain a Set of seen permalink hrefs (`a[href*="/status/"]` within each `[data-testid="tweet"]`). On each snapshot after scrolling, only add tweets whose permalink is NOT already in the Set. The Set size equals the true unique post count.

4. **Scroll at 800px with 500ms-1000ms wait between iterations:** X/Twitter lazy-loads tweets as the user scrolls. Scroll amount of 800px (~1 viewport height) triggers the virtual scroll to load new tweets while removing offscreen ones. After each scroll, use wait_for_stable (DOM quiescence for 300-500ms) to let new tweets render before capturing a snapshot. Too-fast scrolling may outpace the content loading, resulting in empty snapshots.

5. **Filter ads and suggestion modules from tweet counts:** Not every `[data-testid="cellInnerDiv"]` in the timeline is a tweet. Filter out: (a) promoted posts containing `[data-testid="placementTracking"]`, (b) "Who to follow" suggestion modules (no `[data-testid="tweet"]` child), (c) "Subscribe to Premium" banners, (d) topic suggestion cards. Only count elements where `[data-testid="tweet"]` exists as a child of the `cellInnerDiv`.

6. **Expect 10-15 new unique tweets per scroll iteration:** Based on the virtualized DOM architecture and ~800px scroll amount, each scroll cycle typically reveals 10-15 new tweets. To reach 150 unique tweets: expect approximately 10-15 scroll iterations after the initial snapshot (~25 tweets). Budget for 20 scrolls maximum with a break condition of 3 consecutive scrolls yielding zero new tweets.

7. **Break scroll loop on 3 consecutive empty scrolls:** If get_dom_snapshot returns no new permalinks (all already in seenPermalinks Set) for 3 consecutive scroll iterations, the feed has likely ended or an auth wall has appeared. Do not continue scrolling indefinitely. Check for auth wall indicators and classify the outcome based on how many unique posts were counted.

8. **Extract tweet text during scroll, not as a separate pass:** Capture `[data-testid="tweetText"]` content for each tweet during the scroll loop, not just the permalink. Store the first 200 characters of text per tweet in the postList array. This avoids needing to scroll back to a specific tweet later (impossible with virtualized DOM -- the 150th tweet element will have been recycled out of the DOM by the time you finish scrolling). The text captured during the scroll is the only text you will have.

9. **Manage memory for 150+ tweet entries with truncated text:** Each entry in postList needs only: {index, permalink, textSnippet}. Limit textSnippet to 200 characters per tweet to keep memory under ~50KB for 150 entries. Do not store full DOM element data or full tweet HTML. The postList is the authoritative record -- the DOM will not contain these elements later.

10. **Retry strategy for stalled tweet loading:** If a scroll iteration returns tweets but fewer than expected (e.g., only 2-3 new tweets instead of 10-15): (a) wait an additional 1000ms for slow-loading content, (b) try a smaller scroll amount (400px) to give the virtualizer time to catch up, (c) try scrolling up 200px then down 1000px to force a re-render cycle. If tweets consistently stop loading at a specific count (e.g., always stops at ~50 tweets), this may indicate a rate limit or auth gate -- document the specific threshold.
