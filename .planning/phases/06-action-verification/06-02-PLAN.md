<task type="auto">
  <name>Task 1: Add verification to tools.type with alternative selector support</name>
  <files>content.js</files>
  <action>
Modify tools.type to:

1. Accept `params.selectors` array in addition to `params.selector` (backward compatible)
2. Loop through selectors, trying each until one succeeds with verified effect
3. For each selector attempt:
   - Find element with querySelectorWithShadow
   - Check readiness with ensureElementReady (existing pattern)
   - Capture pre-state with captureActionState(element, 'type')
   - Execute the typing action (existing implementation)
   - Call `await waitForPageStability({ maxWait: 1000, stableTime: 200 })` for DOM/network to settle
   - Capture post-state with captureActionState(element, 'type')
   - Verify effect with verifyActionEffect(preState, postState, 'type')
   - If verified, return success with verification details
   - If not verified, log and try next selector

4. On success, return:
```javascript
{
  success: true,
  typed: params.text,
  selector: usedSelector,
  selectorIndex: i,
  usedFallback: i > 0,
  hadEffect: true,
  verification: { changes, ... }
}
```

5. On all selectors exhausted without effect:
```javascript
{
  success: false,
  error: 'Type action had no effect with any available selector',
  hadEffect: false,
  selectorsTriad: selectors.length,
  suggestion: 'Input may be readonly, disabled, or requires focus first'
}
```

IMPORTANT: Keep the existing ensureElementReady check and staleness handling after scroll.

<task type="auto">
  <name>Task 3: Add verification to tools.pressEnter</name>
  <files>content.js</files>
  <action>
Modify tools.pressEnter to add verification:

1. Accept params.selectors array for fallback options
2. Before pressing Enter:
   - Capture pre-state with captureActionState(element, 'pressEnter')
3. After pressing Enter:
   - Call `await waitForPageStability({ maxWait: 2000, stableTime: 300 })` (longer wait - Enter often triggers navigation or async operations)
   - Capture post-state
   - Verify with verifyActionEffect - expects anyOf: urlChanged, elementCountChanged, contentChanged, focusChanged
4. pressEnter is more lenient - many valid uses have no immediate effect (e.g., pressing Enter in a text area to add newline)
5. Only report failure if element was inside a form and nothing changed (form submission expected but failed)

Return format on success:
```javascript
{
  success: true,
  selector: usedSelector,
  hadEffect: verification.verified,
  verification: { changes, ... }
}
```

On form submission with no effect:
```javascript
{
  success: false,
  error: 'Enter key pressed but form submission had no effect',
  hadEffect: false,
  suggestion: 'Form may have validation errors or require button click instead'
}
```

For non-form contexts, return success even without observable effect (Enter in textarea is valid).
