```javascript
fail_task -> emits final failure state, then clears same token deterministically

// token omitted
preserve current hadEffect:false narration/final semantics
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Shared task-status tools -> autopilot callers | New token-aware behavior must not break the long-standing no-effect narration/final semantics used by the agent loop. |
| Session token -> visible surface ownership | A stale or guessed token must never clear or overwrite a newer client-owned session. |
| Final-state UI -> cleanup | Final overlay states must clear deterministically so the glow never appears stuck after success/partial/failure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-203-02-01 | Tampering | Shared task-status handlers | mitigate | Branch on `session_token`; omitted token preserves existing behavior, matching token enables client-owned lifecycle, stale token fails safely. |
| T-203-02-02 | Denial of Service | Final-state cleanup | mitigate | Emit deterministic clear after final freeze and remove the client-owned session record only after same-token cleanup. |
| T-203-02-03 | Spoofing | Stale token clears | mitigate | Use stored token + version matching and ignore mismatched clears or updates. |
| T-203-02-04 | Regression | Agent loop narration | mitigate | Keep `report_progress` without token as `hadEffect: false` and lock that behavior in no-regression tests. |
</threat_model>
