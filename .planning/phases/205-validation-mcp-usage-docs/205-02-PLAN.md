<interfaces>
The reader-facing lifecycle should stay simple and canonical:
```text
1. start_visual_session(client_label, task, optional tab hint)
2. report_progress(session_token, task/detail/progress_percent as work advances)
3. complete_task | partial_task | fail_task(session_token, outcome details)
4. end_visual_session(session_token) when explicit cleanup is still needed
```

Decision boundary that docs must make obvious:
```text
Use explicit visual-session tools when your MCP client already owns the steps
and only wants FSB to show trusted progress in the browser.

Use run_task when you want FSB autopilot to decide and execute the browser steps.
```
</interfaces>
</context>
