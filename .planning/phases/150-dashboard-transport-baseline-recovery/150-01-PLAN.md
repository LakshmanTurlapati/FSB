  <name>Task 1: Define an explicit extension recovery contract for stream readiness and reconnect state</name>
  <files>ws/ws-client.js, background.js</files>
  <read_first>.planning/ROADMAP.md (Phase 150 section), .planning/REQUIREMENTS.md (STRM-01 and STRM-02), ws/ws-client.js (_sendStateSnapshot, _handleMessage, _forwardToContentScript), background.js (_streamingTabId, _streamingActive, _handleStreamTabSwitch, domStreamReady)</read_first>
  <action>
Add one explicit extension-to-dashboard transport message for readiness recovery and make reconnect snapshots carry stream intent.

1. In `ws/ws-client.js`, add a new outbound message type named `ext:stream-state` with payload keys `{ status, reason, streamIntentActive, tabId, url, source }`. Use concrete `status` values `ready`, `not-ready`, and `recovering`. Use concrete `reason` values `restricted-tab`, `no-streamable-tab`, `waiting-for-page-ready`, and `tab-closed` when those conditions occur.
2. Extend `_sendStateSnapshot()` so `ext:snapshot` includes stream recovery fields alongside task fields: `streamIntentActive`, `streamTabId`, `streamTabUrl`, `streamStatus`, `streamReason`, and `snapshotSource`. Keep `taskRunning`, `task`, `progress`, `phase`, and `elapsed`.
3. When `connect()` opens the extension WebSocket, `_sendStateSnapshot()` must always emit either `ext:page-ready` for a usable tab or `ext:stream-state` with `status: 'not-ready'` and a concrete `reason`; do not rely on missing messages as the not-ready signal.
4. In `_handleMessage('dash:dom-stream-start')`, resolve a candidate tab using the existing `_streamingTabId -> active lastFocusedWindow -> any active tab` chain before forwarding. If the chosen tab is restricted or missing, emit `ext:stream-state` with `status: 'not-ready'` instead of only calling `_forwardToContentScript('domStreamStart')`.
5. In `background.js`, update `_handleStreamTabSwitch()` and `domStreamReady` so both paths emit the same ready/not-ready contract. `_handleStreamTabSwitch()` must send `ext:stream-state` for restricted or closed tabs, and `domStreamReady` must send `ext:stream-state` with `status: 'ready'` before or alongside `ext:page-ready`.
6. Preserve `_streamingActive` as the dashboard's stream intent flag. `dash:dom-stream-start` sets it `true`, `dash:dom-stream-stop` sets it `false`, and reconnect snapshots must reflect that flag even when `_streamingTabId` needs to be recomputed.
  </action>

- `rg -n "restricted-tab|no-streamable-tab|waiting-for-page-ready|tab-closed" ws/ws-client.js background.js` returns the explicit not-ready reasons.
- `rg -n "dash:dom-stream-start" ws/ws-client.js` shows candidate-tab validation happening before `_forwardToContentScript('domStreamStart'`.

<task type="auto">
  <name>Task 2: Make the dashboard preview state machine consume recovered stream intent and retry deterministically</name>
  <files>showcase/js/dashboard.js</files>
  <read_first>.planning/ROADMAP.md (Phase 150 section), .planning/REQUIREMENTS.md (STRM-01 and STRM-02), showcase/js/dashboard.js (setPreviewState, connectWS, handleWSMessage, ext:snapshot handler, ext:page-ready handler, ext:stream-tab-info handler)</read_first>
  <action>
Replace the dashboard's blind loading behavior with an explicit recovery state machine driven by `ext:snapshot` and `ext:stream-state`.

1. In `showcase/js/dashboard.js`, add recovery bookkeeping with concrete names `previewLoadStartedAt`, `previewNotReadyReason`, `lastRecoveredStreamState`, `pendingStreamRecovery`, and `scheduleStreamRecovery()`.
2. On `ws.onopen`, keep sending `dash:request-status`. Only send `dash:dom-stream-start` when `streamToggleOn` is true, and immediately start a 5-second recovery watchdog via `scheduleStreamRecovery('ws-open')` instead of assuming the preview should remain in `loading` forever.
3. Update `handleWSMessage('ext:snapshot')` to consume `streamIntentActive`, `streamStatus`, `streamReason`, and `streamTabUrl`. If the snapshot says `not-ready`, transition to `disconnected` with concrete dashboard text based on `previewNotReadyReason`. If the snapshot says `ready` or `recovering`, keep `loading` until `ext:dom-snapshot` arrives.
4. Add a dedicated `handleWSMessage('ext:stream-state')` branch. `status: 'ready'` sets `pageReady = true` and schedules or maintains loading, `status: 'recovering'` keeps loading with recovery text, and `status: 'not-ready'` clears `pageReady`, records `previewNotReadyReason`, and moves the preview to explicit disconnected/not-ready UI without manual refresh.
5. When `ext:status` flips from offline to online or when `ws.onclose` followed by reconnect occurs, reassert stream intent exactly once by calling `scheduleStreamRecovery('extension-online')`, which resends both `dash:request-status` and `dash:dom-stream-start` if `streamToggleOn` is still true.
6. Keep existing `ext:page-ready` and `ext:dom-snapshot` behavior, but make `ext:page-ready` update `lastRecoveredStreamState = 'ready'` and clear any not-ready text so a later real page can move from not-ready back to loading without a manual refresh.
  </action>
