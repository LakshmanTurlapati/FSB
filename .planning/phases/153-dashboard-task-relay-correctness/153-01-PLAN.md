<objective>
Make dashboard task submission and progress delivery identify one concrete task run end to end, so the website dashboard can tell accepted work from stale or cross-run task updates.

Purpose: The current task relay depends mostly on timestamps and a locally optimistic running state. Phase 153 needs an explicit task-run contract plus normalized running-state payloads to keep task progress coherent across reconnects and retries.
Output: Task submit/progress/reconnect payloads carry a stable `taskRunId`, accepted runs publish an immediate normalized running payload, and the dashboard tracks the active run explicitly.
</objective>
