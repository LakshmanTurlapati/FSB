**Step 2: Research Google Solitaire DOM structure**
Use MCP `navigate` to open `https://www.google.com/search?q=solitaire`. Wait for the page to load with `wait_for_stable`.

Use `get_dom_snapshot` to inspect the Google Solitaire game widget. Discover:
- Whether the game is rendered as DOM elements (divs with background-image cards) or a canvas element
- The game container element (likely an iframe or a div with a specific data attribute)
- Card elements -- how individual cards are represented (look for elements with card suit/rank data attributes, aria-labels, or class names)
- Pile/stack regions -- tableau columns, foundation piles, stock pile, waste pile
- The "Play" or "Start" button to launch the game
- Any difficulty selector (easy vs hard mode)
- Undo button, new game button, and other game controls

**Step 3: Discover card interaction patterns**
After inspecting the DOM:
- If DOM-based: Cards will be clickable div/img elements. Test with `click` on a card to see if it auto-moves to a valid target, or if drag is required.
- If canvas-based: Cards require `click_at` or `drag` at computed viewport coordinates based on card layout grid positions.
- Check if double-click on a card auto-sends it to the foundation (common solitaire behavior).
- Check if click-to-select then click-target is supported (select source, click destination).
- Check if the game requires drag-and-drop (mousedown on card, mousemove to target, mouseup).
