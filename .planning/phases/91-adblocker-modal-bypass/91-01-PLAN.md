Create a new file site-guides/utilities/adblocker-bypass.js. This site guide targets adblocker detection modals that deliberately have NO close button, forcing the user to disable their adblocker or leave. This is the DARK-05 dark pattern avoidance edge case.

**Distinction from DARK-04 (camouflaged close):** DARK-04 has a close button that is visually hidden but present in the DOM. DARK-05 has NO close button at all -- the modal is designed to be completely inescapable through normal UI interaction. The bypass must use DOM manipulation to remove the blocking elements.

**Adblocker detection techniques documented in the guide:**

1. **Bait element technique:** Site inserts a div with class="ad-banner" or id="ad-container" (bait element), then checks if it was hidden/removed by the adblocker. If hidden, the detection script triggers the modal.
2. **Script load test:** Site loads a script from a known ad domain (e.g., pagead2.googlesyndication.com/pagead/js/adsbygoogle.js). If the script fails to load (blocked by adblocker), detection triggers.
3. **Ad server request check:** Site makes a fetch/XHR to an ad server endpoint. If the request fails (blocked), detection triggers.
4. **DOM measurement check:** Site creates an ad-like element and checks its offsetHeight/clientHeight. If 0 (hidden by adblocker), detection triggers.

**Common adblocker detection libraries and their modal DOM patterns:**

1. **BlockAdBlock (blockadblock.com):** Creates modal with id="blockadblock" or class="blockadblock", body class="blockadblock". Modal is a full-screen overlay div. No close button.
2. **FuckAdBlock / detectAdBlock:** Similar to BlockAdBlock. Creates overlay element, adds class to body or html element. Detection variable: fuckAdBlock, detectAdBlock.
3. **Admiral (getadmiral.com):** Enterprise-level adblock recovery. Injects iframe or shadow DOM container. Modal may appear in a shadow root or cross-origin iframe making direct DOM access harder.
4. **Custom bait-div implementations:** Most common approach. Sites write their own detection: create a bait div, wait for adblocker to hide it, then show a custom modal. Modal structure varies by site but typically: full-screen overlay div (position:fixed, z-index:9999+) with a message asking to disable adblocker.

Unlike DARK-04 (camouflaged close button) where a hidden close element exists in the DOM, adblocker modals deliberately omit any close mechanism. The only intended user options are: (1) disable the adblocker and reload, (2) whitelist the site in the adblocker, or (3) leave. The modal typically: (a) covers the entire viewport with a high z-index overlay (position:fixed, top:0, left:0, width:100%, height:100%, z-index:9999+), (b) locks body scroll (overflow:hidden on body/html), (c) blurs or dims the page content behind the overlay (filter:blur, opacity reduction, pointer-events:none on content), (d) may disable keyboard interaction (tabbing, Escape), (e) may continuously re-inject the modal if removed (observer-based re-detection).      Step A: IDENTIFY THE MODAL -- Use get_dom_snapshot to find the overlay element. The modal container is typically a direct child of body or injected at document end.
     Step B: REMOVE THE MODAL OVERLAY -- Remove the modal element from the DOM (element.remove()) or set display:none. If the modal is inside a shadow DOM or iframe, direct removal may not work; use CSS override instead.
     Step C: REMOVE THE BACKDROP -- Many modals add a separate backdrop div (position:fixed, background:rgba(0,0,0,0.5), z-index just below the modal). Find and remove it. Look for: class containing "backdrop", "overlay-bg", "mask", "dimmer", or sibling of the modal element with lower z-index.
     Step D: RESTORE BODY SCROLL -- The modal script typically adds overflow:hidden to body and/or html elements to prevent scrolling. Remove this: body.style.overflow='auto', html.style.overflow='auto'. Also check for position:fixed on body (mobile scroll lock pattern) and restore to position:static.
     Step E: REMOVE CONTENT BLOCKING -- The modal script may add CSS to the main content to prevent interaction: filter:blur(5px), opacity:0.3, pointer-events:none, user-select:none. Find the main content container and remove these styles. Look for: main, article, #content, .content, .main, or the first significant child of body that is not the modal.
     Step F: NEUTRALIZE RE-DETECTION -- Some sites use MutationObserver to re-inject the modal if removed. Strategy: after removing the modal, check if it reappears within 2-3 seconds.      Fallback 4: Use Reader Mode if the browser supports it -- extracts article text without running page scripts
     Fallback 5: If the page content loaded behind the modal, scroll the body element directly via JavaScript (window.scrollTo) and extract text from the DOM elements that are behind the overlay
