# Phase 162.1: Partial Completion Lifecycle - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning
**Source:** Verification gap evidence + codebase inspection

<domain>
## Phase Boundary

Introduce a first-class terminal outcome for sessions that completed useful work but cannot finish the final requested step because of an external blocker. This phase is about the lifecycle contract and the persisted result model. It must distinguish partial/blocked completion from both full success and hard error. It does not yet teach the agent auth-wall-specific behavior in detail; that follow-on guidance belongs to Phase 162.2.

</domain>

<decisions>
## Implementation Decisions

### Core outcome model
- **D-01:** A useful-but-blocked run must not be recorded as `error`. The runtime needs a first-class partial/blocked terminal outcome distinct from completed, stopped, and error.

### Structured preserved result
- **D-02:** Claude's Discretion. The terminal partial outcome should preserve structured information, not only a flat string. At minimum the model must carry what was completed and what blocker prevented full completion so downstream surfaces can render it without treating it like a crash.

### Agent-facing lifecycle tooling
- **D-03:** Claude's Discretion. The agent loop currently exposes only `complete_task` and `fail_task`. Phase 162.1 should introduce the generic lifecycle mechanism the agent can call for partial/blocked completion, while Phase 162.2 will handle auth-wall-specific prompt guidance and handoff wording.

### Surface propagation
- **D-04:** Popup, sidepanel, and MCP session history must render partial outcomes as non-crash results. Existing partial-result chat styling in popup/sidepanel should be reused where possible rather than redesigned.

### Compatibility guardrails
- **D-05:** Preserve the existing setTimeout-chained agent loop and the current `automationComplete` transport path. Extend the payload/result schema rather than inventing a second terminal event channel.

### Existing partial semantics
- **D-06:** Today `partial: true` is already used for user-stopped sessions in `automationComplete`. Phase 162.1 must not conflate "stopped by user" with "task partially completed due to blocker"; if one schema covers both, it needs an explicit outcome/reason field that preserves the distinction.

### Scope guardrails
- **D-07:** Do not bundle auth-wall-specific prompt examples, LinkedIn-specific handling, or unrelated UI redesign into this phase. The goal is the reusable lifecycle contract and non-crash result persistence.

### Claude's Discretion
- Exact naming of the new lifecycle tool and/or terminal status values
- Whether the canonical persisted shape lives on new session fields, a nested outcome object, or both
- Which existing surfaces beyond popup/sidepanel/MCP history should be updated for consistency if they already read the same persisted session index

</decisions>

<code_context>
## Existing Code Insights

### Confirmed behavior
- `ai/tool-definitions.js` exposes only two terminal task lifecycle tools: `complete_task` and `fail_task`.
- `ai/agent-loop.js` system prompt explicitly instructs the model to always end with `complete_task` or `fail_task`.
- `ai/agent-loop.js` handles `complete_task` by setting `session.status = 'completed'`, but handles `fail_task` by setting `session.status = 'error'` and finalizing through the same terminal path used for real failures.
- `ai/agent-loop.js` currently sends `automationComplete` with `partial: isError`, which means the transport conflates blocked partial completion with true error.
- `background.js` `executeAutomationTask()` completion resolution converts `message.partial` into `success: false` and an error string, so partial outcomes are treated as failed executions in dashboard/MCP result consumers.
- `utils/automation-logger.js` persists only coarse session fields such as `status`, `task`, `actionCount`, `logs`, and successful actions; it does not preserve a structured partial outcome object today.
- `ui/popup.js` and `ui/sidepanel.js` already have a visual partial-result path for live completion bubbles, but their persisted history views currently just display raw `session.status`.
- `ai/session-schema.js` allows `completed`, `failed`, `stopped`, and `error`, but no partial/blocked terminal status.

### Architectural implication
- The UI already knows how to show a partial completion message, but the runtime and persistence layers still classify blocked-but-useful sessions as failures.
- Fixing only the UI would be insufficient because MCP history and dashboard completion results would continue to misclassify the run as an error.

### Likely integration points
- Add a new terminal lifecycle tool and/or explicit outcome payload in `ai/tool-definitions.js` and the agent-loop system prompt
- Extend session schema plus terminal result/finalization helpers in `ai/agent-loop.js`
- Update `background.js` completion resolution so partial outcomes are non-crash results instead of `success: false`
- Persist outcome metadata in `utils/automation-logger.js` and expose it through MCP/session history consumers
- Reuse existing partial-result rendering in popup/sidepanel while making status/history views outcome-aware

</code_context>

<specifics>
## Specific Ideas

- One plan should focus on the core lifecycle contract: task tool, session status/result schema, agent-loop finalization, and executeAutomationTask result semantics.
- A second plan should focus on persistence and rendering: automation logger/session index, MCP-visible history/detail fields, and popup/sidepanel history/live result presentation.
- If options/history views already read the same persisted session index, updating them in the second plan is acceptable as consistency work, even though the explicit requirement names popup, sidepanel, and MCP history.

</specifics>

## Deferred Ideas

- Auth-wall-specific prompt rules and manual-handoff wording for LinkedIn or other login-gated sites
- Any new UI styling beyond reusing the existing partial-result affordance
- Broader terminal outcome taxonomy redesign beyond what is needed for partial/blocked lifecycle support

</deferred>
