<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local process -> hub port | Any local process can attempt to connect as extension or relay on the bridge port. |
| Relay MCP host -> hub | Relay processes send requests through the hub and receive topology state from the hub. |
| Bridge state -> diagnostics/HTTP | Topology state is exposed through CLI status and HTTP health. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-198-10 | Spoofing | `relay:hello` classification | mitigate | Keep explicit `relay:hello` classification, require `instanceId`, and do not classify relay handshake as extension readiness. |
| T-198-11 | Tampering | relay extension readiness | mitigate | Hub broadcasts authoritative `relay:state`; relays set `isConnected` from `extensionConnected`, not from `relay:welcome` alone. |
| T-198-12 | Denial of Service | hub-exit promotion | mitigate | Preserve randomized promotion jitter, configurable short test jitter, reconnect backoff capped at `maxReconnectDelayMs`, and timer cleanup on `disconnect()`. |
| T-198-13 | Information Disclosure | diagnostics and HTTP health | mitigate | Expose only topology metadata: mode, booleans, counts, instance IDs, timestamps, and disconnect reason. Do not expose MCP payloads, tab URLs, credentials, card data, or API keys. |
| T-198-14 | Repudiation | bridge disconnects | mitigate | Track `lastDisconnectReason` and `lastExtensionHeartbeatAt` in live topology state so status output can explain current state. |
</threat_model>
