<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local MCP host -> extension WebSocket | A local process can connect to the extension bridge endpoint and trigger reconnect behavior. |
| Service-worker event -> bridge client | Browser events wake the MV3 worker and can re-arm local WebSocket connection attempts. |
| Bridge lifecycle -> Chrome storage | Non-secret bridge state crosses into extension storage. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-198-05 | Spoofing | `ws/mcp-bridge-client.js` WebSocket endpoint | accept | Phase 198 does not change localhost bridge authentication; mitigate only by avoiding secret persistence and preserving narrow existing MCP routing. |
| T-198-06 | Information Disclosure | `chrome.storage.session.mcpBridgeState` | mitigate | Persist only status, timestamps, reconnect counts, wake reasons, and disconnect classes; never persist MCP payloads, tab content, API keys, passwords, card numbers, or CVVs. |
| T-198-07 | Denial of Service | reconnect scheduling | mitigate | Keep the existing `_reconnectTimer` single-timer guard, cap delay with `MCP_RECONNECT_MAX_MS = 30000`, and use one named alarm `fsb-mcp-bridge-reconnect`. |
| T-198-08 | Tampering | `background.js` wake path arming | mitigate | Preserve sender-id validation in `runtime.onMessage` before arming so external senders cannot use rejected messages to drive reconnect attempts. |
| T-198-09 | Repudiation | lifecycle state | mitigate | Record `lastWakeReason`, `lastConnectAttemptAt`, `lastConnectedAt`, and `lastDisconnectReason` in session state for live troubleshooting without long-lived history. |
</threat_model>
