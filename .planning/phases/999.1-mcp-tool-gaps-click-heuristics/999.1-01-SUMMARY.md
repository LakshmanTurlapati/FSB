# Phase 999.1 Plan 01: MCP Bridge Route-Aware Dispatch and execute_js Handler Summary

Route-aware MCP tool dispatch using TOOL_REGISTRY _route field, with dedicated chrome.scripting.executeScript handler for execute_js in MAIN world.

## What Was Done

### Task 1: Load TOOL_REGISTRY into service worker scope and add route-aware dispatch

**Commit:** 6c592d2

Added `importScripts('ai/tool-definitions.js')` to background.js (line 8) to make TOOL_REGISTRY, getToolByName, and getToolsByRoute available in the service worker scope where mcp-bridge-client.js runs.

Replaced the existing `_handleExecuteAction` method in ws/mcp-bridge-client.js with route-aware dispatch:
- Looks up the tool name in TOOL_REGISTRY using `getToolByName(payload.tool)`
- If `_route === 'background'`, dispatches to `_handleExecuteBackground`
- Otherwise falls through to `_sendToContentScript` (existing behavior preserved)
- Added `source: 'mcp-manual'` to content script messages

Added `_handleExecuteBackground` method that:
- Special-cases `execute_js` to `_handleExecuteJS`
- Routes other background tools via `_dispatchToBackground` (chrome.runtime.sendMessage)
- Uses `_contentVerb` from tool definition as the action name, falling back to tool name

### Task 2: Implement execute_js background handler

**Commit:** 6c592d2 (same commit -- both tasks modify mcp-bridge-client.js)

Added `_handleExecuteJS` method to MCPBridgeClient that:
- Validates `params.code` is a non-empty string
- Uses `chrome.scripting.executeScript` with `world: 'MAIN'` for full page DOM access
- Executes user code via `new Function(code)` inside the injected function for multi-statement support
- Serializes return values (undefined/null as strings, objects via JSON.stringify, others via String())
- Catches both user code errors (inside the injected function) and chrome API errors (restricted pages)
- Returns `{ success, result, tool }` matching the existing MCP result format
