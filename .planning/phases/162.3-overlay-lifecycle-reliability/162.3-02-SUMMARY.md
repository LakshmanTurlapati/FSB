# Phase 162.3 Plan 02: Long-Wait Overlay Summary

**Active overlays now survive long model waits with heartbeat-backed waiting states, and dashboard previews resync from the same canonical content overlay state**

## Accomplishments

- `background.js` now sends active-session overlay heartbeats from the cached canonical overlay snapshot and follows those sends with overlay rebroadcast requests for the streamed tab.
- `content/messaging.js` replaces the hard 60-second overlay destruction path with a degraded waiting state plus a second-stage orphan timer, while still clearing terminal overlays cleanly.
- `content/dom-stream.js` forces immediate overlay broadcasts on stream start, resume, and explicit overlay requests so the dashboard rehydrates from `FSB.overlayState` without waiting for a later mutation.

## Files Created/Modified

- `background.js` - Added cached-overlay heartbeat scheduling, waiting-state degradation metadata, and a shared helper that requests DOM-stream overlay rebroadcasts for the active streamed tab.
- `content/messaging.js` - Replaced the single-stage 60-second watchdog cleanup with degrade-first behavior and extended orphan cleanup, and cleared stale watchdogs on final overlays.
- `content/dom-stream.js` - Allowed forced overlay broadcasts and emitted them immediately on `domStreamStart`, `domStreamResume`, and `domStreamRequestOverlay`.

## Decisions Made

- Heartbeats never create a second overlay schema; they resend the last canonical status payload through `sendSessionStatus`.
- Dashboard preview synchronization stays on the existing `ext:dom-overlay` consumer path instead of branching preview state into a separate model.
- The watchdog degrades active overlays to a waiting/reconnecting state before final cleanup so healthy but idle runs do not appear dead.
