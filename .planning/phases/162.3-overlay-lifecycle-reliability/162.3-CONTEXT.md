# Phase 162.3: Overlay Lifecycle Reliability - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning
**Source:** Debug evidence + roadmap gap insertion

<domain>
## Phase Boundary

Keep the glow animation, progress overlay, and debugger/dashboard overlay trustworthy throughout an active automation run. The phase covers mid-run disappearance caused by navigation, content-script reconnects, sparse `sessionStatus` delivery, and long provider waits. It does not add new overlay visuals or redesign the UI. It is reliability and lifecycle wiring only.

</domain>

<decisions>
## Implementation Decisions

### Canonical progress channel
- **D-01:** The canonical source for content-overlay lifecycle remains `sendSessionStatus` / `overlayState`, not popup/sidepanel `sessionStateEvent` messages. Phase 162 UI emitter wiring is useful for operator surfaces but does not solve content overlay survival.

### Rehydration after reconnect
- **D-02:** Claude's Discretion. When navigation or a background-port reconnect destroys overlay instances in the content script, the active session must be able to re-apply the latest overlay state without waiting for a rare special transition such as tab switch or initial session start.

### Agent-loop progress integration
- **D-03:** Claude's Discretion. Normal iteration flow should participate in overlay updates. `report_progress` and/or iteration boundaries should refresh overlay state so the content overlay reflects live automation progress instead of relying on startup-only messages.

### Long-wait behavior
- **D-04:** Claude's Discretion. The 60-second watchdog may remain as a cleanup safety net, but active sessions must not silently lose overlays during provider waits or retry windows. A heartbeat refresh, waiting-state re-emission, or degraded overlay state is acceptable.

### Dashboard synchronization
- **D-05:** Claude's Discretion. Dashboard/debugger overlay state should remain aligned with the same canonical overlay state used by the content script. Fixes should avoid creating a second independent progress model for the dashboard.

### Scope guardrails
- **D-06:** Do not redesign visual styles, add new user-facing controls, or change unrelated automation behavior. This phase is about lifecycle correctness and state continuity.

### Claude's Discretion
- Exact ownership split between background.js, agent-loop.js, and content scripts
- Whether to emit overlay refreshes per iteration, per tool, per report_progress, or a hybrid
- Whether watchdog behavior should extend timeout, degrade state, or both
- Whether reconnect handling should pull state from activeSessions, persisted session data, or both

</decisions>

<code_context>
## Existing Code Insights

### Confirmed behavior
- `content/lifecycle.js` destroys `viewportGlow`, `progressOverlay`, and `actionGlowOverlay` on port disconnect, and clears `FSB.overlayState`
- `content/messaging.js` rebuilds overlays only when a fresh `sessionStatus` message arrives
- `content/messaging.js` watchdog destroys overlays after 60 seconds without `sessionStatus`
- `background.js` sends `sessionStatus` at session start and a few special transitions, but not as a routine per-iteration progress signal
- `ai/agent-loop.js` receives `sendSessionStatus` in options as `sendStatus`, but normal iterations do not call it; `report_progress` only mutates `session.lastAiReasoning`

### Architectural implication
- Overlay survival currently depends on sparse state pushes plus luck around navigation timing
- The dashboard/debugger overlay derives from `FSB.overlayState`, so once content overlay state is lost, downstream overlay fidelity also drops

### Likely integration points
- Add canonical overlay refresh points in `ai/agent-loop.js`
- Persist or reconstruct the latest overlay payload in `background.js` for active sessions
- Rehydrate active overlay state after content reconnect / ready signals
- Soften or refresh watchdog behavior in `content/messaging.js` for active-running sessions

</code_context>
