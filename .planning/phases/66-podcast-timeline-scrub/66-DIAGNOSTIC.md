## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-66):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55-65 and now Phase 66. Without it, no tool (DOM or CDP based) can execute. This is the primary blocker for all live MCP testing in this milestone.
- **click_at on input[type="range"] behavior unknown:** The `input[type="range"]` HTML element may or may not respond to CDP `Input.dispatchMouseEvent` the same way it responds to native user clicks. Some browsers handle range input clicks by setting the value to the proportional position clicked; others require specific interaction with the thumb element. The Buzzsprout scrubber uses a native range input, which is more promising than a custom div-based progress bar, but the CDP click behavior on range inputs has not been verified.
- **Total duration extraction requires multiple fallback strategies:** Buzzsprout provides `data-duration="2144"` and `aria-valuemax="2144"` (reliable, exact seconds). Spreaker provides `$store.audio.getDuration()` (JS-only, requires script execution). Apple Podcasts and Spotify may use different attributes. A `get_audio_duration` tool or a standardized extraction method would simplify this across platforms.
- **Time display text parsing for verification:** After seeking, the current time needs to be parsed from text (e.g., "14:22", "0:14:22", or "14:22 / 35:44"). Different platforms format time differently. A standardized time text parser for MM:SS and HH:MM:SS with tolerance comparison would be valuable.
- **set_audio_time tool (proposed):** A tool that directly calls `document.querySelector('audio').currentTime = 862` via JavaScript execution would bypass all progress bar coordinate calculations and pixel granularity issues. This would be deterministic and exact. The tool would: (a) find the audio element, (b) set currentTime to the target seconds, (c) return the actual currentTime after setting. This is the most reliable way to seek audio but bypasses testing the progress bar click interaction.
- **No get_bounding_rect tool (carried from Phases 57-65):** To determine the precise viewport coordinates of the progress bar for click_at, a dedicated `get_bounding_rect(selector)` tool returning exact {x, y, width, height} would eliminate coordinate estimation. Currently coordinates must be inferred from DOM snapshot position data.
- **Alpine.js SPA player detection:** Spreaker and similar SPA podcast sites render the player via JavaScript frameworks. The site guide selectors may not match until the framework initializes. A `wait_for_element` approach is needed before attempting DOM snapshot on SPA players.

## Autopilot Recommendations

1. **Detect audio player type from DOM before choosing interaction strategy:** Before attempting any timeline scrub, scan for the audio player implementation type. Check for: (a) Native `<audio>` element with `data-duration` or duration attribute -- most reliable for direct seeking, (b) `input[type="range"]` as the progress bar -- supports native click-to-seek on most browsers, (c) Custom div-based progress bar with `role="slider"` and `aria-valuemax` -- requires click_at at calculated position, (d) SPA-rendered player (Alpine.js, React, Vue) where player elements only exist after JavaScript initialization -- requires wait_for_element before scanning. Buzzsprout uses native `<audio>` + `input[type="range"]` (ideal). Spreaker uses Alpine.js store (requires JS execution or wait_for_element).

2. **Extract total duration using multiple fallback strategies in priority order:** (a) `data-duration` attribute on the `<audio>` element (Buzzsprout: `data-duration="2144"`, exact seconds), (b) `aria-valuemax` on `input[type="range"]` or `role="slider"` element (Buzzsprout: `aria-valuemax="2144"`), (c) `audio.duration` property via JavaScript execution (works on any platform after audio loads), (d) Duration text display element text content parsed from "MM:SS" or "HH:MM:SS" format, (e) Estimation from common podcast lengths if no other source available (30min=1800s, 45min=2700s, 60min=3600s). Always verify that 14:22 (862 seconds) is within the total duration before attempting to seek.

3. **Use click_at on the progress bar track as the preferred seeking method:** Most audio players respond to click events on the progress bar by seeking to the proportional position. Calculate targetX = trackLeft + (targetSeconds / totalDurationSeconds) * trackWidth. For Buzzsprout: targetX = trackLeft + (862/2144) * trackWidth = trackLeft + 0.402 * trackWidth. targetY = trackTop + trackHeight/2. If the progress bar is an `input[type="range"]`, the browser's native range input click handling should set the value directly. If the progress bar is a custom div, the player's JavaScript click handler should intercept and seek.

4. **Ensure audio is loaded and playing before attempting to seek:** Most podcast players only load the audio file on first play click. Buzzsprout uses `preload="metadata"` on the audio element, meaning only metadata (duration) is pre-loaded but audio data is not. The seeking workflow must: (a) Click the play button (`data-player-target="playButton"` on Buzzsprout, `$store.audio.playNow(episode)` on Spreaker), (b) Wait 2-3 seconds for audio to start loading/playing, (c) Verify audio state by checking for pause button visibility or `data-player-status` attribute change, (d) Then attempt the timeline scrub. Some players auto-pause after seeking -- this is acceptable as long as the position is correct.

5. **Verify seek position using time display text with 5-second tolerance:** After clicking at the 14:22 position, verify by reading the time display element: (a) On Buzzsprout: read `[data-player-target="progressTime"]` text content, (b) On Spreaker: read the `<time>` element with `x-text` binding showing `SP.Utils.formatTime($store.audio.getCurrent())`, (c) Parse the displayed time from "MM:SS" format to total seconds, (d) Accept any time between 14:17 and 14:27 (862 +/- 5 seconds) as success. Also check `aria-valuenow` on the range input (should be approximately 862) and progress fill width (should be approximately 40.2%).

6. **Use drag method as fallback when click_at does not seek correctly:** If click_at on the progress bar does not change the current time, attempt the drag method: (a) Locate the current thumb/handle position on the range input (initially at left edge if aria-valuenow=0), (b) Execute `drag(thumbX, thumbY, targetX, targetY, 10, 20)` where targetX is the calculated 40.2% position, (c) The drag produces mousePressed at thumb, 10 intermediate mouseMoved events, then mouseReleased at target. Some custom players only respond to drag events on the thumb, not click events on the track.

7. **Handle platform-specific audio player rendering differences:** Different podcast platforms render their players differently: (a) **Buzzsprout**: Server-rendered HTML with Stimulus.js controller, `<audio>` element present in initial HTML, `input[type="range"]` scrubber with ARIA attributes -- best for DOM-based automation, (b) **Spreaker**: Alpine.js SPA, entire player rendered via JavaScript store, no player elements in server HTML -- requires wait_for_element and JavaScript-aware interaction, (c) **SoundCloud**: Custom `.playbackTimeline__` class-based progress bar with different structure, (d) **Spotify**: Authentication-walled, avoid for unauthenticated testing. Prioritize Buzzsprout for MICRO-10 testing due to server-rendered DOM.

8. **Dismiss cookie/consent banners before interacting with the audio player:** Check for and dismiss common cookie banners that may overlay the player: `#onetrust-accept-btn-handler`, `[class*="cookie"] button[class*="accept"]`, `[class*="consent"] button`. On Buzzsprout, also check for any signup/newsletter modals. Dismiss by clicking the accept/close button before attempting play or seek interactions.

9. **Consider keyboard-based seeking as an alternative to click_at:** Buzzsprout's scrubber instructions explicitly state: "Use Left/Right to seek, Home/End to jump to start or end. Hold shift to jump forward or backward." This means `press_key` could be used: (a) Focus the range input via click or tab, (b) Use Right arrow to seek forward in small increments, or (c) Use Home to jump to start, then seek forward. However, keyboard seeking is imprecise for a specific timestamp (14:22) and would require many key presses. click_at at the calculated position is more efficient for exact seeking.

10. **Retry with adjusted position if initial seek misses by more than 5 seconds:** If the first click_at lands outside the 14:17-14:27 tolerance window: (a) Read the actual current time after first attempt, (b) Calculate the error in seconds (actualTime - 862), (c) Calculate pixel adjustment: adjustX = -(errorSeconds / totalDurationSeconds) * trackWidth, (d) Retry click_at at (targetX + adjustX, targetY), (e) Maximum 3 attempts with decreasing error. If all 3 attempts fail to land within tolerance, classify as PARTIAL and document the closest achieved time.
