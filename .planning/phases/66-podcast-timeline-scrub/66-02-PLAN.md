Execute the MICRO-10 edge case test using MCP manual tools following the scrubTimeline workflow from the podcast-player site guide. Test steps:

1. Use MCP navigate tool to go to a podcast episode page with an audio player. Prioritized targets (try in order until one loads with a working audio player):
   a. A Podbean public podcast episode page (e.g., search for "podbean podcast" to find a public episode)
   b. A Buzzsprout public podcast episode page
   c. A SoundCloud podcast episode (e.g., soundcloud.com/search for a podcast episode)
   d. A Spreaker podcast episode page
   e. Any public page with an HTML5 audio element and visible timeline/progress bar
   The goal is to find a page with a visible audio timeline that can be interacted with. Avoid pages that require authentication (Spotify web player needs login). Prefer platforms with public, no-auth podcast episode pages.

2. Use read_page to verify the page loaded and an audio player is visible. Dismiss any cookie/consent banners via click if present. Note the page title and URL.

3. Use get_dom_snapshot to map the page elements. Identify:
   - The audio player container
   - The progress bar / timeline track (horizontal bar showing playback position)
   - The progress bar thumb/handle (if present as a separate draggable element)
   - The current time display element (showing MM:SS)
   - The total duration display element (showing MM:SS or HH:MM:SS)
   - The play/pause button (needed to start audio if not auto-playing)
   - Any HTML5 audio element (for currentTime/duration attributes)
   Record CSS selectors for the progress bar, time display, and play button.

4. If audio is not playing, click the play button to load and start the audio. Wait 2-3 seconds for the audio to load (use waitForDOMStable or a short delay).

5. Determine total episode duration:
   - Read total duration from: duration text display, aria-valuemax on slider, audio element duration attribute
   - Parse MM:SS or HH:MM:SS format to total seconds
   - If duration text shows "1:23:45" = 5025 seconds; "45:30" = 2730 seconds; etc.

6. Calculate the 14:22 position on the progress bar:
   - 14:22 = 862 seconds
   - Get progress bar track bounding rect from DOM snapshot
   - targetX = trackLeft + (862 / totalDurationSeconds) * trackWidth
   - targetY = trackTop + (trackHeight / 2)
   - Log the calculation: "862 / {totalSeconds} = {percentage}%, trackLeft={left}, trackWidth={width}, targetX={targetX}"

7. STRATEGY A -- CLICK ON PROGRESS BAR:
   Use click_at(targetX, targetY) on the progress bar at the 14:22 position.
   Most audio players listen for click events on the progress bar and seek to the clicked position.

8. VERIFY: Use read_page or get_dom_snapshot to read the current time display after seeking.
   - Check if current time shows approximately "14:22" (accept 14:17 to 14:27 = within 5 seconds)
   - Also check aria-valuenow if present
   - If verification succeeds: PASS
   - If time moved but not to 14:22: PARTIAL
   - If time did not change: try Strategy B (drag)

9. STRATEGY B -- DRAG THUMB (if Strategy A failed):
   - Locate the progress bar thumb element position
   - drag(thumbX, thumbY, targetX, targetY, 10, 20) to slide thumb to 14:22 position
   - Re-verify current time display

10. If both strategies attempted, document which worked and which failed.

Record every step outcome (tool used, target, result, notes) for the diagnostic report.
