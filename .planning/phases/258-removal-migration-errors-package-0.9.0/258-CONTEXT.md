## Implementation Decisions

### Removal shape: stubs, not deletion

The two `server.tool()` registrations in `mcp/src/tools/visual-session.ts` for `start_visual_session` and `end_visual_session` STAY registered, but their handlers are replaced with typed-rejection stubs that return `TOOL_REMOVED` immediately. Reasons:

1. **Discoverability** -- a caller invoking the old tool name by mistake gets a STRUCTURED error pointing at the migration recipe, not a generic "tool not found". MCP clients can route on the typed error code.
2. **`tools/list` posture** -- the stub registrations are still listed by `tools/list`, but their descriptions are rewritten to say "REMOVED in v0.9.0 -- use the implicit contract; see CHANGELOG / mcp/README.md visual-session section". This is more informative than the tool being absent.
3. **Minimal diff** -- the existing input schemas + `queue.enqueue(...)` machinery is largely unchanged; only the handler body becomes a 4-5 line stub that throws / returns the typed error.

The stub description text (planner refines):
```
[REMOVED in v0.9.0 -- see TOOL_REMOVED error body for migration recipe]
This tool was removed in v0.9.0 of fsb-mcp-server. The visual session is now implicit on every action tool call via the required visual_reason + client field bundle (see .planning/v0.9.62-CONTRACT.md). Calling this tool returns the typed TOOL_REMOVED error with a migration recipe pointer. Use action tools directly (click, type_text, navigate, ...) with the field bundle instead.
```

### `TOOL_REMOVED` body shape

The error body MUST contain (planner enumerates the exact JSON shape):

- `code`: `'TOOL_REMOVED'`
- `removed_tool`: the literal tool name that was called (`'start_visual_session'` or `'end_visual_session'`).
- `removed_in_version`: `'0.9.0'`.
- `migration_recipe`: a short prose description of the new contract (1-3 sentences naming required `visual_reason` + `client` + optional `is_final`, the sliding 60s window, and `is_final: true` for early clear).
- `references`: `[ 'CHANGELOG.md#v0.9.0', 'mcp/README.md#visual-session' ]` or whatever anchors land.

Follows the v0.9.60 typed-error precedent: structured body, code-only top-level field, layered detail built by `buildLayeredDetail`.

### CHANGELOG entry

Add a new entry at the top of `CHANGELOG.md` (or `mcp/CHANGELOG.md` -- planner discovers which exists; if both, the package-level changelog under `mcp/` is primary):

```markdown
## v0.9.0 -- 2026-05-11 -- Implicit Visual Session Contract (BREAKING)

### Breaking changes

- Removed `start_visual_session` and `end_visual_session` MCP tools. Calling either by name returns the typed `TOOL_REMOVED` error.
- Every MCP action tool now requires `visual_reason` (string) and `client` (allowlisted string) in its input. Calls missing either field are rejected with `VISUAL_FIELDS_REQUIRED`. Calls with a non-allowlisted `client` are rejected with `BADGE_NOT_ALLOWED`.

### Migration recipe

Before (v0.8.0 / v0.9.36 explicit contract):

```
mcp> start_visual_session({ client: 'Claude', visual_reason: 'Logging in' })  -> session_token
mcp> click({ selector: '#submit' })
mcp> end_visual_session({ session_token: '...' })
```

After (v0.9.0 implicit contract):

```
mcp> click({ selector: '#submit', visual_reason: 'Logging in', client: 'Claude', is_final: true })
```

The visual session is created implicitly on the first action call, refreshed on each subsequent action call (sliding 60-second death timer), and cleared either by `is_final: true` (immediate) or by 60 seconds of silence (auto-clear).
```

### `mcp/README.md` flag

Top-of-visual-session-section banner:

```markdown
> **v0.9.0 breaking change** -- The explicit `start_visual_session` / `end_visual_session` tools were REMOVED. Action tools now require `visual_reason` + `client` fields. See [CHANGELOG.md](./CHANGELOG.md#v0.9.0) for the migration recipe.
```
