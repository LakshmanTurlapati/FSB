<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Background memory -> persisted session store | Persisted client-owned session state must remain bounded, serializable, and safe to restore after service-worker churn. |
| Persisted owner -> newly active tab surface | Replay must target only the currently owned tab and token, never a stale or replaced owner. |
| Stale timers -> visible overlay | Timeout and final-freeze replay must clear stale sessions instead of extending their life indefinitely. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-204-02-01 | Tampering | Persisted stale owner | mitigate | Token-scoped restore/replay and same-tab replacement logic prevent stale persisted sessions from clearing or overwriting newer owners. |
| T-204-02-02 | Denial of Service | Replayed stale glow | mitigate | Reuse 60s degrade and orphan cleanup timing, and clear immediately when the final-freeze deadline is already expired. |
| T-204-02-03 | Repudiation | Runtime churn | mitigate | Persisted session records include timestamps and explicit final-clear metadata so replay/cleanup decisions are deterministic and testable. |
| T-204-02-04 | Regression | Existing lifecycle flow | mitigate | Keep replay inside the existing `sendSessionStatus()` pipeline and lock persistence behavior with focused contract tests. |
</threat_model>
