<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host page to Shadow DOM | Page CSS must not leak into overlay styling; Shadow DOM provides isolation |
| Top layer to Page stacking context | Popover API promotes overlay above all page z-index contexts |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-170-01 | Tampering | Host page CSS leaking into Shadow DOM | mitigate | Shadow DOM `:host { display: block !important }` + `* { box-sizing: border-box; margin: 0; padding: 0; }` reset prevents all inheritance leakage; `all: initial !important` on host element cuts remaining chains |
| T-170-02 | Denial of Service | Heavy DOM pages (Amazon) causing overlay rAF jank | accept | `contain: layout paint` isolates overlay from page layout recalculation; `will-change: transform` on progress bar promotes to compositor layer; rAF timer is lightweight (single textContent write per frame) |
| T-170-03 | Elevation of Privilege | Page JS accessing overlay Shadow DOM | accept | Shadow DOM mode: 'open' allows page JS to query shadow root, but overlay is non-interactive (pointer-events: none) with no sensitive data. No XSS vector -- overlay only displays task progress text from the extension's own background.js |
| T-170-04 | Information Disclosure | Overlay displaying task details on sensitive pages | accept | Task names and action summaries are user-initiated; the overlay shows what the user asked the extension to do. No credentials or page content are displayed in the overlay |
</threat_model>
