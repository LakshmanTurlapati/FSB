Purpose: DARK-10 requires scraping text from a site that blocks right-clicks (oncontextmenu=false), disables text selection (user-select:none, onselectstart=false), and masks CSS classes (randomized/obfuscated class names). The critical insight is that ALL of these protections operate at the browser UI layer -- they prevent human users from using right-click menus, selecting text with mouse, and identifying elements by readable class names. However, the DOM tree itself still contains the full text content in element textContent/innerText properties, and content scripts (running in the page context) can read DOM elements regardless of CSS user-select rules, JS event handler blocks, or class name obfuscation. The MCP tools (get_dom_snapshot, read_page, get_text) access the DOM programmatically, completely bypassing these UI-layer restrictions. The site guide must document: (1) how to detect each anti-scrape protection type, (2) why each protection is ineffective against DOM-level access, (3) the specific MCP tools and techniques to extract text despite protections, (4) target sites known to use these protections with site-specific patterns, and (5) a workflow for systematically extracting text from any anti-scrape protected site.

Create a new file site-guides/utilities/anti-scrape-text-extraction.js. This site guide targets sites that use JavaScript and CSS anti-scrape protections to prevent text extraction. This is the DARK-10 dark pattern avoidance edge case.

**Distinction from other DARK patterns:** DARK-10 is NOT about hidden buttons (DARK-04), deceptive ads (DARK-01), or modal overlays (DARK-05). It is about text access restriction: the site deliberately prevents users from right-clicking, selecting, and copying text using JavaScript event handlers and CSS rules, plus uses obfuscated CSS class names to frustrate automated scraping. The counter-strategy is DOM-level text extraction using content script tools that bypass the UI layer entirely.

**Anti-scrape protection types documented in the guide:**

1. **Right-click disabled (oncontextmenu):** Site adds `oncontextmenu="return false"` to body/document or uses `document.addEventListener('contextmenu', e => e.preventDefault())`. This prevents the browser context menu from appearing when right-clicking. BYPASS: Irrelevant to MCP -- get_dom_snapshot and read_page never use the context menu. Content script accesses DOM directly.

2. **Text selection disabled (user-select:none + onselectstart):** Site applies CSS `user-select: none` or `-webkit-user-select: none` to body or content elements, and/or adds `onselectstart="return false"` or `document.addEventListener('selectstart', e => e.preventDefault())`. This prevents mouse-drag text selection and Ctrl+A selection. BYPASS: Irrelevant to MCP -- get_text reads element.textContent which is unaffected by CSS user-select. read_page extracts innerText regardless of selection restrictions.

3. **Copy disabled (oncopy):** Site adds `oncopy="return false"` or `document.addEventListener('copy', e => e.preventDefault())`. This prevents Ctrl+C from copying selected text to clipboard. BYPASS: Irrelevant to MCP -- text extraction via DOM properties does not trigger the copy event.

4. **CSS class name obfuscation:** Site uses randomized or hashed class names (e.g., `_a3f2b`, `css-1x92kp`, `sc-bwzfXH`) generated by CSS-in-JS libraries (styled-components, Emotion, CSS Modules) or deliberate obfuscation. This makes it hard to write stable CSS selectors. BYPASS: Use element type, tag name, data attributes, aria attributes, text content matching, or structural position (nth-child, parent-child relationships) instead of class names. get_dom_snapshot returns element types and text content regardless of class names.

5. **Transparent overlay div:** Site places a transparent `<div>` with `position:absolute; z-index:999; user-select:none` over the content area, intercepting all mouse events (clicks, selections). The actual text is behind this div. BYPASS: get_dom_snapshot reads ALL elements including those behind overlays. The overlay div has no text content; the real content elements underneath contain the text. Use get_text on the content elements directly, ignoring the overlay.

6. **JavaScript-rendered text:** Site loads text content dynamically via JavaScript (AJAX, fetch, WebSocket) rather than including it in server HTML. Combined with obfuscated class names, this makes HTTP-based scraping nearly impossible. BYPASS: Content script runs after JavaScript execution, so get_dom_snapshot sees the fully rendered DOM including dynamically loaded text.

7. **CSS content property text:** Site renders visible text via CSS `content` property in `::before`/`::after` pseudo-elements rather than in DOM text nodes. This text appears visually but is not in element.textContent. BYPASS: Use `window.getComputedStyle(el, '::before').content` via execute_js to extract pseudo-element text. Rare but documented as edge case.

8. **Image-based text (CAPTCHA-like):** Site renders text as images or canvas instead of DOM text. BYPASS: NOT bypassed by DOM tools -- this requires vision/OCR which MCP does not have. Documented as a limitation.

**Target sites known to use anti-scrape protections:**

1. **Medium.com (metered paywall + selection restriction):** Medium applies user-select restrictions on articles behind the metered paywall. Article content is in `<article>` elements with `section` children. Text is in `<p>`, `<h1>`-`<h4>`, `<blockquote>` tags. Class names are CSS Module hashes (e.g., `pw-post-body-paragraph`). Selectors: article, section > p, .meteredContent, [data-selectable-paragraph].

2. **Bloomberg.com (premium content protection):** Bloomberg uses JS-based copy/selection prevention on premium articles. Article body is in `<article>` or `[data-component="article-body"]`. Paragraphs use obfuscated class names from styled-components. Content may be partially server-rendered with full text loaded via client-side hydration. Selectors: article, [data-component="article-body"], .body-content p.

3. **Genius.com / lyrics sites (right-click + selection block):** Lyrics sites commonly disable right-click and text selection to protect copyrighted lyrics. Genius uses `[data-lyrics-container]` for lyrics content with `user-select:none` applied. Text is in span elements with `[data-reactid]` or similar React attributes. Selectors: [data-lyrics-container], .lyrics, .Lyrics__Container-sc-*.

4. **Recipe/content sites with ad protection (e.g., Allrecipes, Food Network):** These sites combine anti-scrape with aggressive ad placement. Content is in structured article elements but wrapped with overlay divs and ad containers that interfere with selection. Recipe text in `.recipe-body`, `.ingredients-section`, or schema.org structured data. Selectors: [itemtype*="Recipe"], .recipe-body, .ingredients-item.

5. **News paywalls (WSJ, NYT behind paywall):** Use a combination of CSS class obfuscation (styled-components), selection restriction, and content truncation. The full article text may be in the DOM but hidden by CSS (display:none on later paragraphs). Selectors: article, [data-testid="article-body"], .article-body p.

6. **Academic paper sites (ResearchGate, Academia.edu):** Restrict text selection on paper abstracts and content to force downloads. Content in article-like containers with obfuscated class names. Selectors: .research-detail__content, .Abstract, [itemprop="description"].
