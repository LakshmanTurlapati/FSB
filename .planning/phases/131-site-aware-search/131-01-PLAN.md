<objective>
Make the MCP `search` tool site-aware: detect and use the current site's own search input via DOM heuristics, falling back to Google only when no site search exists.

Purpose: Users on Amazon/YouTube/GitHub who say "search for X" currently get redirected to Google, losing the site context. Site-aware search keeps users on the site and returns site-native results.
Output: Modified `content/actions.js` with `detectSiteSearchInput()` + `tools.siteSearch()`, and updated `mcp-server/src/tools/manual.ts` verb mapping.
</objective>

    Add a `detectSiteSearchInput()` helper function BEFORE the `tools` object (near line 1467, after `findSubmitButton`). This function implements the 4-tier selector cascade per D-01:

    **Tier 1:** `input[type="search"]`
    **Tier 2:** `[role="search"] input, [role="search"] textarea`
    **Tier 3:** `input[name="q"], input[name="query"], input[name="search_query"], input[name="search"], input[name="keyword"], input[name="keywords"]`
    **Tier 4:** `input[placeholder*="Search" i], input[aria-label*="search" i], textarea[placeholder*="Search" i]`
    **Tier 5 (bonus):** `form[action*="search"] input[type="text"], form[action*="search"] input:not([type])`

    Per D-02: If a tier returns multiple matches, filter by visibility (offsetParent !== null or getComputedStyle display !== 'none') and prefer elements whose getBoundingClientRect shows them in the viewport. Return the first visible match from the highest-priority tier that has one. Return null if no tier finds a visible match.

    Per D-03: No AI calls -- pure DOM queries.

    Then add `tools.siteSearch` as an async function in the `tools` object, placed immediately AFTER `searchGoogle` (around line 3483). Implementation:

    1. Validate `params.query` exists, return error if not
    2. Call `detectSiteSearchInput()` to find the site's search input
    3. If no input found: call `tools.searchGoogle(params)` and return its result with an added `method: 'google-fallback'` field (per D-06)
    4. If input found:
       a. Call `FSB.smartEnsureReady(searchInput, 'click')` to ensure element is ready (this also dismisses cookie consent per Phase 130)
       b. Click the input to focus it: `searchInput.focus(); searchInput.click()`
       c. Clear existing text: `searchInput.value = ''; searchInput.dispatchEvent(new Event('input', { bubbles: true }))`
       d. Type the query character by character is NOT needed -- set value directly and dispatch input event: `searchInput.value = params.query; searchInput.dispatchEvent(new Event('input', { bubbles: true }))`
       e. Wait 200ms for autocomplete to populate (some sites need this)
       f. Submit per D-04: dispatch Enter keydown+keyup on the input. Use the same KeyboardEvent pattern from pressEnter (key: 'Enter', code: 'Enter', keyCode: 13, which: 13, bubbles: true, cancelable: true)
       g. Wait for page stability: `await FSB.waitForPageStability({ maxWait: 3000, stableTime: 500 })` (per D-05)
       h. If URL did not change and DOM did not change significantly after Enter, try submit button fallback: find the form via `searchInput.closest('form')`, call `findSubmitButton(form)`, click it if found
       i. Return `{ success: true, query: params.query, method: 'site-search', searchInputSelector: [selector that matched], url: window.location.href }`

    Finally, export the helper: add `FSB.detectSiteSearchInput = detectSiteSearchInput;` in the exports block (near line 5421 alongside other exports).
