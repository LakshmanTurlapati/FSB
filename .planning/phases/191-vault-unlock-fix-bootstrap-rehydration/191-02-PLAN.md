<objective>
Add eager session key rehydration at service worker startup so the vault remains usable across SW restart cycles without requiring re-unlock.

Purpose: Chrome kills idle service workers after ~30 seconds. When the SW restarts, the in-memory `_credentialSessionKey` field on the SecureConfig singleton is null. The key IS persisted in chrome.storage.session (survives SW restarts but not browser close), so we need to proactively reload it into memory at startup. Without this, every credential operation after a SW restart returns `{errorCode: 'locked'}` even though the user already unlocked.

Output: A bootstrap call near the existing `restoreSessionsFromStorage()` invocation that eagerly loads both the credential session key and payment access state from chrome.storage.session into SecureConfig's in-memory fields.
</objective>

```javascript
// Eagerly rehydrate vault session key on service worker startup (Phase 191 - VAULT-03)
// chrome.storage.session survives SW restarts but the in-memory SecureConfig fields reset to null.
// Calling these methods loads persisted keys back into memory so vault operations work
// without requiring re-unlock after Chrome kills the idle service worker.
(async () => {
  try {
    const sessionKey = await secureConfig._loadCredentialSessionKey();
    await secureConfig._loadPaymentAccessState();
    if (sessionKey) {
      console.log('[FSB] Vault session key rehydrated from chrome.storage.session');
    }
  } catch (err) {
    console.warn('[FSB] Failed to rehydrate vault session on wake:', err.message || err);
  }
})();
```
