## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-64):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55, 56, 57, 58, 59, 60, 61, 62, 63, and now 64. Without it, no tool (DOM or CDP based) can execute. This is the primary blocker for all live MCP testing in this milestone.
- **DragEvent acceptance by dropzone libraries unverified:** The drop_file tool creates a synthetic File via `new File([content], name, { type })` and injects it into a DataTransfer via `dataTransfer.items.add(file)`. It then dispatches `new DragEvent('drop', { bubbles: true, cancelable: true, dataTransfer })`. The key question is whether dropzone libraries (Dropzone.js, react-dropzone, native HTML5 handlers) process synthetic DragEvents identically to real browser drag-drop events. Potential issues: (a) Some libraries check `event.isTrusted` which is `false` for dispatched events. (b) DataTransfer constructor behavior differs across browser versions. (c) Some libraries access `dataTransfer.files` (FileList) rather than `dataTransfer.items` (DataTransferItemList) -- the synthetic DataTransfer may populate items but not files.
- **DataTransfer.files vs DataTransfer.items discrepancy:** The drop_file tool adds the file via `dataTransfer.items.add(file)`. Most modern browsers synchronize `items` and `files`, but older Chrome versions or specific library code may read only `dataTransfer.files` directly. If `files` is not populated, the library would see an empty file list despite the item being present. This requires live browser testing to confirm.
- **event.isTrusted check in dropzone libraries:** Dispatched events have `isTrusted: false`. Some security-conscious dropzone implementations may check this property and reject untrusted events. Dropzone.js source code would need to be audited for this check. If present, CDP-level `Input.dispatchDragEvent` would be needed instead of JavaScript DragEvent dispatch.
- **CDP Input.dispatchDragEvent as potential fallback:** Chrome DevTools Protocol provides `Input.dispatchDragEvent` which creates trusted drag events at the browser level. This would bypass `isTrusted` checks but requires CDP access (available through the extension's debugger API). The current drop_file tool uses content script DragEvent dispatch which is simpler but potentially less reliable than CDP dispatch.
- **Programmatic input.files assignment:** Setting `inputElement.files = dataTransfer.files` is restricted in most browsers (the `files` property is read-only on `<input type="file">` elements for security). Some workarounds exist (using Object.defineProperty to override the getter, or using the DataTransfer trick to create a FileList) but these are browser-version dependent. This is a known limitation of Strategy B.
- **No get_bounding_rect tool (carried from Phase 57):** If CDP-level drag dispatch is needed as a fallback, precise viewport coordinates of the dropzone element are required. A dedicated `get_bounding_rect(selector)` tool would eliminate coordinate estimation.

## Autopilot Recommendations

1. **Detect dropzone type before choosing drop_file selector:** Before calling drop_file, the AI should identify which dropzone library the site uses. Dropzone.js sites have `.dropzone` or `.dz-clickable` classes. React-dropzone sites have `div[role="presentation"]` or `div[role="button"]` containers. Native HTML5 drop handlers may use any element. File.io uses `body#upload-drag-drop` as the page-level drop target. The selector passed to drop_file must match the actual element that has the drop event listener -- this is not always the visually apparent "upload area."

2. **Use the drop_file tool with the most specific selector available:** Prefer `#id` selectors over class selectors for drop_file. On file.io, use `body#upload-drag-drop` or even just `body` (since the body element has the drag-drop id). On Dropzone.js sites, use `.dropzone` or `form.dropzone`. For react-dropzone, look for the container div with `role="presentation"` or `role="button"`. If multiple potential dropzone elements exist, try the innermost one first (the one visually appearing as the upload area), then fall back to the outermost container.

3. **Check the accept attribute on hidden file inputs to match file types:** Before dropping a file, check if there is an `input[type="file"]` near the dropzone with an `accept` attribute (e.g., `accept="image/*"`, `accept=".pdf,.doc,.docx"`). If the accept attribute restricts file types, the drop_file mimeType should match. For text file testing, `text/plain` is universally accepted. For image-only upload zones, use `image/png` or `image/jpeg` as the mimeType (the content can still be a text string -- the browser checks MIME type, not actual file content, for accept validation).

4. **Verify file acceptance via DOM snapshot after drop_file:** After calling drop_file, immediately use get_dom_snapshot to check for file acceptance indicators: (a) the file name appearing in text content near the dropzone, (b) a progress bar or upload percentage, (c) `.dz-preview` or `.dz-success` elements (Dropzone.js), (d) toast notifications or status messages, (e) the dropzone changing from "empty" to "has file" state (CSS class changes, text changes from "Drag files here" to the file name). Do NOT assume the drop succeeded based on the drop_file return value alone -- always verify via DOM.

5. **Fall back to hidden input[type=file] click when DragEvent fails:** If drop_file returns success but no visual change appears in the DOM (file name not shown, no progress bar), try clicking the dropzone element (`click(".dropzone")` or `click(".dz-clickable")`). On many sites, clicking the dropzone triggers the file input's click handler, which opens the native file picker. Since content scripts cannot interact with the native file picker dialog, this is a dead end for automation. However, clicking first may initialize event listeners that make a subsequent drop_file call succeed.

6. **Handle JS-rendered upload areas (SPA sites):** Many file upload sites render the upload area entirely via JavaScript (React, Vue, Svelte, etc.). The dropzone element may not exist in the initial HTML. After navigation, use wait_for_element or wait_for_stable before attempting get_dom_snapshot. For Dropzone.js demo (dropzone.dev), the `.dropzone` element appears only after Svelte hydration. For Gofile.io, the main upload UI appears only after JavaScript loads. The AI should wait for page stability (DOM stable for 500ms+ with no mutations) before scanning for dropzone elements.

7. **Authentication detection for file upload services:** Some file upload services require login before uploading (Google Drive, Dropbox, OneDrive). Others allow anonymous uploads (file.io, gofile.io, catbox.moe). The AI should check for login walls before attempting file drops: look for redirect to `/login` or `/signin`, login form overlays, "Please sign in" messages. If authentication is required and no credentials are available, document as SKIP-AUTH and try an alternative site.

8. **File size limitations for synthetic files:** The drop_file tool creates files from the `fileContent` string parameter. The file size equals `fileContent.length` bytes. For testing purposes, small files (50-100 bytes) are sufficient. For sites that enforce minimum file sizes, increase the content length. Maximum practical size is limited by available memory in the content script context -- files over ~50MB may cause memory pressure. Real filesystem files (photos, PDFs, videos) cannot be used because content scripts have no filesystem access.

9. **Multiple file upload -- repeat drop_file for each file:** Some dropzones accept multiple files. To upload N files, call drop_file N times with different fileNames. Each call dispatches its own DragEvent sequence. Some dropzone libraries batch process files from a single drop event (reading all files from `dataTransfer.files`). Since drop_file creates a DataTransfer with a single file, each call represents a separate "drop." Dropzone.js supports `maxFiles` configuration -- check if the dropzone has a file count limit before dropping multiple files.

10. **Real file access limitation -- content scripts cannot read local filesystem:** The most fundamental limitation of the drop_file approach is that content scripts cannot access local files. The `fileContent` parameter must be a string passed from the MCP tool call. This means the tool can only drop synthetic text-based files. For real file uploads (photos, documents, videos), a different approach would be needed: either (a) CDP-level file dialog interception using `Page.setInterceptFileChooserDialog` + `Page.handleFileChooser`, (b) network-level upload via the Fetch API directly, or (c) a hybrid approach where the user provides a file URL and the extension downloads it first. None of these are currently implemented.
