## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-63):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55, 56, 57, 58, 59, 60, 61, 62, and now 63. Without it, no tool (DOM or CDP based) can execute. This is the primary blocker for all live MCP testing in this milestone. This gap makes it impossible to test whether the hover tool triggers mega-menu opening via JS mouseenter events, or whether CDP coordinate path (Strategy B) triggers CSS :hover menus.
- **Hover tool vs CSS :hover behavior unverified:** The MCP hover tool dispatches JavaScript mouseenter and mouseover events on the target element. It is unverified whether these JS events trigger CSS `:hover` pseudo-class state on the element. If the mega-menu relies on pure CSS `:hover` (not JS mouseenter), the hover tool would NOT open the menu. Strategy B (CDP drag to move the real browser cursor) would be needed for CSS :hover menus. This is the core question for MICRO-07 and cannot be answered without live browser execution.
- **No dedicated CDP move_mouse tool:** The drag tool is used as a workaround for CDP cursor positioning (drag with 1-pixel displacement). A dedicated `move_mouse(x, y)` tool that sends CDP `Input.dispatchMouseEvent` with type "mouseMoved" would be cleaner for hover-based interactions. The drag tool adds unnecessary mousePressed/mouseReleased events and requires calculating start/end coordinates with a displacement offset.
- **Click-based mega-menu detection missing from site guide:** Lowes uses click-to-open modals (data-method="post") rather than hover-triggered mega-menus. The mega-menu site guide focuses on hover interaction (Strategy A and B) but does not cover click-based mega-menus which are increasingly common on e-commerce sites. A Strategy C for click-based mega-menus should be added.
- **Dynamic modal content loading:** Lowes' mega-menu content is loaded via XHR/fetch after button click (`data-path="/modal/masthead"`). After clicking a department button, the AI would need to wait for the modal to load before searching for sub-links. The existing `waitForElement` tool could help, but the AI needs to know which selector to wait for (the modal panel container).
- **Styled-components class instability:** Lowes uses styled-components which generates CSS class hashes (e.g., `sc-dWZqqJ fPPlcZ`, `ButtonBase-sc-1ngvxvr-0`). These hashes can change on every build/deploy. The `data-linkid` attribute is stable and should be the primary selector strategy. The site guide should recommend `data-*` attributes over class selectors for styled-components sites.
- **No get_bounding_rect tool (carried from Phase 57):** Strategy B (CDP coordinate path) requires precise viewport coordinates of the nav item and mega-menu panel. A dedicated `get_bounding_rect(selector)` tool would eliminate coordinate estimation from DOM snapshot data.

## Autopilot Recommendations

1. **Detect mega-menu trigger mechanism before choosing interaction strategy:** Before attempting hover, the AI should inspect nav item elements for trigger indicators. Click-based menus: `data-method="post"`, `data-path`, `onclick`, `@click` (Vue), JavaScript event listeners. Hover-based menus: CSS `:hover` rules, `mouseenter`/`mouseleave` handlers, `aria-haspopup="true"` + no click handler. If the nav items are `<button>` elements with explicit data-method/data-path attributes (like Lowes), use click rather than hover. If the nav items are `<a>` links with adjacent hidden panels (like Best Buy's flyout pattern), try hover first.

2. **Three interaction strategies for mega-menus (not two):** Strategy A: DOM hover + click (JS mouseenter menus -- site guide current preferred). Strategy B: CDP coordinate hover path + click_at (CSS :hover menus -- site guide current fallback). **Strategy C: Click-to-open + click sub-link** (modal/panel menus like Lowes -- should be added). Strategy C is used when nav items are `<button>` elements with explicit click handlers. Click the department button, wait for the panel/modal to load, then click the sub-link. No hover path maintenance needed.

3. **Mega-menu detection from DOM aria attributes and interaction patterns:** Scan for `aria-haspopup="true"` to identify elements that open popups. Check `aria-expanded` attribute: if present and set to "false", the element toggles a panel on interaction. After hover or click, check if `aria-expanded` changed to "true" -- this confirms the menu opened. For Lowes, only "HomeCare+" had `aria-haspopup="true"` -- the department buttons (Appliances, Tools, etc.) lacked this attribute despite opening panels. Autopilot should NOT rely solely on aria attributes -- check `data-method`, `data-path`, and class patterns as fallbacks.

4. **Hover timing -- sustain hover for 300-500ms before checking for menu panel:** Most hover-triggered mega-menus have an intentional delay (hover intent) of 150-300ms before opening, plus a CSS transition of 200-400ms. After triggering hover, the AI should wait at least 300ms before re-snapshotting the DOM. The MCP tool chain natural latency (500ms-2s between tool calls) usually covers this, but for fast tool chains, an explicit wait may be needed. For click-based menus (Strategy C), the wait should be for the XHR/fetch response -- typically 200-800ms for modal content loading.

5. **Sub-link targeting inside multi-column mega-menu panels:** Mega-menu panels typically organize sub-links in 2-6 columns with category headings. The AI should: (a) identify the category heading closest to the target sub-link, (b) find sub-links within that column/section, (c) prefer `<a>` tags with href attributes over `<span>`/`<div>` elements, (d) use text content matching to find the specific sub-link (e.g., "Computer Accessories" text content). For Lowes-style modals, the panel content is loaded dynamically -- snapshot after the modal opens to find sub-links.

6. **Hover path maintenance for CSS :hover menus (L-shaped path):** When using Strategy B (CDP coordinate path), maintain the cursor inside the menu area by following an L-shaped path: (a) move vertically from the nav item straight down into the mega-menu panel, (b) then move horizontally to the target sub-link. This avoids crossing adjacent nav items which would close the current menu and open another. Use drag tool with steps=5 and stepDelayMs=30 for smooth cursor movement. The close grace period (200-500ms after mouse leaves) provides a small buffer but is not reliable enough for cross-nav-item jumps.

7. **Sticky/fixed nav bar coordinate handling:** E-commerce sites almost universally use sticky/fixed navigation bars that remain at the top of the viewport during scrolling. This means nav item coordinates are always viewport-relative (top: 0-80px typically). The CDP click_at and drag tools use viewport coordinates, so no scroll offset adjustment is needed. However, if the user has scrolled down the page, the mega-menu panel may render below the fold -- check that the panel is within viewport bounds before attempting to click sub-links.

8. **Lazy-loaded mega-menu content detection:** Some mega-menus (including Lowes) load panel content dynamically on interaction. Indicators: (a) nav buttons with `data-method`/`data-path` attributes (fetch on click), (b) empty panel containers in initial DOM that populate after hover/click, (c) `aria-busy="true"` during loading. After triggering the menu, use `waitForElement` targeting a sub-link selector inside the panel (e.g., `[role="menu"] a[href]`). If no sub-links appear within 3 seconds, the panel content may have failed to load -- retry the trigger interaction.

9. **Click-based mega-menus (hamburger menus and department buttons):** Many modern e-commerce sites use click-to-open navigation instead of hover-triggered mega-menus. Lowes uses `<button>` elements with `data-method="post"` that fetch panel content via JavaScript. Amazon uses a hamburger menu ("All") that opens a side panel. These are NOT hover-triggered -- the hover tool will not open them. The AI should detect button elements with explicit click handlers and use the click tool instead. After clicking, wait for the panel to render, then interact with sub-links.

10. **Fallback: direct URL navigation if mega-menu interaction fails:** If both hover and click strategies fail to open the mega-menu or the sub-link click does not navigate, the AI should fall back to direct URL construction. Most e-commerce category pages follow predictable URL patterns: Best Buy (`/site/[category-slug]`), Lowes (`/c/[category-name]/[numeric-id]` or `/pl/[path]/[id]`), Home Depot (`/b/[category]/[id]`). The AI can construct a direct navigate URL from the department and sub-category names. This bypasses the mega-menu entirely but achieves the user's goal of reaching the category page. Direct URL navigation should be the last resort, not the default -- mega-menu interaction is the skill being tested.
