Execute the SCROLL-07 edge case test using MCP manual tools following the panMapForListings workflow from the airbnb.js site guide. Test steps:

1. Use MCP navigate tool to go to airbnb.com/s/San-Francisco--CA/homes (or airbnb.com/s/New-York--NY/homes -- pick a major city with many listings). Record the URL used.

2. Use read_page to verify the search results page loaded with both listing cards and map visible. Check for:
   - Listing cards visible in the left panel ([data-testid="card-container"])
   - Map container visible on the right (div[data-testid="map/GoogleMap"] or div[aria-label*="Map"])
   - Listing pins visible on the map (button[data-testid="map/markers/BasePillMarker"])
   - Cookie/consent popup presence
   Record: page URL, listing card count, pin count, map container found (yes/no), cookie popup present.

3. Dismiss cookie/consent popup if present:
   - Look for div[data-testid="main-cookies-banner"] or section[aria-label*="cookie"]
   - Click the accept/OK button within the banner

4. BASELINE RECORDING (before pan):
   - Count listing pins on map: button[data-testid="map/markers/BasePillMarker"] count
   - Count listing cards in left panel: [data-testid="card-container"] count
   - Read 2-3 pin price labels for comparison after pan
   - Record: "Baseline: {pin_count} pins, {card_count} cards, sample prices: [$X, $Y, $Z]"

5. IDENTIFY MAP CENTER for drag:
   - Map is typically in the right half of the viewport
   - Map canvas center: approximately (viewport.width * 0.75, viewport.height * 0.5)
   - If viewport is 1920x1080: map center approximately (1440, 540)
   - Record the coordinates used for the drag

6. PAN THE MAP using CDP drag:
   - Execute: drag(startX=centerX+200, startY=centerY, endX=centerX-200, endY=centerY, steps=20, stepDelayMs=25) to pan eastward by ~400px
   - This should shift the map viewport significantly enough to show a different area
   - Record: drag parameters used, any error returned

7. WAIT FOR NEW LISTINGS:
   - Wait 2000-3000ms for Airbnb API to respond and new listing pins to render
   - Use wait_for_stable to detect when DOM changes have settled
   - Record: wait duration, DOM stability achieved (yes/no)

8. VERIFY NEW LISTINGS LOADED (after pan):
   - Count listing pins on map again: button[data-testid="map/markers/BasePillMarker"] count
   - Count listing cards in left panel again: [data-testid="card-container"] count
   - Read 2-3 pin price labels for comparison against baseline
   - Compare pin count, card count, and prices against baseline from step 4
   - Record: "After pan: {pin_count} pins, {card_count} cards, sample prices: [$A, $B, $C]"
   - Determine: did the listings change? (count changed OR prices changed OR both)

9. OPTIONAL SECOND PAN (if first pan succeeded or for additional data):
   - Pan in a different direction (e.g., southward: drag from center to above center)
   - Wait and verify again
   - This provides additional evidence of map pan working reliably

10. OUTCOME CLASSIFICATION:
    - PASS = map panned successfully via CDP drag AND new listing pins loaded in the new area (different pins/prices than baseline)
    - PARTIAL = map panned but new listings did not load (search-as-you-move may be off), OR panned and loaded but could not verify via DOM (selectors mismatch), OR WebSocket bridge disconnect blocked execution
    - FAIL = could not interact with the map at all (drag failed, no map canvas found, or page did not load)
    - Note: Airbnb browsing is public, no auth required -- SKIP-AUTH should not apply

Record every step outcome (tool used, target, result, notes) for the diagnostic report.
