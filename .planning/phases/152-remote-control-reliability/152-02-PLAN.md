<objective>
Harden remote-control debugger lifecycle so repeated toggles, dispatch failures, and streaming-tab changes do not strand the dashboard in a broken remote-control state.

Purpose: The current remote-control implementation assumes debugger reuse too loosely and does not actively retarget the debugger when the streaming tab changes. That leaves `click`, `type`, and `scroll` dead after attach errors or tab switches.
Output: Remote control tracks whether it is enabled, knows whether it owns the debugger attachment, retargets or detaches cleanly when the streaming tab changes, and can recover after dispatch failures.
</objective>

  <action>
Replace the implicit remote debugger lifecycle with an explicit state machine.

1. Add explicit remote-control enabled/disabled state and whether the remote-control path owns the current debugger attachment.
2. Make remote-control start idempotent for repeated toggles on the same streaming tab.
3. When the streaming tab changes while remote control is enabled, detach from the old tab if owned and reattach to the new stream target, or disable remote control cleanly if no valid target remains.
4. On stop, only detach when remote control actually owns the debugger attachment so other debugger users in the extension are not broken accidentally.
5. Record enough console-level lifecycle evidence to distinguish start, reuse, retarget, detach, and failure cases during debugging.

<task type="auto">

  <files>background.js</files>
  <read_first>background.js (handleRemoteClick, handleRemoteKey, handleRemoteScroll)</read_first>
  <action>
Make remote input dispatch recoverable instead of permanently wedging remote control after one failure.

1. Route remote click/key/scroll dispatch through a shared ensure-or-reattach helper before sending CDP commands.
2. On dispatch failure, clear or repair only the remote-control debugger state that is actually invalid, then allow the next interaction to re-establish a session automatically if possible.
3. Treat missing stream target, closed tabs, and debugger-detached conditions as recoverable state transitions rather than silent dead ends.
4. Keep the failure path bounded; do not loop endlessly on repeated attach failures.
5. Preserve existing `console.warn` / `console.error` visibility so debugging still has concrete evidence.
  </action>
