Execute the CONTEXT-02 edge case test using MCP manual tools following the forkAndEditCell workflow from the observable.js site guide. Test steps:

1. Use MCP navigate tool to go to a public Observable notebook. Primary target: https://observablehq.com/@observablehq/five-minute-introduction (Observable's own introductory notebook -- public, well-structured, multiple cells with data). Fallback targets: https://observablehq.com/@d3/gallery or any public notebook with at least 3 cells containing data arrays. Record the URL used and response status.

2. Use read_page to verify the notebook page loaded. Check for:
   - Notebook title visible
   - Cell containers present in the DOM (div.observablehq--cell or similar)
   - At least 3 cells exist
   - Cell editors (CodeMirror: div.cm-editor) present or revealable by clicking
   - Cell output areas rendering content
   Record: page URL, number of cells found, cell container selector that matched, whether editors are visible or hidden.

3. Dismiss any cookie consent banners or notification prompts if present.

4. COUNT AND IDENTIFY CELLS:
   - Use get_dom_snapshot to enumerate all cell containers
   - Count total cells to confirm at least 3 exist
   - Identify cell 1 (first container) and cell 3 (third container) by DOM position
   - For cell 3: read its current code content to identify the data array to modify
   - Record: total cell count, cell 1 position confirmed, cell 3 position confirmed, cell 3 current content

5. CAPTURE CELL 1 BASELINE:
   - Use read_page or getText to extract cell 1's visible output text
   - Store this text as the "before" state
   - Record: cell 1 output text (first 200 characters), method used to extract

6. CAPTURE CELL 3 BASELINE:
   - Read cell 3 current code and output
   - Identify the specific data value to modify (e.g., a data array like [1, 2, 3])
   - Record: cell 3 code content, cell 3 output text, data value identified for modification

7. ATTEMPT FORK:
   - Click the fork button
   - If a sign-in modal/prompt appears: document as SKIP-AUTH for forking
   - If fork succeeds: document the forked notebook URL
   - Record: fork attempted (yes/no), result (succeeded/skip-auth/failed), reason

8. TINKER MODE (if fork requires auth):
   - Click directly on cell 3 to reveal/open its CodeMirror editor
   - Verify the cm-editor element becomes visible and focusable
   - Record: tinker mode available (yes/no), editor revealed (yes/no)

9. EDIT CELL 3:
   - Click inside the cm-content area of cell 3's editor to focus
   - Use Ctrl+A (select all) to select existing code
   - Use type_text to type the modified data (e.g., change [1, 2, 3] to [10, 20, 30] or modify any data array in the cell)
   - Record: edit attempted (yes/no), type_text sent (content), editor focused (yes/no)

10. WAIT FOR REACTIVE UPDATE:
    - Wait 2-3 seconds or use wait_for_stable for Observable's reactive runtime to re-evaluate
    - Record: wait duration, any visible loading/updating indicators

11. VERIFY CELL 3 CHANGED:
    - Use read_page or getText to read cell 3's output after edit
    - Confirm the output reflects the new data values
    - Record: cell 3 new output text, matches expected change (yes/no)

12. VERIFY CELL 1 UNCHANGED:
    - Use read_page or getText to capture cell 1's output text
    - Compare against the baseline captured in step 5
    - Record: cell 1 output text after edit, matches baseline (yes/no/partially)

13. OUTCOME CLASSIFICATION:
    - PASS = forked notebook (or used tinker mode) AND edited cell 3 data AND cell 3 output changed AND cell 1 output verified unchanged
    - PARTIAL = accessed notebook and identified cells but cell editing failed (CodeMirror interaction issue), OR cell 3 edited but cell 1 verification could not be performed, OR notebook accessed but WebSocket bridge disconnect prevented interaction
    - FAIL = could not access Observable notebook or no cells found
    - SKIP-AUTH = fork requires authentication AND tinker mode also not available
