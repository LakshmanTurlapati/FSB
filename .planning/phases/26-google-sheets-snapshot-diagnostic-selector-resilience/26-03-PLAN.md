# Plan 26-03: Thread guideSelectors to buildMarkdownSnapshot

**Objective:** Fix the wiring gap where `background.js` calls `getMarkdownSnapshot` without passing the site guide's `selectors` and `fsbElements`, making the entire Phase 26 selector resilience pipeline dead code.

## Root Cause

`buildMarkdownSnapshot` accepts `guideSelectors` but background.js never passes it. The site guide is resolved in `ai-integration.js:2274` (inside `buildPrompt`) which runs AFTER the markdown snapshot is already fetched. The content script has no access to site guides (they're only loaded in the service worker).

### Fix 1: Resolve guide selectors before getMarkdownSnapshot calls

The fix needs to get the site guide's selectors object at two callsites where `getMarkdownSnapshot` is sent to the content script. Since `getGuideForTask` is available in the background.js scope (imported via `site-guides/index.js` at line 16), we can call it directly.

**Callsite 1: Automation loop (line ~8548-8553)**

Before the `getMarkdownSnapshot` message at line 8550, resolve the site guide using the current tab URL which is already available as `currentUrl` (defined around line 8987 from `tabInfo.url`). However, this code block is at line 8548, which is BEFORE `currentUrl` is defined at line 8987. So we need to get the URL from `tabInfo` or `domResponse`.

Look at where `currentUrl` is first available in the loop. The tab URL can be obtained from the already-fetched `domResponse.structuredDOM.url` or by checking the session. Actually, check lines around 8900-8990 — `currentUrl` is defined there from `tabInfo.url`. The `getMarkdownSnapshot` call at 8548 runs before that.

Instead, use the tab URL that's available from the DOM response itself, or resolve it inline:

```javascript
// At line ~8548, before the getMarkdownSnapshot call:
// Resolve site guide selectors for fsbElements injection
let mdGuideSelectors = null;
if (typeof getGuideForTask === 'function') {
  try {
    const tabForGuide = await chrome.tabs.get(session.tabId);
    const guide = getGuideForTask(session.task, tabForGuide?.url);
    if (guide) {
      mdGuideSelectors = { ...guide.selectors, fsbElements: guide.fsbElements };
    }
  } catch (e) {
    // Tab lookup failed, proceed without guide selectors
  }
}
```

Then update the message at line 8552:
```javascript
// BEFORE:
options: { charBudget: 12000, maxElements: 80 }
// AFTER:
options: { charBudget: 12000, maxElements: 80, guideSelectors: mdGuideSelectors }
```

**Callsite 2: prefetchDOM (line ~1272-1274)**

The `prefetchDOM` function only has `tabId` and `options`. Add an optional `guideSelectors` parameter to `prefetchDOM`:

```javascript
// BEFORE (line 1253):
async function prefetchDOM(tabId, options = {}) {
// AFTER:
async function prefetchDOM(tabId, options = {}, guideSelectors = null) {
```

Then at line 1274:
```javascript
// BEFORE:
options: { charBudget: 12000, maxElements: 80 }
// AFTER:
options: { charBudget: 12000, maxElements: 80, guideSelectors }
```

Update both `prefetchDOM` callsites (lines ~9160 and ~9354) to pass guideSelectors. At those points, `currentUrl` IS available (defined at ~8987). Cache the resolved guide selectors in a variable at the top of the iteration block so both the main callsite and prefetch can use it:

```javascript
// Around line 8987 (after currentUrl is defined), add:
let iterationGuideSelectors = null;
if (typeof getGuideForTask === 'function') {
  const guide = getGuideForTask(session.task, currentUrl);
  if (guide) {
    iterationGuideSelectors = { ...guide.selectors, fsbElements: guide.fsbElements };
  }
}
```

Wait — there's a sequencing problem. The `getMarkdownSnapshot` at line 8548 runs BEFORE `currentUrl` is defined at line 8987. So for the main callsite, we need the inline `chrome.tabs.get` approach shown above. For prefetch calls (lines 9160, 9354), we can use `iterationGuideSelectors` since they run after `currentUrl` is available.

Actually, simplify: resolve the guide selectors ONCE early in the iteration, right after we have the tab. Check where `session.tabId` is first available and add the resolution there. The session object is available from line 8112. Add the guide resolution right before the DOM fetch block (around line 8340):

```javascript
// Before DOM fetch block, resolve guide for this iteration
let iterationGuideSelectors = null;
if (typeof getGuideForTask === 'function') {
  try {
    const guideTab = await chrome.tabs.get(session.tabId);
    const guide = getGuideForTask(session.task, guideTab?.url);
    if (guide) {
      iterationGuideSelectors = { ...guide.selectors, fsbElements: guide.fsbElements };
    }
  } catch (e) {
    // Tab inaccessible, proceed without
  }
}
```

Then use `iterationGuideSelectors` at ALL THREE callsites:
1. Line 8552: `options: { charBudget: 12000, maxElements: 80, guideSelectors: iterationGuideSelectors }`
2. Line 9160: `pendingDOMPrefetch = prefetchDOM(session.tabId, { ... }, iterationGuideSelectors);`
3. Line 9354: `pendingDOMPrefetch = prefetchDOM(session.tabId, { ... }, iterationGuideSelectors);`

NOTE: The `guide.selectors` is the flat selector map (e.g., `{nameBox: '#t-name-box', ...}`). The `guide.fsbElements` is the multi-strategy config. Both need to be passed because `buildMarkdownSnapshot` uses `guideSelectors.fsbElements` for Stage 1b injection AND passes the whole `guideSelectors` object to `buildGuideAnnotations` for annotation matching. Merging them with spread (`{ ...guide.selectors, fsbElements: guide.fsbElements }`) preserves both.

Handle the case where `guide.fsbElements` is undefined (most site guides don't have it) — this is fine, the spread will just not include it, and `buildMarkdownSnapshot` checks `guideSelectors?.fsbElements` which will be undefined/null.
