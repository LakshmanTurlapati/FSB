## Context Bloat Analysis

### Estimated Context Per Batch Cross-Reference Cycle

Based on the CONTEXT-09 workflow (extract 10 names from CRM, switch to HR portal, cross-reference batch, switch back):

- **Step 1 (navigate to CRM + initial read_page):** ~5-15KB (page structure, table headers, first 10 rows of employee data). DemoQA's react-table renders approximately 10 rows per page by default.
- **Step 2 (extract batch of 10 names from CRM):** ~1-3KB (10 name records at ~30-50 chars each from table cell content). With targeted column extraction: 10 x `"Cierra Vega"` = ~150 chars. With full row content: 10 x ~200 chars = ~2KB.
- **Step 3 (open_tab to HR portal):** ~1-2KB (open_tab result with tabId, initial page structure)
- **Step 4 (read_page on HR portal):** ~3-8KB (full table content from both table1 and table2). Herokuapp server HTML is 5,808 bytes total. Table data portion is approximately 2-3KB of employee records.
- **Step 5 (cross-reference 10 names against HR rows):** ~0.5-1KB per name comparison. Agent scans HR table rows for matching firstName+lastName. Total for 10-name batch: ~5-10KB if reading HR table per name, or ~0.5KB if HR data is cached from step 4.
- **Step 6 (switch_tab back to CRM):** ~0.5-1KB (switch confirmation, CRM page verification)
- **Step 7 (advance to next page or scroll for next batch):** ~1-2KB (click .-next button, wait for page load, verify new rows)
- **Step 8 (store batch results):** ~0.65KB per batch (10 names x 65 chars per compact record)

**Per-batch cycle (steps 2-8):** ~12-35KB without optimization, ~3-8KB with targeted extraction and HR data caching.

### Compact Result Tracking: Under 3,000 Characters for 50 Names

The cross-reference result format specified in crm-hr-cross-ref.js:

```
Full format (65 chars/record):
{name: "Emily Johnson", crmFound: true, hrFound: false, match: false}
50 records = 3,250 chars (slightly over budget)

Compact format (40 chars/record):
{n:"Emily Johnson",c:1,h:0,m:0}
50 records = 2,000 chars (well within budget)

Minimal format (25 chars/record):
Emily Johnson:CRM-only
50 records = 1,250 chars (minimal overhead)
```

Recommendation: Use the compact format (`{n, c, h, m}`) for autopilot tracking. The full verbose format is useful for debugging but unnecessary during automated execution. The minimal format loses structured queryability but is the most context-efficient.

### Multi-Tab Switching Overhead

Each tab switch involves:
- switch_tab tool call: ~0.5KB (request + response with tab confirmation)
- Verification read after switch: ~1-3KB (confirm correct tab is active, page state preserved)
- Total per switch: ~1.5-3.5KB

For 5 batches of 10 names with 2 switches per batch (CRM->HR, HR->CRM):
- 10 switch_tab calls x ~2KB average = ~20KB total switching overhead
- This is constant overhead regardless of batch size or name count

## Autopilot Recommendations

1. **Extract employee names from CRM in batches of 10, not all 50 at once, to manage context consumption.** Each batch read from the CRM table consumes approximately 1-3KB of context (10 rows x 30-50 chars of name data per row). Reading all 50 names at once would require either a single massive DOM read (~15-30KB) or scrolling through multiple pages while retaining all previous data. The batch approach caps per-cycle context at ~3-8KB and allows compact result storage between batches. If the CRM table has pagination (like DemoQA's .-next button), click the next page button after extracting each batch of 10.

2. **Store cross-reference results in compact format: {n:"FirstName LastName", c:1, h:0, m:0} per name -- never retain full DOM snapshots from either tab between batches.** Each compact record is approximately 40 characters. For 50 names, the total result tracking overhead is ~2,000 characters -- well under the 3,000-character budget from crm-hr-cross-ref.js. After each batch cross-reference, discard the raw table data from both tabs and retain only the compact result records. The accumulated compact results are the only data that persists across batch cycles.

3. **Use list_tabs immediately after setup to confirm tab IDs before any switch_tab calls.** Call list_tabs after navigating to the CRM and after open_tab for the HR portal to record {crmTabId, hrTabId}. These IDs are used for all subsequent switch_tab calls. Verify both IDs are present and the correct URLs are associated with each. If a tab closes unexpectedly during the workflow, list_tabs will reveal the missing tab so the autopilot can re-open it rather than failing silently on switch_tab.

4. **Follow the sequential open-and-read pattern: extract batch from CRM tab, switch to HR tab, cross-reference, switch back.** Do NOT open both tabs and then try to read from them in an interleaved fashion. The sequential pattern ensures: (a) you know which tab you are on at all times, (b) you complete one batch's cross-reference fully before advancing to the next, (c) context from the previous tab is naturally replaced by the current tab's content. Each batch cycle has exactly 2 switch_tab calls: CRM->HR and HR->CRM.

5. **Handle CRM pagination by clicking the .-next button (DemoQA) or using ?skip=N&limit=10 (DummyJSON API) to access all 50 names across multiple pages.** DemoQA's react-table shows 10 rows per page by default. After extracting the first 10 names, click the .-next button to load the next page of results. Repeat until 50 names have been extracted (5 pages of 10). For DummyJSON API fallback, use the query parameter pagination: /users?limit=10&skip=0, /users?limit=10&skip=10, etc. Always verify the new page has loaded (new row content) before extracting names.

6. **Deduplicate processed names using a lowercase-normalized Set to prevent re-checking names across pagination overlaps or table reloads.** Maintain a Set of already-cross-referenced names: `processedNames.add(name.trim().toLowerCase())`. Before cross-referencing a name, check if it is already in the Set. This prevents duplicate lookups when: (a) CRM pagination overlaps (some tables show the last row of the previous page as the first row of the next), (b) the HR portal is re-read and the same names are scanned again, (c) employee names appear in both table1 and table2 on Herokuapp (which has duplicate data across its two tables).

7. **If the HR portal has a search feature, use it (type_text into search box, read filtered results); if it is a static table, scan all rows once and cache the full name list.** Herokuapp's tables have no search functionality -- all 8 rows are visible on page load. The optimal strategy is: read HR portal once during setup, cache the 4 unique names as a Set, and compare each CRM batch against the cached Set without re-reading the HR DOM. This eliminates repeated HR table reads (saving ~3-8KB per batch cycle). If the HR portal were a dynamic application with search (like DemoQA's #searchBox), type each name into the search box and check if a matching row appears -- more efficient for large HR datasets.

8. **If a tab becomes unresponsive after switch_tab (no DOM content, error page, or stale data), re-navigate to the original URL rather than retrying the switch.** Tab state can degrade during extended multi-tab workflows: the service worker may disconnect, a page may enter the BF cache, or JavaScript execution may pause in background tabs. If switch_tab succeeds but read_page returns unexpected content, call navigate with the original URL (e.g., demoqa.com/webtables) to force a fresh page load. This is more reliable than repeated switch_tab attempts. Re-navigation costs ~2-5KB of context but recovers from tab state corruption.

9. **Build the final cross-reference report incrementally after each batch, not at the end from accumulated memory.** After each batch cross-reference, append the batch results to the running report: "Batch 1: 10 names processed, 0 matched, 10 CRM-only. Batch 2: 10 names processed, 1 matched, 9 CRM-only." This running summary (under 200 chars per batch) means the final report is already built by the time all 5 batches complete. Do not attempt to reconstruct the full cross-reference from 50 individual compact records at the end -- that requires re-reading all accumulated tracking data.
