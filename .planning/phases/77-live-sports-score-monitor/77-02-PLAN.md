Execute the CONTEXT-01 edge case test using MCP manual tools following the monitorLiveScores workflow from the live-scores.js site guide. Test steps:

1. Use MCP navigate tool to go to https://www.espn.com/ (primary target -- main scoreboard with multi-sport ticker). If ESPN homepage does not show clear scoreboard data, try sport-specific page: https://www.espn.com/nba/scoreboard or https://www.espn.com/mlb/scoreboard (pick whichever sport is currently in-season). Record the URL used and response status.

2. Use read_page to verify the scoreboard page loaded. Check for:
   - Game containers visible with team names and scores
   - At least one game with scores displayed (live, final, or upcoming)
   - Game status indicators present (LIVE, FINAL, time remaining)
   - Score values extractable as numeric text
   Record: page URL, number of games visible, which sport(s), any games currently live.

3. Dismiss any cookie consent banners, ad overlays, or notification permission prompts if present.

4. INITIAL SNAPSHOT:
   - Use read_page or get_dom_snapshot to capture all visible game scores
   - For each game: extract team names, scores, game status, period/quarter/clock
   - Record as structured snapshot: { gameId: "TeamA-vs-TeamB", team1: "TeamA", score1: X, team2: "TeamB", score2: Y, status: "LIVE/FINAL/etc", period: "3rd Qtr" }
   - Record: number of games in snapshot, how many are live vs final vs upcoming

5. POLLING LOOP (attempt sustained monitoring):
   - Wait 30-60 seconds between polls
   - Use read_page to extract current game scores
   - Compare current scores against previous snapshot
   - For each change detected: log timestamp, game, old score, new score, status
   - Also detect status changes (upcoming->live, live->final)
   - Replace previous snapshot with current (only keep 2 snapshots at a time)
   - Continue for as many cycles as feasible (target: 30 minutes / ~30-60 cycles)
   - Record: number of cycles completed, total changes detected, elapsed time

6. IF NO LIVE GAMES AVAILABLE:
   - Check multiple sports: NBA, MLB, NHL, NCAA (depending on season)
   - If only final/upcoming games: document that no live games were in progress
   - Take 2-3 snapshots to verify scores of completed games remain stable
   - Outcome can still be PARTIAL if the polling mechanism works but no changes occur

7. IF ESPN PAGE IS CLIENT-RENDERED WITH NO VISIBLE SCORES:
   - Check if scores are in __NEXT_DATA__ or window.__espnfitt__ JSON
   - Try fallback: navigate to https://www.cbssports.com/scores/
   - Or try https://www.nba.com/games
   - Document what was found and what is missing

8. CONTEXT BLOAT ASSESSMENT (critical for CONTEXT category):
   - Record cumulative context size after each polling cycle (approximate)
   - Note if/when context grows too large for continued polling
   - Document how many polling cycles were feasible before context pressure
   - Assess whether the 2-snapshot strategy (current + previous only) is sufficient
   - This is the key difference from previous phases: sustained operation vs one-shot

9. OUTCOME CLASSIFICATION:
   - PASS = navigated to scoreboard AND extracted game scores AND polled for 30 minutes AND detected at least one score change AND generated change log
   - PARTIAL = accessed scoreboard and extracted scores but could not sustain 30-minute polling (WebSocket bridge disconnect, context bloat, or no live games with score changes), OR polling worked but no score changes occurred during monitoring window
   - FAIL = could not access scoreboard at all or no game scores extractable
   - SKIP-AUTH = scoreboard page requires authentication

Record every step outcome (tool used, target, result, notes) for the diagnostic report.
