## E2E Flow Trace

### Flow 1 — First-visit user, `Accept-Language: ja`
- GET `/` reaches Express → `accept-language` middleware (`server.js:171`).
- Header present + no `fsb-locale` cookie → `pickBestLocale` → `'ja'` (alias map covers `ja`).
- `res.redirect(302, '/ja/')` → static prerender at `dist/.../ja/index.html`.
- Prerender carries `<html lang="ja">` (Angular i18n compiler), Japanese trans-units (`messages.ja.xlf` 420/420 translated), and 7-tag hreflang + canonical via `emitLocaleHead` (verify:hreflang 301/0).
- **VERDICT: COMPLETE.** Live smoke in 267 SUMMARY: `GET /` + `Accept-Language: ja` → `302 -> /ja/`.

### Flow 2 — Picker-driven switch then bare `/`
- User clicks picker → `LanguagePickerComponent` → `LocaleService.setLocale(code)`.
- `locale.service.ts:76` writes `document.cookie = 'fsb-locale=<code>; max-age=...; path=/; samesite=lax'`.
- Next bare `/` request: middleware reads `req.headers.cookie`, `parseCookieHeader` extracts `fsb-locale`, value ∈ `supportedSet` → middleware calls `next()` (no redirect). Cookie wins per 267 SUMMARY live smoke ("Accept-Language: ja + Cookie: fsb-locale=de → no redirect").
- **Caveat (WARNING-02):** when cookie value is a NON-en supported locale and middleware does nothing, Express then serves whatever static `/index.html` exists at root (the EN prerender). The picker itself navigates the SPA to `/<locale>/`, but a fresh-tab bare `/` honoring the cookie does NOT 302 to `/<locale>/` — middleware only ever redirects when there is no cookie. Per CONTEXT D-02 / T-267-03 this is the locked behavior (cookie path is read-only; "cookie wins" semantics = short-circuit, not re-redirect). Document as design lock, not a bug.
- **VERDICT: COMPLETE** for the explicit locked behavior; flag noted.

### WARNINGS

**WARNING-01 — Duplicated supported-locale list in middleware wiring (mild ROUTE-02 / ROUTE-03 drift risk)**
- `server.js:171-176` passes a hardcoded literal `supported: ['es', 'de', 'ja', 'zh-CN', 'zh-TW']` to the middleware factory rather than deriving from `showcase/server/src/utils/locale-constants.js` (the CJS registry that 261 made canonical).
- Today this list matches the registry (excluding `en`, the source locale, which is correct), but it bypasses the `verify-locale-sync.mjs` parity check at runtime: a future locale add that updates both registries would still leave this hardcoded literal stale, and `verify-locale-sync` would NOT catch it (the script only diffs the two `locale-constants.*` files).
- **Suggested follow-up (v0.9.64+):** import `LOCALES`/`SOURCE_LOCALE` from `./src/utils/locale-constants` in `server.js` and derive `supported = LOCALES.filter(l => l !== SOURCE_LOCALE)`. Zero behavior change today.

**WARNING-02 — Picker-cookie + bare `/` does not actively redirect (design lock, but counterintuitive)**
- 267 middleware short-circuits (`next()`) when `fsb-locale` cookie is present, even if the cookie says `ja` and the request is for `/`. Result: a returning user with `fsb-locale=ja` who lands on bare `/` (e.g. shared link) sees the EN prerender at root, not `/ja/`. This is locked per CONTEXT D-02 / T-267-03 ("cookie path is read-only").
- The picker's SPA navigation handles the in-session case; this is only relevant for fresh-tab/shared-link returns.
- **Suggested follow-up (defer to a future phase only if telemetry shows it):** flip cookie-precedence semantics from "skip middleware" to "redirect to cookie locale" — opens T-267-02 redirect-loop surface, needs explicit `/{locale}/` skip guard. Not in this milestone's scope.

**WARNING-03 — Process gap: no VERIFICATION.md files for any phase in this milestone**
- `261-i18n-scaffold/`, `262-marketing-strings/`, `264-per-locale-bootstrap/`, `265-translator-fill/`, `266-verification-baseline/`, `267-accept-language-detection/` all ship CONTEXT/PLAN/SUMMARY but no `VERIFICATION.md`. Phase SUMMARYs embed self-check sections that perform the verification role, and CI runs are linked, so the audit trail exists — but the milestone's own template normally expects a separate VERIFICATION.md file per phase.
- **Suggested follow-up:** either backfill VERIFICATION.md per phase using the CI run IDs already in SUMMARYs (cheap), or update the milestone template to acknowledge SUMMARY-embedded self-check as sufficient.
