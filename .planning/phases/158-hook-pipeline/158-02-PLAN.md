**Create ai/hooks/safety-hooks.js (HOOK-02):**

Per D-03: Thin wrappers around existing functions. The original checkSafetyBreakers and detectStuck remain in agent-loop.js independently testable. These factories receive the functions via closure -- they do NOT import agent-loop.js.

```javascript
// Factory: createSafetyBreakerHook(checkSafetyBreakersFn)
// Returns a handler function suitable for pipeline.register(LIFECYCLE_EVENTS.AFTER_ITERATION, handler)
```

`createSafetyBreakerHook(checkSafetyBreakersFn)`:
- Returns a function `safetyBreakerHandler(context)` where context has `{ session }`.
- Calls `checkSafetyBreakersFn(context.session)`.
- If result.shouldStop is true, returns `{ shouldStop: true, reason: result.reason, source: 'safetyBreaker' }`.
- If not, returns `{ shouldStop: false }`.
- Wrapped in try/catch per D-05: on error, console.warn and return `{ shouldStop: false }`.

`createStuckDetectionHook(detectStuckFn)`:
- Returns a function `stuckDetectionHandler(context)` where context has `{ session, toolResults }`.
- Calls `detectStuckFn(context.session, context.toolResults)`.
- If result.isStuck is true, returns `{ shouldStop: false, isStuck: true, hint: result.hint, source: 'stuckDetection' }`.
  Note: stuck detection does NOT stop the pipeline -- it provides a hint. The agent loop reads the hint and injects it into the next prompt. Only safety breakers return shouldStop: true.
- If not stuck, returns `{ shouldStop: false, isStuck: false }`.
- Wrapped in try/catch per D-05.

Module boilerplate: 'use strict', var declarations, function/prototype is not needed (these are factory functions, not classes), typeof module guard exporting `{ createSafetyBreakerHook, createStuckDetectionHook }`. 2-space indent, single quotes, semicolons.

**Create ai/hooks/permission-hook.js (HOOK-03):**

`createPermissionHook(permissionContext)`:
- Takes a PermissionContext instance via closure.
- Returns a function `permissionHandler(context)` where context has `{ toolName, origin }`.
- Calls `permissionContext.isAllowed(context.toolName, context.origin || '')`.
- If NOT allowed:
  - Calls `permissionContext.createDenial(context.toolName)`.
  - Returns `{ shouldStop: false, denied: true, denial: denialResult }`.
  - The denial object shape: `{ denied: true, toolName: string, reason: string }`.
  - Phase 159 will use this denial to construct a tool_result error for the AI.
- If allowed: returns `{ shouldStop: false, denied: false }`.
- Wrapped in try/catch per D-05: on error, console.warn and return `{ shouldStop: false, denied: false }` (fail-open -- a buggy permission check must not block automation).

Module boilerplate: 'use strict', var declarations, typeof module guard exporting `{ createPermissionHook }`. 2-space indent, single quotes, semicolons.

Create ai/hooks/progress-hook.js (HOOK-04).

Per D-04: This phase creates the progress hook handler only. Phase 159 replaces the 10+ scattered sendStatus calls in agent-loop.js with hook emissions. This module is the handler that will receive those emissions.

`createProgressHook(emitter)`:
- Takes a SessionStateEmitter instance via closure.
- Returns a function `progressHandler(context)` where context varies by lifecycle event:
  - For afterToolExecution: `{ sessionId, toolName, toolResult, iteration }`.
    Emits `STATE_EVENTS.TOOL_EXECUTED` via the emitter with `{ sessionId, toolName, success: !toolResult.error, iteration, timestamp: Date.now() }`.
  - For afterIteration: `{ sessionId, iteration, totalCost, inputTokens, outputTokens }`.
    Emits `STATE_EVENTS.ITERATION_COMPLETE` via the emitter with `{ sessionId, iteration, cost: totalCost, inputTokens, outputTokens, timestamp: Date.now() }`.
  - For onCompletion: `{ sessionId, result, totalCost, iterations }`.
    Emits `STATE_EVENTS.SESSION_ENDED` via the emitter with `{ sessionId, result, totalCost, iterations, timestamp: Date.now() }`.
  - For onError: `{ sessionId, error, iteration }`.
    Emits `STATE_EVENTS.ERROR_OCCURRED` via the emitter with `{ sessionId, error: (error.message || String(error)), iteration, timestamp: Date.now() }`.
- The handler inspects `context.event` (set by the caller or the pipeline) to determine which STATE_EVENT to emit. If context.event is not recognized, emit nothing (no-op).
- Always returns `{ shouldStop: false }` (progress hooks never halt, per D-05).
- Wrapped in try/catch per D-05: on error, console.warn and return `{ shouldStop: false }`. A buggy progress hook must never kill automation.

Alternative simpler design: Instead of one handler checking context.event, export 4 focused factory functions:
- `createToolProgressHook(emitter)` -- for afterToolExecution
- `createIterationProgressHook(emitter)` -- for afterIteration
- `createCompletionProgressHook(emitter)` -- for onCompletion
- `createErrorProgressHook(emitter)` -- for onError

Use the simpler 4-function approach. Each is a focused factory returning a single handler. Cleaner registration in Phase 159.

Module boilerplate: 'use strict', var declarations, typeof module guard exporting `{ createToolProgressHook, createIterationProgressHook, createCompletionProgressHook, createErrorProgressHook }`. 2-space indent, single quotes, semicolons.
