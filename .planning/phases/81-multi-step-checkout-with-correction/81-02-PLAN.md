   Record: product name, add-to-cart success

3. VIEW CART AND PROCEED TO CHECKOUT:
   - Navigate to cart page (click cart icon or Cart link)
   - Verify at least 1 item in cart with visible price
   - Click "Proceed to Checkout" or equivalent
   Record: cart page loaded, item count, subtotal visible

4. HANDLE AUTH GATE (if present):
   - If redirected to login: check for "Guest Checkout" or "Continue as Guest" option
   - If guest checkout available: select it and continue
   - If no guest option and saucedemo.com: login with standard_user / secret_sauce
   - If no guest option on any target: document as SKIP-AUTH

5. FILL SHIPPING FORM -- WRONG ZIP:
   - Fill name: "Test User" (or first name "Test", last name "User")
   - Fill address: "123 Test Street"
   - Fill city: "Anchorage"
   - Fill state: "Alaska" or select "AK" from dropdown
   - Fill country: "United States" or select from dropdown
   - Fill zip/postal code: "99501" (Alaska -- this is the INTENTIONALLY WRONG zip)
   - Fill any other required fields (email: test@example.com, phone: 5551234567)
   Record: each field filled, zip value entered

6. SUBMIT SHIPPING WITH WRONG ZIP:
   - Click Continue/Submit/Next to advance past shipping form
   - Wait for order summary or next checkout step to load via wait_for_stable
   Record: form submitted successfully, next step loaded

7. CAPTURE PRE-CORRECTION TAX:
   - Use read_page with order summary selectors to find: subtotal, tax amount, shipping cost, total
   - If tax not visible on current step: advance to review/payment step
   - Store as pre_correction = {zip: "99501", subtotal: "$X", tax: "$Y", total: "$Z"}
   - If no tax line exists: note "no tax displayed" and check if it appears at a later step
   Record: subtotal, tax, shipping, total values (or "not displayed" for each)

8. NAVIGATE BACK TO EDIT ZIP:
   - Look for "Edit" or "Change" link near the shipping address
   - If no edit link: use browser back button or navigate directly to shipping step URL
   - Verify zip field is accessible for editing
   Record: navigation method used, zip field accessible (yes/no)

9. CORRECT THE ZIP CODE:
   - Use clear_input on the zip code field to remove "99501"
   - Use type_text to enter "10001" (New York, NY)
   - Update city field: clear_input then type_text "New York"
   - Update state field: select "New York" or "NY" from dropdown (or clear_input + type_text)
   - If form reset all fields on back navigation: re-fill all required fields with New York address
   Record: clear_input success on zip, type_text success with "10001", other fields updated

10. RESUBMIT SHIPPING WITH CORRECT ZIP:
    - Click Continue/Submit/Next again
    - Wait for order summary to load via wait_for_stable
    Record: form resubmitted successfully

11. CAPTURE POST-CORRECTION TAX:
    - Use read_page with same order summary selectors
    - Store as post_correction = {zip: "10001", subtotal: "$X", tax: "$Y2", total: "$Z2"}
    Record: subtotal, tax, shipping, total values

12. COMPARE TAX VALUES:
    - Compare pre_correction.tax vs post_correction.tax
    - Expected: they differ (Alaska 0% vs New York ~8.875%)
    - If they are the same: check if the store has flat tax or no zip-based tax variation
    - If no tax is displayed at all: note that the demo store does not calculate tax
    Record: tax comparison result, values match or differ, explanation

13. OUTCOME CLASSIFICATION:
    - PASS = checkout navigated, wrong zip entered, pre-correction tax captured, zip corrected via clear_input + type_text, post-correction tax captured, tax values differ
    - PARTIAL = checkout reached but one of: zip correction did not work, tax not displayed, tax did not change, form reset lost data, or WebSocket bridge disconnect prevented interaction
    - FAIL = could not reach checkout form or could not enter zip at all
    - SKIP-AUTH = all target stores require authentication with no guest checkout
