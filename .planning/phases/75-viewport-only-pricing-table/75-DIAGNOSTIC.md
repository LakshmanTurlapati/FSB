## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-75):** The MCP server process runs on port 7225 with an established TCP connection, but the Chrome extension side returns HTTP 426 "Upgrade Required" for browser action dispatch. This has blocked every live MCP test since Phase 55. Without the bridge, no MCP tool can execute against the live browser DOM. The scroll-read-deduplicate loop -- the core test for SCROLL-09 -- requires live browser execution.
- **Notion uses CSS Module classes, not standard table/ARIA patterns:** The pricing-table.js site guide's generic selectors assume HTML table elements or ARIA grid roles. Notion's pricing grid uses React component classes (PricingGrid_*, PlanFeatures_*) with deployment-specific hashes. The site guide needs CSS Module class prefix patterns (class*="PricingGrid_row__", class*="PricingGrid_section__") alongside generic selectors.
- **No explicit cross/not-included indicator:** Notion uses absence of checkmark to indicate "not included" rather than an explicit cross icon. The site guide's crossIndicator selectors (svg[class*="cross"], aria-label*="not included") do not match any Notion DOM elements. Autopilot needs to detect "empty cell" as the not-included signal.
- **Checkmark detection relies on class name, not aria-label:** Notion's checkmarks have no aria-label attribute. Detection must use the PlanFeatures_check__ class prefix on the SVG wrapper span element. The site guide's aria-label-based checkmark selectors do not match.
- **Server-rendered content means scroll may not be needed for extraction:** Since all 58 rows exist in server HTML, a read_page or get_dom_snapshot call without scrolling might capture all feature rows. The viewport-gating behavior (whether rows are truly lazy-loaded vs just off-screen) can only be confirmed in a live browser. If rows are NOT lazy-loaded, the scroll-read-deduplicate loop is unnecessary for Notion (though still important for other SaaS sites that do use lazy loading).
- **CSS Module hash instability:** The class hashes (e.g., PricingGrid_row__bHvcz) change between Notion deployments. Selectors must use partial matching (class*="PricingGrid_row__") to survive hash changes while maintaining specificity.
- **Screenshot/visual comparison not available:** For verifying that the comparison grid is visually rendered correctly (sticky header visible, rows aligned with plan columns), a screenshot tool would provide definitive evidence. Current MCP tools rely on DOM text extraction only.

## Autopilot Recommendations

1. **Notion pricing page is fully server-rendered -- all 58 feature rows exist in initial HTML.** Unlike TikTok (Phase 74), Twitter/X (Phase 67), and most SPAs tested in previous phases, Notion's pricing comparison grid is rendered server-side by Next.js SSR. A single read_page or get_dom_snapshot call may capture all feature rows without any scrolling. Autopilot should first attempt a full-page read before initiating the scroll-read-deduplicate loop.

2. **Scroll past hero pricing cards first to locate the comparison grid.** The page layout places hero pricing cards (PricingCardGroup_cardsGrid__BgVV6) at the top. The comparison grid (PricingGrid_pricingGrid__eqybz) is below the fold. Use scroll(direction="down", amount=600) 1-2 times to bring the comparison grid into viewport before attempting row extraction.

3. **Use smaller scroll increments (400-600px) to avoid skipping feature rows.** Each feature row is approximately 40-60px tall. A 500px scroll reveals 8-12 new rows per cycle. Larger scrolls risk skipping rows in viewport-gated tables where lazy rendering depends on IntersectionObserver thresholds.

4. **Feature name text is the unique deduplication key.** Each of the 58 rows has a unique feature name (e.g., "File uploads", "Page history", "External guest limit"). After each scroll cycle, compare newly visible feature names against the already-captured set. Only add rows with new feature names. This prevents double-counting rows that remain visible across overlapping scroll positions.

5. **Detect checkmarks via PlanFeatures_check__ class prefix, not aria-label.** Notion's checkmark indicators use a span with class PlanFeatures_check__ujQKZ wrapping an SVG path element. There is no aria-label on checkmarks. Autopilot should check for the presence of a child element matching class*="PlanFeatures_check__" within each plan value cell. Absence of this element means "not included" (Notion does not use an explicit cross/X icon).

6. **Billing toggle state awareness: check monthly vs annual before extracting prices.** The billing toggle (BillingIntervalToggle_toggle__SmUIW) switches between monthly and annual pricing. The current HTML shows monthly prices ($0/$10/$20). Autopilot should document which billing period is active when extracting plan prices. To get annual prices, click the toggle first, wait for DOM update, then re-read the pricing cards.

7. **Category header discrimination: PricingGrid_sectionLabel__ vs PricingGrid_row__.** Section labels (Content, Sharing & collaboration, Notion AI, etc.) use PricingGrid_sectionLabel__ class. Feature data rows use PricingGrid_row__ class. Autopilot should capture section labels as grouping metadata, not as feature rows. The 9 sections organize the 58 feature rows into logical categories.

8. **Sticky header handling: PricingGrid_stickyHeader__ stays visible during scroll.** The plan column names (Free, Plus, Business, Enterprise) are in a sticky header that remains visible as the user scrolls through feature rows. This means the header row may appear in every read_page result -- autopilot should extract header names once and skip re-reading them on subsequent scroll cycles.

9. **Wait for DOM stable after each scroll, especially for lazy content rendering.** Although Notion server-renders all rows, some SaaS sites use content-visibility:auto or IntersectionObserver for below-fold rows. After each scroll, call wait_for_stable or waitForDOMStable before reading rows. This ensures any lazy-loaded content has finished rendering. Target: 300-500ms stability window.

10. **Completeness check: verify "Support" section (last category) was fully captured.** The 9th and final section is "Support" with 3 rows (Priority support, Premium support, Customer success manager). After the scroll-read-deduplicate loop completes (2 consecutive scrolls with 0 new rows), verify that Support section rows are in the captured set. If the last section is missing, one more scroll cycle is needed.

11. **Fallback site strategy: use Airtable pricing if Notion restructures.** If Notion's pricing page layout changes or becomes inaccessible, fall back to airtable.com/pricing. The generic selectors in the site guide are designed for common SaaS pricing table patterns. Also consider: slack.com/pricing, github.com/pricing, asana.com/pricing as additional fallback targets.

12. **CSS Module class partial matching for React-based SaaS sites.** Notion uses CSS Module hashed classes (PricingGrid_row__bHvcz). The hash suffix changes between deployments. Use class*="PricingGrid_row__" (partial match on prefix) rather than exact class matching. This pattern applies to all React-based SaaS sites that use CSS Modules for component styling.
