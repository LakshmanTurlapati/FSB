1. IDENTIFY TEST TARGETS FOR SHUFFLED CANCEL BUTTONS:

   (a) Search for publicly accessible dark pattern demonstration sites that showcase randomized button patterns. Try: https://www.deceptive.design/ (formerly darkpatterns.org), https://darkpatterns.uxp2.com/, or similar demo sites

   (c) HTTP fetch each candidate to analyze their HTML structure for cancel/unsubscribe flows with multiple buttons
   (d) As fallback, analyze real subscription service cancellation pages via HTTP to document the button layout (even if behind auth): Amazon Prime cancellation, NYTimes subscription, Spotify cancellation, Adobe Creative Cloud cancellation
   (e) Record: URL tested, HTTP status, page size, whether cancellation buttons found, number of buttons, button text content

2. VALIDATE DEMO DARK PATTERN SITE (primary target):
   (a) Fetch the primary demo site candidate page
   (b) Look for pages with two or more buttons where one is "cancel/unsubscribe" and another is "keep/stay"
   (c) Analyze the HTML to determine: are buttons inside a dialog/modal, how many buttons exist, what text each button contains, are there any JavaScript randomization indicators (Math.random, shuffle, randomize in script tags)
   (d) Record: page structure, button count, button text for each, any randomization code found

3. ANALYZE REAL SUBSCRIPTION CANCELLATION PAGES (HTTP validation):
   (a) Fetch https://www.amazon.com/gp/primecentral (Amazon Prime) -- look for cancellation flow indicators, button patterns, retention steps
   (b) Fetch https://help.nytimes.com/hc/en-us/articles/115014893968-Cancel-a-subscription (NYTimes) -- look for documented cancellation flow, button descriptions
   (c) Fetch https://support.spotify.com/us/article/cancel-premium/ (Spotify) -- look for cancellation instructions revealing button layout
   (d) Fetch cancellation documentation pages that describe the flow (since actual cancellation pages require auth)
   (e) Record: HTTP status, page size, cancellation flow documentation found, button descriptions/screenshots referenced

4. ANALYZE BUTTON RANDOMIZATION MECHANISMS:
   (a) For each page with visible button HTML, check for JavaScript that randomizes button order:
       - document.querySelector/querySelectorAll + insertBefore/appendChild (DOM reordering)
       - CSS flexbox order property randomization (order: Math.random())
       - CSS grid column placement randomization
       - Server-side rendering with randomized button order (no client-side JS needed)
       - React/Vue conditional rendering with randomized state
   (b) Look for A/B testing frameworks (Optimizely, VWO, Google Optimize) that may serve different button layouts
   (c) Record: randomization method detected (if any), client-side vs server-side randomization

5. TEXT-BASED IDENTIFICATION VALIDATION:
   (a) For every page with cancel/keep buttons found, extract all button text and classify:
       - List each button with: text content, CSS classes, data attributes, aria-label, href (if link)
       - Classify each as cancel-intent or keep-intent using the keyword lists from shuffled-cancel.js
       - Verify that text-based classification correctly identifies the cancel button regardless of position
   (b) Check for confirmshaming patterns: emotional language, double negatives, loss-framing
   (c) Record: total buttons found, text classification result for each, confirmshaming detected (yes/no), correct cancel button identified (yes/no)

6. MULTI-STEP RETENTION FLOW ANALYSIS:
   (a) For sites with documented cancellation flows, map the retention steps:
       - Step count: how many pages/screens between "I want to cancel" and actual cancellation
       - Retention offers: discounts, plan downgrades, pause options presented
       - Reason collection: mandatory survey/dropdown before proceeding
       - Final confirmation: the shuffled button page
   (b) Record: number of retention steps per site, types of retention tactics, button text at each step

7. DARK PATTERN SEVERITY ASSESSMENT:
   (a) For each tested target, classify the shuffled button dark pattern severity:
       - Level 1 (Mild): Cancel and Keep buttons exist but Keep is more prominent (larger, colored)
       - Level 2 (Moderate): Button positions are randomized between page loads
       - Level 3 (Severe): Randomized positions PLUS confirmshaming language PLUS visual deception (Cancel styled as link, Keep styled as button)
       - Level 4 (Extreme): All of Level 3 PLUS multiple retention steps PLUS the cancel option disappears or changes position mid-interaction
   (b) Assess whether DOM text analysis is sufficient to handle each severity level
   (c) Note the ratio of clicks to cancel vs clicks to keep (the asymmetry metric)

8. OUTCOME CLASSIFICATION:
   - PASS = demo site or real site found with shuffled buttons, text-based identification correctly selected Cancel button, cancellation completed and verified via MCP tools
   - PARTIAL = HTTP validation confirms cancellation page structure with button text analysis possible, text-based classification strategy validated against real button HTML, but live click execution blocked by WebSocket bridge disconnect or auth requirement
   - FAIL = could not find any page with cancel/keep button pairs for analysis, or text-based classification failed to identify the correct button
   - SKIP-AUTH = all cancellation pages require authentication with no public demo alternative available
