<objective>
Create a shuffled-cancel.js site guide with a cancelSubscription workflow and DARK-03 guidance for identifying the correct Cancel button on subscription cancellation pages where Keep/Cancel button positions are randomized on each page load.

Purpose: DARK-03 requires cancelling a subscription on a site that shuffles Keep/Cancel button positions. This is a "Visual Ambiguity & Dark Patterns" edge case -- subscription cancellation flows randomize the position of the "Keep subscription" and "Cancel subscription" buttons so the user cannot rely on muscle memory (e.g., "Cancel is always on the left"). The AI must NOT rely on button position, DOM order, or visual placement. Instead, it must: (a) use get_dom_snapshot to read all button text content in the cancellation confirmation dialog, (b) identify which button contains "Cancel" / "End subscription" / "Unsubscribe" text regardless of its position, (c) click the correct button by text content match, (d) verify cancellation was confirmed. The site guide must document: demo and real subscription cancellation targets, text-based button identification strategy, multi-step cancellation flow navigation, and the full cancelSubscription workflow.

Output: site-guides/utilities/shuffled-cancel.js and background.js updated with importScripts entry
</objective>

4. `guidance`: Multi-line template string with these sections:
   - SHUFFLED CANCEL BUTTON INTELLIGENCE (DARK-03)
   - DARK PATTERN CONTEXT section explaining: Subscription cancellation flows intentionally randomize the position of "Keep subscription" and "Cancel subscription" buttons on each page load. This exploits users' muscle memory -- after training users that the right button or the colored button means "confirm," the site swaps button positions on the cancellation page so the trained behavior leads to clicking "Keep" instead of "Cancel." Some implementations also use confirmshaming language ("Yes, I want to miss out" vs "No, keep my benefits") to further confuse the choice. Additional tricks include: (a) making "Keep" visually prominent (colored, larger) while "Cancel" is plain text or a link, (b) changing button order between page loads via JavaScript Math.random() or server-side randomization, (c) adding extra confirmation steps only for cancellation (not for keeping), (d) using double negatives ("Don't cancel" vs "Don't keep"). The AI must use TEXT CONTENT analysis, never position or visual style, to identify the correct button.
   - TEXT-BASED BUTTON IDENTIFICATION STRATEGY section: The core approach is to read all button/link text in the cancellation dialog and match against known cancel-intent keywords. This section documents:
     Cancel-intent keywords (the button the AI should click): "Cancel", "Cancel subscription", "Cancel anyway", "Cancel my subscription", "End subscription", "Unsubscribe", "Yes, cancel", "Confirm cancellation", "I want to cancel", "Proceed with cancellation", "Cancel my account", "End my trial", "Stop subscription"
     Keep-intent keywords (the button the AI must NOT click): "Keep", "Keep subscription", "Keep my subscription", "Stay subscribed", "Don't cancel", "Go back", "Never mind", "I changed my mind", "Keep my benefits", "No, keep it", "Stay", "Continue subscription", "Keep my plan", "Reconsider"
     Confirmshaming detection: Some sites phrase cancel as a negative to discourage clicking: "Yes, I want to miss out on savings", "No, I don't want exclusive access." The AI must parse the SEMANTIC MEANING, not the emotional framing. A button phrased as "lose benefits" but whose action IS cancellation should be clicked.
   - MULTI-STEP CANCELLATION FLOW section: Many subscription sites have multiple retention steps before the final cancel button:
     Step 1: Navigate to account settings or subscription management page
     Step 2: Click "Cancel subscription" or "Manage subscription" link
     Step 3: Retention offer page -- "We'll give you 50% off, stay?" with "Accept offer" / "No thanks, cancel" buttons. Click the cancel variant.
     Step 4: Reason selection page -- dropdown or radio buttons asking "Why are you leaving?" with a "Continue" or "Next" button. Select any reason and proceed.
     Step 5: Final confirmation -- the shuffled button page. "Are you sure?" with randomized "Keep" / "Cancel" button positions. Identify Cancel by text, click it.
     Step 6: Confirmation page -- "Your subscription has been cancelled" success message.
   - BUTTON POSITION INDEPENDENCE section: Documenting why position-based identification fails:
     The button DOM order (first child vs second child) may be randomized via JavaScript on page load. CSS flexbox order or grid placement can visually reorder buttons without changing DOM order. Some sites use absolute positioning with randomized left/right offsets. The AI must NEVER use: element index (first/second button), CSS position (left/right), visual prominence (color, size), or DOM order. The AI must ALWAYS use: text content (innerText/textContent), aria-label attribute, data-action or data-testid attribute value, href for link-based cancel buttons.
   - VERIFICATION section: After clicking cancel: (a) look for confirmation text: "cancelled", "subscription ended", "successfully cancelled", "cancellation confirmed," (b) check that no "Your subscription is still active" or error message appears, (c) if redirected back to the subscription page, check the subscription status shows "Cancelled" or "Inactive."

   Step 1: NAVIGATE TO CANCELLATION PAGE -- Navigate to the subscription management or cancellation page. This may require navigating through account settings first.
   Step 2: DETECT CANCELLATION DIALOG -- Use get_dom_snapshot to identify the cancellation confirmation dialog or page. Look for containers with cancel/confirmation/modal classes and buttons within them.
   Step 3: READ ALL BUTTON TEXT -- Use get_dom_snapshot or get_text to extract the text content of EVERY button and clickable link in the cancellation area. Record each button's text, its selector, and its position. Do NOT decide based on position alone.
   Step 4: CLASSIFY BUTTONS BY TEXT INTENT -- For each button, classify as cancel-intent or keep-intent by matching text against the keyword lists. If confirmshaming is detected (emotional language), parse the semantic action (what will happen if clicked) rather than the emotional framing.

Use the button's text-based selector (e.g., button containing "Cancel") rather than positional selector (e.g., first button, left button).
