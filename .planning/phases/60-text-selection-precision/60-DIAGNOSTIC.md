## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-60):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55, 56, 57, 58, 59, and now 60. Without it, no tool (DOM or CDP based) can execute. This is the primary blocker for all live MCP testing in this milestone.
- **No "find sentence in paragraph" tool:** The current workflow requires the AI to calculate sentence boundaries manually (find ". [A-Z]" patterns, handle abbreviations, handle citation brackets). A dedicated `find_sentences(selector)` tool that returns an array of `{sentenceIndex, text, startOffset, endOffset}` objects would eliminate manual boundary calculation and make text selection prompts significantly easier for autopilot. This tool would encapsulate the abbreviation list, citation handling, and edge cases in one place.
- **No get_text_offsets tool:** A `get_text_offsets(selector, regex)` tool that returns character positions for regex matches within an element's textContent would help autopilot find specific text positions without manual scanning. For text selection tasks, this would return the exact offsets for any matching substring, accounting for inline elements.
- **thirdParagraph CSS selector inaccuracy:** The site guide's `thirdParagraph` selector (`p:nth-of-type(3)`) is a static CSS selector that does not account for empty `<p>` elements. On the current Albert Einstein article, this selector targets the 2nd non-empty paragraph, not the 3rd. The correct approach requires JavaScript filtering: `querySelectorAll('#mw-content-text .mw-parser-output > p')` then filtering by `textContent.trim().length > 10`. A dedicated tool or site guide enhancement would be needed to handle "Nth non-empty paragraph" semantics.
- **No get_bounding_rect tool (carried from Phase 57):** For the CDP drag fallback approach to text selection, precise character-level coordinates are needed. A `get_bounding_rect(selector)` tool would simplify coordinate calculations. For text selection, this would need to work at the character level (using Range API + getBoundingClientRect for individual characters), which is more granular than element-level bounding rects.
- **TreeWalker inline element validation gap:** The selectTextRange action uses TreeWalker to traverse text nodes, but this has not been validated against Wikipedia's specific inline element structure. Wikipedia paragraphs contain `<a>` (wikilinks), `<b>` (bold), `<i>` (italic/emphasis), `<sup class="reference">` (citation superscripts like [10]), and occasionally `<span>` elements. The TreeWalker SHOW_TEXT filter should handle all of these, but edge cases like nested inline elements (`<sup><a>[10]</a></sup>`) may produce unexpected text node orderings.

## Autopilot Recommendations

1. **Paragraph identification must filter empty paragraphs:** Wikipedia articles contain empty `<p class="mw-empty-elt">` elements before the lead section. Autopilot must NOT use `p:nth-of-type(N)` CSS selectors for "Nth paragraph" requests. Instead: (a) use get_dom_snapshot to find all `<p>` elements in `#mw-content-text .mw-parser-output`, (b) filter out paragraphs with textContent.trim().length < 10, (c) index into the filtered list. For "third paragraph", take index 2 (0-based) of the filtered non-empty list. On the current Albert Einstein article, the 3rd non-empty paragraph is p:nth-of-type(4), not p:nth-of-type(3).

2. **Sentence boundary detection strategy:** Use ". [A-Z]" as the primary sentence boundary pattern, with the following refinements: (a) Skip known abbreviations: "Dr.", "Mr.", "Mrs.", "U.S.", "St.", "Jr.", "Sr.", "Inc.", "Ltd.", "Corp.", "e.g.", "i.e.", "vs.", "al.", "Prof.", "No.", "Vol.", "Fig.", "eq.". (b) Handle citation brackets: text like "papers.[10] In them" has the period at the end of "papers", then "[10]" as citation text, then " In" starting the next sentence. The boundary detection must skip over "[N]" notation when looking for the space + uppercase pattern after a period. (c) Count the boundary occurrences to find the Nth sentence: first boundary = end of sentence 1, second boundary = end of sentence 2, etc.

3. **Citation text impacts character offsets:** Wikipedia textContent includes citation superscript text like "[10]", "[11]", "[12]". These add 4-6 characters per citation to the total text length. Example: "...papers.[10] In them..." has "[10]" occupying positions 108-111, pushing "In them" to position 113 instead of 109. Autopilot must: (a) Read the full textContent (which includes citation text) for offset calculation, (b) NOT strip citations from the text before calculating offsets, because select_text_range operates on the raw textContent including citation text nodes, (c) Be aware that the "visible" sentence text (what a human reads) differs from the textContent (what the API operates on) by the citation bracket characters.

4. **select_text_range vs CDP drag for text selection:** Use select_text_range as the primary method for text selection tasks. Advantages: (a) deterministic -- works regardless of font size, zoom level, or viewport width, (b) handles inline elements via TreeWalker text node traversal, (c) does not require coordinate calculations, (d) returns selectedText for immediate verification. Use CDP drag as fallback only if select_text_range fails (element not found, Range API error, or incorrect text selected). CDP drag requires: calculate getBoundingClientRect for start and end characters using Range API, then drag(startX, startY, endX, endY, steps=30, stepDelayMs=15). CDP drag is less reliable because character coordinates depend on font rendering, line wrapping, and zoom level.

5. **Verification strategy for text selection:** After select_text_range returns, verify the result: (a) Check selectedText starts with an uppercase letter and ends with a period (for sentence selection), (b) Compare selectedText length against expected sentence length (within 5% tolerance for minor textContent differences), (c) Check selectedText contains key expected words (for the Albert Einstein article, the second sentence should contain "photoelectric effect" or "Brownian motion"), (d) If verification fails, recalculate offsets by re-reading the paragraph textContent and re-scanning for sentence boundaries. Do NOT retry with the same offsets -- if offsets were wrong, recalculation is needed.

6. **Cross-language Wikipedia support considerations:** Sentence boundary patterns differ across languages: (a) English/French/Spanish: ". [A-Z]" works for most cases, (b) German: nouns are capitalized mid-sentence, so ". " followed by a common noun looks like a sentence boundary but is not. Additional heuristic: check if the word after ". " is a known German noun determiner ("Der", "Die", "Das", "Ein", "Eine") vs a mid-sentence capitalized noun, (c) Chinese/Japanese: sentence boundary is a full-width period (U+3002) with no space after it, (d) Arabic/Hebrew: right-to-left text direction affects offset calculation. For non-English Wikipedia, autopilot should detect the language from the URL (e.g., de.wikipedia.org for German) and apply language-specific sentence boundary rules.

7. **Fallback if select_text_range fails:** If select_text_range returns an error (element not found, TreeWalker failure, Range API exception): (a) First retry with a different selector for the paragraph (try element reference from get_dom_snapshot instead of CSS selector), (b) If still failing, try selectText (existing tool) to select the entire paragraph text, then narrow manually, (c) If element-based selection fails entirely, fall back to CDP drag: use Range API to get start and end character bounding rects, calculate viewport coordinates, then drag(startX, startY, endX, endY, steps=30, stepDelayMs=15). CDP drag will create a browser-native text selection, (d) If CDP drag also fails (coordinates off-screen, element obscured), scroll the paragraph into viewport center first using scroll tool, then retry CDP drag.

8. **Element reference resolution for paragraphs with inline markup:** Wikipedia paragraphs contain extensive inline markup: `<a>` (wikilinks, up to 20+ per paragraph), `<b>` (bold terms), `<i>` (Latin phrases like "annus mirabilis"), `<sup class="reference">` (citation superscripts). The select_text_range tool uses TreeWalker with NodeFilter.SHOW_TEXT which visits all text nodes regardless of their parent inline element. This means: (a) A single paragraph's text may be split across 30+ text nodes, (b) Character offsets in textContent map correctly because TreeWalker visits text nodes in document order, (c) The Range API correctly handles selections that span multiple inline elements (e.g., selecting "...the [photoelectric effect](link), explained..." crosses an `<a>` element boundary). Autopilot should not attempt to "flatten" or pre-process the HTML -- just use textContent for offsets and let TreeWalker handle the node traversal.

9. **Handling dynamic content and article structure changes:** Wikipedia articles are edited frequently. The paragraph structure, sentence content, and citation numbering may change between test runs. Autopilot should: (a) Always re-read the paragraph text before calculating offsets (never cache paragraph content across sessions), (b) Use relative references ("third non-empty paragraph") rather than absolute CSS selectors (p:nth-of-type(4)) which break when paragraphs are added or removed, (c) If the target paragraph has fewer sentences than expected (e.g., "second sentence" but paragraph has only one sentence), report this as a content mismatch rather than a tool failure.

10. **Multi-sentence selection and range extension:** For tasks that require selecting multiple sentences or extending a selection, autopilot should: (a) Calculate the combined startOffset (start of first target sentence) and endOffset (end of last target sentence), (b) Pass the combined range to a single select_text_range call rather than making multiple calls (multiple calls would each clear the previous selection), (c) For "select from sentence 2 to sentence 4", find boundary[1].next_sentence_start as startOffset and boundary[3].period_pos + 1 as endOffset. The select_text_range tool handles any arbitrary character range within the element.
