## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-65):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55, 56, 57, 58, 59, 60, 61, 62, 63, 64, and now 65. Without it, no tool (DOM or CDP based) can execute. This is the primary blocker for all live MCP testing in this milestone.
- **GEETEST widget is fully JS-rendered (no server-side DOM):** The GEETEST slider CAPTCHA widget loads entirely via JavaScript. The `initGeetest` function downloads the widget SDK from api.geetest.com and renders the slider track, thumb, puzzle piece, and background image inside the container div. This means: (a) get_dom_snapshot after page load may not see the widget if JS hasn't finished loading, (b) wait_for_element or wait_for_stable is critical before attempting to find slider elements, (c) the widget may be inside an iframe created by the GEETEST SDK.
- **GEETEST canvas rendering blocks DOM-based gap detection:** GEETEST renders the puzzle background and piece on HTML5 canvas elements (`.geetest_canvas_bg`, `.geetest_canvas_slice`). The gap position is drawn on the canvas, not represented as a DOM element. This means: (a) the gap X position cannot be read from DOM attributes or CSS, (b) vision/image analysis of the canvas would be needed for precise gap detection, (c) estimation heuristics (50-60% of track width) are the only DOM-based approach. The `geetest_canvas_fullbg` element may encode gap position via CSS transform or clip-path, but this is implementation-specific.
- **Ease-in-out timing curve precision at low delays:** The drag_variable_speed handler uses `setTimeout(r, delay)` for inter-step delays. JavaScript setTimeout has a minimum resolution of ~4ms in modern browsers (due to timer throttling). With minDelayMs=5, the actual delay at peak speed may be 5-10ms (clamped by browser timer resolution). This may or may not affect the speed curve shape visible to GEETEST's anti-bot analysis. Higher-precision timing would require Web Workers or requestAnimationFrame.
- **CDP Input.dispatchMouseEvent through iframes:** If the GEETEST widget renders inside an iframe (common for third-party CAPTCHA providers), the CDP commands may need to target the iframe's session/frame, not the main page. The current handleCDPMouseDragVariableSpeed uses the tab's main frame for CDP commands. If the CAPTCHA is in an iframe, mouse coordinates would need to be iframe-relative and the CDP target may need to be the iframe.
- **No get_bounding_rect tool (carried from Phases 57-64):** To determine the slider thumb's precise viewport coordinates for drag_variable_speed, a dedicated `get_bounding_rect(selector)` tool would return exact {x, y, width, height} without parsing DOM snapshot text. Currently, coordinates must be estimated from DOM snapshot position data which may be imprecise.

## Autopilot Recommendations

1. **Detect slider CAPTCHA type from DOM before choosing interaction strategy:** Before attempting any drag, the AI should identify the CAPTCHA provider by scanning for provider-specific class patterns. GEETEST: look for `.geetest_panel`, `.geetest_slider`, `[class*="geetest"]`. Tencent: look for `.tc-captcha-popup`, `.tc-slider`, `#tc-captcha`. Generic: look for elements with both "slider" and "captcha" in class/id, or elements with "slide" and "verify". The provider determines which selectors to use for the thumb element and which success/failure indicators to check.

2. **Always use drag_variable_speed for slider CAPTCHAs, not regular drag:** Anti-bot CAPTCHA providers (GEETEST, Tencent, hCaptcha) analyze the drag speed profile as part of their bot detection. Constant-speed drag (uniform stepDelay) is a strong bot signal. The drag_variable_speed tool's ease-in-out timing curve (slow start at 40-45ms, fast center at 5ms, slow end at 40-45ms) mimics human acceleration/deceleration. Fall back to regular drag only for demo/test CAPTCHAs that check position only, not speed profile.

3. **Wait for GEETEST widget to fully load before scanning for slider elements:** GEETEST widgets render entirely via JavaScript after `initGeetest` API call. The widget SDK downloads from api.geetest.com and renders asynchronously. Always use `wait_for_element` targeting `.geetest_slider_button` or `.gt_slider_knob` (or `wait_for_stable` with 1000ms+ stability threshold) before attempting get_dom_snapshot. Without waiting, the DOM snapshot will contain only the empty container div (`#embed-captcha`) and miss all slider elements.

4. **Estimate target position at 50-60% of track width as initial guess, then adjust:** Without vision/image analysis, the puzzle gap position in GEETEST CAPTCHAs cannot be determined from DOM. Start with targetX = trackLeft + trackWidth * 0.55 (center estimate). If the CAPTCHA fails (slider resets), retry with +20px, then -20px, then +40px. GEETEST has a position tolerance of approximately 5-10px for production CAPTCHAs and 15-20px for demo CAPTCHAs. Maximum 3-4 attempts before reporting failure.

5. **Use recommended drag parameters: steps=25, minDelayMs=5, maxDelayMs=45 for GEETEST:** These parameters produce a smooth 25-point ease-in-out curve over approximately 625ms total drag duration. The quadratic speed curve `1 - 4*(t-0.5)^2` creates: (a) slow start at ~45ms per step (first 5 steps), (b) fast center at ~5ms per step (middle 15 steps), (c) slow end at ~45ms per step (last 5 steps). For basic/demo CAPTCHAs, fewer steps (15-20) with wider delay range (minDelayMs=3, maxDelayMs=60) may work. For production CAPTCHAs, more steps (30-40) with tighter delay range (minDelayMs=8, maxDelayMs=35) may appear more human-like.

6. **Check for CAPTCHA iframe before targeting slider elements:** Third-party CAPTCHA providers often render their widgets inside iframes. If get_dom_snapshot finds no `.geetest_slider_button` on the main page but finds an iframe with a geetest.com or captcha-related src, the slider elements are inside that iframe. CDP coordinates must be iframe-relative. Check for: `iframe[src*="geetest"]`, `iframe[src*="captcha"]`, `iframe[class*="geetest"]`. If found, note that CDP Input.dispatchMouseEvent may need frame-level targeting.

7. **Differentiate GEETEST v3 (slider) from GEETEST v4 (click/icon select):** GEETEST offers multiple CAPTCHA types. V3 uses the classic slider puzzle where a puzzle piece is dragged to fit a gap. V4 uses click-based challenges (click icons in order, click matching items). The slider-focused approach only applies to GEETEST v3. Detect type by checking: v3 has `.geetest_slider` track element, v4 has `.geetest_item_wrap` click targets. If v4 is encountered, the drag approach does not apply.

8. **Detect success/failure via multiple signals, not just one:** After drag_variable_speed completes, check for success with multiple signals: (a) `.geetest_panel_success` class on panel, (b) text content containing "success", "verified", "pass" (case-insensitive), (c) slider track background changing to green (#52ccba or similar), (d) form submit button becoming enabled (was disabled with CAPTCHA pending), (e) hidden form fields `geetest_challenge`, `geetest_validate`, `geetest_seccode` being populated. For failure: (a) slider thumb position resetting to trackLeft, (b) `.geetest_panel_error` class, (c) "try again" text, (d) puzzle image refreshing (new puzzle loaded). Multiple signals reduce false positive/negative detection.

9. **Handle rate limiting and anti-automation detection gracefully:** GEETEST and other CAPTCHA providers implement rate limiting. If multiple failed attempts occur rapidly, the provider may: (a) show "human verification required" message, (b) switch to a harder CAPTCHA type (click instead of slider), (c) block the IP temporarily, (d) require a page refresh. After each failed attempt, wait 2-3 seconds before retrying. After 3 consecutive failures, refresh the page to get a new CAPTCHA challenge before trying again. Do not attempt more than 5-6 drags per page load.

10. **Fall back to regular drag for basic/demo slider CAPTCHAs that lack anti-bot detection:** Some slider CAPTCHAs on demo pages, test sites, or low-security implementations check only the final position, not the drag speed profile. For these, regular drag with uniform speed works fine: `drag(startX, startY, endX, endY, steps=30, stepDelayMs=15)`. Signs that a CAPTCHA is basic/demo: (a) no geetest/tencent/hcaptcha branding, (b) simple HTML/CSS slider without canvas rendering, (c) no external API calls (no network requests to captcha provider), (d) slider track is a plain `<div>` with `<input type="range">` or custom CSS. Using drag_variable_speed on these still works but is unnecessarily sophisticated.
