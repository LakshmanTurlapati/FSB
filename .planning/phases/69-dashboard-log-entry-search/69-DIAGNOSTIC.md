## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-69):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected in Phases 55-68 and now Phase 69. Without it, no tool (DOM or CDP based) can execute against the live browser. This is the primary blocker for all live MCP testing in this milestone.
- **relative-time element extraction gap:** The site guide assumes `relative-time` and `time[datetime]` elements will be in the DOM with datetime attributes for date comparison. However, these elements are only rendered client-side by JavaScript. In a live browser, `read_page` should capture them after JavaScript execution, but this cannot be validated without the WebSocket bridge. If `read_page` captures the DOM after JS execution, this gap is resolved. If it captures server-rendered HTML, datetime-based comparison will fail.
- **get_element_attribute tool (proposed):** Currently no MCP tool directly extracts a specific attribute value (e.g., `datetime` from `relative-time`). `read_page` returns text content, and `get_dom_snapshot` returns structured DOM elements. A `get_element_attribute(selector, attribute)` tool would enable direct datetime extraction from `relative-time` elements without parsing full page text. Example: `get_element_attribute('relative-time', 'datetime')` returns `["2026-03-18T22:50:29Z", "2026-03-18T21:27:11Z", ...]`.
- **find_element_by_date tool (proposed):** A tool that scrolls until an element matching a date condition is found would encapsulate the scroll-compare-extract loop. Parameters: `find_element_by_date(containerSelector, dateElementSelector, dateAttribute, targetDate, scrollDirection, maxScrolls, waitMs)`. This would handle the entire "scroll until target date found" pattern for any infinite-scroll timeline.
- **scroll_until tool (proposed, variant of find_element_by_date):** A more generic tool that scrolls until a condition is met (element visible, text found, attribute matches). This would generalize the scroll-search pattern beyond date-based searches.
- **GitHub contributions fragment loading:** The profile page loads the contributions section as an `include-fragment` via a separate HTTP request. In a live browser, this fragment loads automatically. MCP tools need to wait for this fragment to load before attempting to read activity entries. `wait_for_stable` should handle this, but worth noting as a timing consideration.

## Autopilot Recommendations

1. **Use the contributions tab, not a nonexistent "activity" tab:** GitHub profiles do not have an `?tab=activity` URL parameter. The activity timeline is in the default "contributions" view at `github.com/{username}?tab=contributions` or simply on the default profile page below the contribution graph. Autopilot should look for `contribution-activity-listing` container rather than clicking an "Activity" tab. Update the site guide `activityTab` selector to `a[href$="?tab=contributions"]` or remove it.

2. **Wait for include-fragment to load before reading activity entries:** The profile page loads the contributions section as an `include-fragment` via a separate async request. After navigating to a profile, use `wait_for_stable` to ensure the contributions fragment has loaded before attempting `read_page`. Look for `.contribution-activity-listing` as the signal that the fragment loaded. Without this wait, `read_page` may capture only the shell page without activity data.

3. **Expect monthly summaries, not individual dated events, in the initial activity view:** GitHub's activity timeline groups events by month ("Created 134 commits in 1 repository" for March 2026). Individual dated events with `relative-time` timestamps may only appear after expanding the monthly summary or scrolling within the expanded details. Autopilot should first expand (`<details>` toggle) the month containing the target date, then look for individual events within.

4. **Select target users with consistent daily activity for reliable 3-day-old entries:** torvalds (Linus Torvalds) has 3,072 contributions in the past year with activity on the target date March 18 (5 contributions confirmed). Alternative targets: gaearon (React contributor), sindresorhus (prolific open source). Avoid inactive users who may not have entries from exactly 3 days ago. Verify the contribution calendar shows a colored square for the target date before attempting to scroll to that date's entries.

5. **Parse datetime from relative-time element attributes, not displayed text:** In the live browser DOM, activity entries contain `<relative-time datetime="2026-03-18T22:50:29Z">3 days ago</relative-time>` elements. Use the `datetime` attribute value for precise date comparison rather than parsing the displayed relative text ("3 days ago" is ambiguous -- does "3 days ago" mean exactly 72 hours or any time on March 18?). Compare the attribute date (YYYY-MM-DD from the ISO string) to the target date.

6. **Allow 1500-2500ms between scroll iterations for GitHub activity feed loading:** GitHub's activity feed loads additional entries when scrolling near the bottom of the contribution activity section. Use `wait_for_stable` after each scroll to confirm new content appeared. If no new `TimelineItem` elements appear after scroll + 3 second wait, the end of available activity has been reached. Expected rate: GitHub typically shows 20-50 events per scroll load, so reaching 3 days ago may require 2-10 scrolls depending on user activity level.

7. **Use date comparison with a 24-hour window for "3 days ago" matching:** When comparing entry datetimes to the target date, check if the date portion (YYYY-MM-DD) matches the target, OR if the datetime is within +/- 24 hours of midnight on the target date. This handles timezone differences and edge cases where "3 days ago" from the user's perspective spans midnight UTC. For torvalds on March 18, commits span 15:06:30Z to 22:50:29Z -- all within the same calendar day.

8. **Deduplicate activity entries by commit SHA or event link href across scroll snapshots:** GitHub activity feeds persist DOM elements as you scroll (not virtualized like Twitter). However, `read_page` may capture overlapping entries between scroll iterations. Track a Set of seen commit SHAs or event link hrefs to avoid processing the same entry twice. For the torvalds profile, commit links follow the pattern `/torvalds/linux/commit/{sha}` -- use the full href as the deduplication key.

9. **Handle the "overshoot" case when entries jump past the target date:** If scrolling reveals entries older than the target date without finding an exact match on the target date, report the closest entries before and after. For example, if the newest visible entry is from March 19 and the oldest is from March 17, report both and note that March 18 entries may have been grouped differently or may be in a collapsed section.

10. **Fall back to GitHub API if DOM scroll fails:** If the activity feed cannot be scrolled to the target date (DOM not loading, JavaScript rendering issues, or WebSocket bridge disconnect), use the GitHub REST API as a fallback: `GET /repos/{owner}/{repo}/commits?author={user}&since={targetDate}T00:00:00Z&until={nextDay}T00:00:00Z`. This returns commit data with full messages and timestamps. The API does not require authentication for public repos (rate limited to 60 requests/hour for unauthenticated access). This was successfully used in this diagnostic to confirm 5 commits on March 18.
