Execute the SCROLL-03 edge case test using MCP manual tools following the findLogEntryByDate workflow from the github.js site guide. Test steps:

1. Use MCP navigate tool to go to github.com/torvalds (Linus Torvalds -- active daily with Linux kernel commits). If torvalds profile is insufficient, try github.com/gaearon or github.com/sindresorhus as fallback targets.

2. Use read_page to verify the profile page loaded. Check for:
   - Cookie/consent banners: dismiss via click if present
   - Login walls: if GitHub requires auth to view activity, document as skip-auth and try alternate public pages
   - Profile presence: user avatar, username, contribution graph visible
   Record the page title, URL, and whether the profile loaded successfully.

3. Scroll down to the contribution activity section of the profile page. The activity timeline is typically below the contribution graph (green squares). Use read_page to identify the activity section location. If an "Activity" tab exists (a[href$="?tab=activity"]), click it.

4. Use read_page to capture the first batch of visible activity entries. For each entry, identify:
   - The event text (what action was performed: committed, opened issue, created PR, starred repo)
   - The repository name associated with the event
   - The timestamp: look for <relative-time> or <time> elements with datetime attributes
   - The event link (href to specific commit/issue/PR) for deduplication
   Record the number of visible entries and the date range they cover.

5. Calculate the target date: current date minus 3 days. For example, if today is 2026-03-21, the target date is 2026-03-18.

6. For each visible entry, compare its datetime attribute to the target date:
   - If entry date matches target date (within 24 hours): extract full entry text as the found log entry
   - If all visible entries are newer than target date: need to scroll down for older entries
   - If visible entries are older than target date: overshot, report closest entries

7. SCROLL LOOP (if target date not yet visible, repeat until found or max 15 scroll iterations):
   a. Use scroll tool to scroll down (scroll direction: "down", amount: "page" or pixels)
   b. Wait 1500-2500ms for new entries to load (use waitForDOMStable or wait_for_stable)
   c. Use read_page to capture newly visible entries
   d. For each new entry: extract datetime, compare to target date
   e. If target date entry found: break loop, extract entry
   f. If entries are now older than target date: overshot, report closest entries
   g. If no new entries loaded after scroll + 3 second wait: end of available activity, break loop
   h. Log: "Scroll iteration {N}: {newest_visible_date} to {oldest_visible_date}, target: {target_date}"

8. AFTER SCROLL LOOP:
   - If entry from target date found: PASS -- report entry text, repo name, exact timestamp
   - If scrolled to entries older than target but none on exact target date: PARTIAL -- report closest entries before and after target date
   - If could not scroll to target date entries: PARTIAL -- report how far scrolling reached
   - If auth required or profile inaccessible: document as skip-auth
   - If no activity feed visible: FAIL -- document what was visible instead

9. REPORT FOUND ENTRY (if applicable):
   - Full event text (e.g., "torvalds pushed 3 commits to torvalds/linux")
   - Repository name
   - Exact datetime from attribute
   - Relative timestamp as displayed (e.g., "3 days ago")
   - Event link (href to commit/issue/PR)

10. VERIFY: Confirm extraction quality:
    - Extracted datetime is within +/- 24 hours of target date
    - Entry text is a coherent event description (not HTML fragments)
    - Repository name is a valid GitHub repo path
    - Timestamp makes sense relative to today's date

Record every step outcome (tool used, target, result, notes) for the diagnostic report.
