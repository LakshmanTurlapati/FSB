## Result Summary
Live MCP test was attempted but could not execute through the full MCP tool chain due to the persistent WebSocket bridge disconnect (ports 3711/3712 not listening, same blocker as Phases 55-70). HTTP-based validation was performed against the official pdf.js demo viewer at https://mozilla.github.io/pdf.js/web/viewer.html (66KB server HTML). The server HTML contains all toolbar elements (pageNumber input, previous/next buttons, zoom controls, scale select, loading bar) but zero page/textLayer/canvas elements -- these are dynamically created by JavaScript when pdf.js renders the PDF document. ## Tool Gaps Identified
- **WebSocket bridge availability (persistent gap, Phases 55-71):** The MCP server runs in stdio mode and requires the WebSocket bridge process on ports 3711/3712 to reach Chrome. This bridge has been disconnected since Phase 55. Without it, no tool (DOM or CDP based) can execute against the live browser. This is the primary blocker for all live MCP testing in this milestone.
- **textLayer text extraction via get_dom_snapshot:** In a live browser, `get_dom_snapshot` should be able to access `.page[data-page-number="N"] .textLayer span` elements and read their textContent. However, this has not been verified against pdf.js specifically. If get_dom_snapshot truncates or omits deeply nested span content, text extraction would fail. A dedicated `read_pdf_page(pageNumber)` tool that specifically targets textLayer spans and concatenates their text would be more reliable.
- **Virtualization state detection:** Checking whether a page is virtualized requires inspecting the `data-loaded` attribute on the `.page` div AND checking whether textLayer spans have content. Current tools (read_page, get_dom_snapshot) should expose these DOM attributes, but the depth of inspection needed (attribute presence + child element content) may require multiple tool calls. A `check_page_loaded(pageNumber)` helper that returns loaded/unloaded status would streamline this.
- **Page input interaction sequence:** Navigating via page number input requires a 4-step sequence: click input, select all text, type number, press Enter. This is 4 separate MCP tool calls with timing between each. A `navigate_to_pdf_page(pageNumber)` composite tool would reduce this to a single call.
- **Wait-after-navigation timing:** After jumping to a page via input, the textLayer needs 200-500ms to populate. Current approach requires manual wait_for_stable calls or explicit delays. pdf.js fires a "textlayerrendered" event when textLayer is ready -- a tool that waits for this specific event would be more reliable than generic DOM stability detection.
- **read_page scope limitation:** If read_page returns the full page HTML, the output for a 14-page PDF viewer could be very large (each page's textLayer + canvas + annotations). A scoped read that targets only the visible/active page's textLayer would be more practical for text extraction.

## Autopilot Recommendations

1. **Use https://mozilla.github.io/pdf.js/web/viewer.html as the primary test target for virtualized PDF reading.** This is the official pdf.js demo viewer serving a 14-page sample document with no authentication required. The page loads at 66KB for HTML plus ~643KB for viewer.mjs. Fallback options: any URL matching `/pdfjs/web/viewer.html` or `viewer.html?file=` patterns. Avoid Google Drive PDF viewer (uses img elements instead of canvas+textLayer, different extraction approach needed).

2. **Navigate between pages using the page number input (input#pageNumber) rather than scroll-distance estimation.** The page input accepts any number from 1 to the total page count. The interaction sequence is: (a) click input#pageNumber to focus, (b) select all existing text (Ctrl+A or triple-click), (c) type the target page number, (d) press Enter. This triggers an internal scrollIntoView to the target page with guaranteed accuracy. Scroll-distance estimation is unreliable because page height varies with zoom level and viewport size.

3. **Extract text from textLayer spans using the selector pattern `.page[data-page-number="N"] .textLayer span`.** Each rendered page has a `div.textLayer` child containing span elements for each text run. Concatenate all span.textContent values with spaces to get the page text. The text follows DOM order (top-to-bottom, left-to-right for LTR documents). Always verify the extracted text is non-empty and contains recognizable words -- empty text means the page has not finished rendering or has been virtualized.

4. **Wait 500-1000ms after page input navigation for textLayer population.** After typing a page number and pressing Enter, the viewer scrolls to the target page and triggers canvas rendering + textLayer population. Canvas render takes 200-300ms, textLayer population takes an additional 100-200ms. Use wait_for_stable or check for span children in the textLayer to confirm render completion. If textLayer is still empty after 1000ms, retry the check once more after 500ms. Maximum wait: 2000ms before classifying as render failure.

5. **Detect page virtualization by checking the `data-loaded` attribute, not just textLayer content.** When pdf.js virtualizes a page, it removes the `data-loaded` attribute from the `.page` div (source: viewer.mjs line 13640). To check if page N is still rendered: `document.querySelector('.page[data-page-number="N"]').hasAttribute('data-loaded')`. If `data-loaded` is absent, the page is virtualized. Additionally check textLayer span content -- even if the textLayer div exists, its span children may have empty textContent when the page is unloaded.

6. **Use a sequential read-and-store pattern: extract text as you visit each page, do not rely on reading all pages at the end.** When reading pages 1 through N, earlier pages will be virtualized by the time you reach page N. Extract and store each page's text immediately after navigating to it. Store at minimum the first 200 characters as a verification sample. This prevents data loss from virtualization. For a 14-page document, aim to read and store text from pages 1, 3, 5, 7 (or similar sampling) rather than all 14 -- limits context size while covering the document.

7. **Verify scroll-back re-read consistency by returning to page 1 after visiting page 5+.** After reading forward through several pages, navigate back to page 1 using the page input. Wait 500-1000ms for re-render. Extract textLayer text again and compare with the originally stored page 1 text. They should match exactly because pdf.js re-renders from the same PDF text stream. Any discrepancy indicates a rendering issue or text extraction inconsistency. This round-trip test is the core validation for SCROLL-05.

8. **Limit reading sessions to 10-15 pages per automation session to avoid context bloat.** Each page's extracted text can be 500-2000 characters depending on content density. Reading all 14 pages of the sample document would produce 7-28KB of text data in the automation context. For documents with 50+ pages, sample every 3-5 pages rather than reading sequentially. Always prioritize the first page, a middle page, and the last page for coverage.

9. **Use default zoom level or "page-fit" for consistent text extraction.** The zoom level affects page dimensions and may affect textLayer positioning. The default zoom ("auto") or "page-fit" option provides consistent sizing. The zoom select element (`select#scaleSelect`) supports values: auto, page-actual, page-width, page-fit, or percentage. Avoid high zoom levels (200%+) which may cause more aggressive page virtualization due to larger viewport requirements.

10. **Error recovery: if textLayer is empty after wait, retry once then classify as unloaded.** The retry sequence: (a) navigate to target page, (b) wait 500ms, (c) check textLayer spans, (d) if empty, wait 500ms more, (e) check again, (f) if still empty, classify page as "unloaded/render-failed". Do not retry more than once -- repeated failures indicate the page is genuinely virtualized or the PDF has rendering issues. Move on to the next page and document the failure. For the scroll-back re-read test, allow up to 1000ms wait since re-rendering a previously virtualized page may take longer than initial render.
