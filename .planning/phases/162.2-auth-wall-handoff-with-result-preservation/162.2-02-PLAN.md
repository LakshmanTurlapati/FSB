Implement D-01, D-02, D-04, D-05, and D-06 in the runtime/auth-signal layer.

In `background.js`, create a narrow auth-wall resolver for login-gated final steps and detected login forms that already sit on top of useful completed work. Reuse the primitives that already exist instead of inventing a new flow: `extractLoginFields`, the existing `getDOM` message path, saved credential lookup via `secureConfig.getCredential`, `fillCredentialsOnPage`, `fillCredentialsOnPageDirect`, and `waitForLoginResponse(sessionId)`. Add an explicit auth trigger point in the active run so the runtime can acquire current DOM/login-field context and start one auth-resolution cycle instead of assuming dormant helpers are already wired. The intended order is:

1. Acquire DOM/login-field context for the active page via the existing `getDOM` path and extract a minimal auth target.
2. If saved credentials exist for the current domain, try one direct fill/sign-in attempt.
3. If there are no saved credentials, the fill fails, or the page still requires auth, emit the existing `loginDetected` payload only when a matching sidepanel is active and able to acknowledge the prompt, then wait for `loginFormSubmitted` or `loginSkipped`.
4. If the user submits credentials and the fill succeeds, resume the same session through an explicit pause/resume contract in `ai/agent-loop.js` so the authenticated page is re-checked without creating a new session or terminal channel.
5. If there is no matching sidepanel, the user skips, the prompt times out, credentials are missing/failed, or the page still requires manual approval / MFA / external verification, stop retrying and convert the run into the preserved partial/manual-handoff path using the blocker taxonomy from Plan 01.

In `ai/agent-loop.js`, add the explicit pause/resume lifecycle needed for that auth-resolution cycle so unresolved auth does not immediately finalize as a terminal partial while runtime resolution is still in flight. Do not assume the current idle-session `followUpContext` path is sufficient for this use case; if you reuse it, first harden it into a real active-run continuation contract. Enforce the sequencing rule from Plan 01: `partial_task` is the terminal outcome for unresolved auth only after one saved-credential attempt and one eligible operator-prompt cycle are exhausted, skipped, unavailable, or fail.

Keep this intentionally thin per D-06: one saved-credential attempt, one prompt cycle, one resumed continuation path. Do not build a generalized autonomous login system, do not automate 2FA, and do not add a second completion event. Also do not modify `content/lifecycle.js` or `content/messaging.js`; those existing content helpers are reference context only unless a real bug proves the background direct-fill path insufficient.
