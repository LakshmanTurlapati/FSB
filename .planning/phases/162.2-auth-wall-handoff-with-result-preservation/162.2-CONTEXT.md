# Phase 162.2: Auth Wall Handoff with Result Preservation - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning
**Source:** Verification gap evidence + adjacent auth/login code inspection

<domain>
## Phase Boundary

Teach the agent and runtime to turn auth-gated final steps into explicit manual handoffs on top of the partial-outcome lifecycle built in Phase 162.1. This phase covers login-required, credential-missing, user-skipped-login, and manual-approval/MFA blockers when useful work is already complete. It does not attempt a broad credential-system redesign, full autonomous sign-in orchestration, or generic 2FA automation.

</domain>

<decisions>
## Implementation Decisions

### Reuse the 162.1 lifecycle
- **D-01:** Phase 162.2 must build on `partial_task` and the `outcome/outcomeDetails` transport from Phase 162.1. Do not create a second auth-specific terminal channel.

### Auth blockers that should hand off
- **D-02:** Treat these as auth-wall/manual-handoff scenarios when useful work is already complete: login required, no saved credentials, user skipped credential entry, credentials failed, and manual approval / MFA / external verification gates.

### Agent-facing auth guidance
- **D-03:** The agent loop prompt and tool guidance must explicitly tell the model to use `partial_task` for auth walls after useful work is done, with `reason` values such as `auth_required`, `credentials_missing`, or `manual_approval`.

### Preserved handoff content
- **D-04:** The final partial result must preserve three things clearly: what was already completed, what exact auth blocker stopped the last step, and what manual next step the user can take to finish or resume.

### Existing login primitives
- **D-05:** The repo already contains saved-credential storage, direct login helpers, and a sidepanel login prompt. This phase may reuse or lightly wire those primitives, but unresolved login must still end as a partial/manual-handoff outcome instead of an error.

### Do not overreach into full auth automation
- **D-06:** Do not turn this phase into a general "autonomous login" project. Credential management UX, broad resume orchestration, and full 2FA flows stay out of scope unless a small amount of wiring is strictly necessary for correct handoff behavior.

### Prompt and site-guide alignment
- **D-07:** Existing site-guide language like `SKIP-AUTH` or `AUTH REQUIRED` should align with the new partial/manual-handoff semantics where it affects the agent's final response. The outcome should be useful and resumable, not just classified.

### Scope guardrails
- **D-08:** No redesign of the sidepanel login prompt, no new credential vault model, and no large UI surface additions. This phase is about auth-blocked terminal behavior and preserved handoff clarity.

### Claude's Discretion
- Exact reason taxonomy for auth blockers (`auth_required`, `credentials_missing`, `manual_approval`, etc.)
- Whether the main guidance lives only in `ai/agent-loop.js` / `ai/tool-definitions.js` or also in shared site-guide guidance
- Whether dormant login-prompt helpers should be lightly wired now or left documented if they are not needed for AUTH-01/AUTH-02

</decisions>

<code_context>
## Existing Code Insights

### Confirmed behavior
- Phase 162.1 already introduced `partial_task`, explicit `outcome`, and persisted `outcomeDetails`; popup/sidepanel/options/MCP history now preserve partial results as non-crash outcomes.
- `ai/tool-definitions.js` already hints that `partial_task.reason` may be `auth_required`, but `ai/agent-loop.js` does not yet explicitly teach auth-wall/manual-handoff usage in the system prompt.
- The verification gap was not a runtime crash. The LinkedIn run finished useful research and then cleanly terminated because the model chose the wrong terminal path for the auth-blocked final step.
- `background.js` has existing login/credential primitives: `waitForLoginResponse`, saved credential lookup via `secureConfig.getCredential`, and direct credential fill helpers.
- `ui/sidepanel.js` has a working login prompt UX that sends `loginFormSubmitted` and `loginSkipped` messages.
- `content/lifecycle.js` and `content/messaging.js` expose a direct-login execution path.

### Important limitation
- Current code inspection found no active `loginDetected` producer in the main automation path, so the sidepanel login prompt infrastructure appears partially built but not canonically wired into the generic agent lifecycle today.

### Architectural implication
- AUTH-02 likely needs prompt/tool/site-guide guidance so the model chooses `partial_task` with auth-specific blocker details.
- AUTH-01 may require a thin runtime/auth-signal layer so unresolved login or missing credentials resolve into a preserved manual handoff rather than a silent skip or generic failure.
- The existing credential/login primitives should be treated as optional helpers, not proof that full auth automation already works.

### Likely integration points
- Strengthen auth-wall instructions in `ai/agent-loop.js` and `ai/tool-definitions.js`
- Standardize auth-blocked handoff wording and reason taxonomy in the partial outcome payload
- Reuse existing login/credential primitives in `background.js`, `ui/sidepanel.js`, and content scripts only where they improve correct handoff behavior
- Align site-guide / shared guidance that currently emits `AUTH REQUIRED` or `SKIP-AUTH` so the model preserves useful results and next steps

</code_context>

<specifics>
## Specific Ideas

- One plan should focus on agent-facing auth-wall guidance and a consistent partial-handoff contract: prompt text, tool descriptions, reason taxonomy, and exact handoff wording expectations.
- A second plan should focus on runtime/auth-signal integration: existing credential/login primitives, unresolved-login outcomes, and any shared/site-guide guidance that needs to feed the manual handoff path.
- The LinkedIn verification case is the concrete trigger, but the plan should stay generic enough to cover checkout, submit, and other credential-gated final actions.

</specifics>
