## 1. metadata.openclaw.install[] schema (SCAFFOLD-01)

**Confidence:** MEDIUM
**Sources:**
- https://docs.openclaw.ai/tools/skills (canonical spec)
- https://github.com/openclaw/clawhub/blob/main/docs/skill-format.md (registry-side, mirror)
- https://openclaw-ai.com/en/docs/tools/skills (mirror)
- https://lumadock.com/tutorials/openclaw-skills-guide (third-party walkthrough)
- Prior project research: .planning/research/STACK.md lines 48-67

### Finding

[OK] `install[]` is a flat array of installer-spec objects (NOT a list of shell commands, NOT a `{step, command, args}` shape). Each entry is keyed by `kind` and the remaining fields vary per `kind`.

Verbatim example from clawhub/docs/skill-format.md (canonical):

```yaml
metadata:
  openclaw:
    install:
      - kind: brew
        formula: jq
        bins: [jq]
      - kind: node
        package: typescript
        bins: [tsc]
```

Accepted `kind` values (verbatim from skill-format.md): "The supported install kinds are: `brew`, `node`, `go`, `uv`."
The docs.openclaw.ai page additionally mentions `download` as a kind (with `url`, `archive`, `extract`, `stripComponents`, `targetDir` fields). That is a fifth value.

Per-kind required fields (inferred from canonical examples; the docs do not exhaustively enumerate field-by-field requirements):

| kind     | Required | Optional |
|----------|----------|----------|
| brew     | `formula`, `bins` | `os`, `label` |
| node     | `package`, `bins` | `os`, `label` |
| go       | `package`, `bins` | `os`, `label` |
| uv       | `package`, `bins` | `os`, `label` |
| download | `url`, `bins`     | `archive`, `extract`, `stripComponents`, `targetDir`, `os`, `label` |

Universally accepted optional fields across all kinds (per docs.openclaw.ai):
- `id` -- string identifier (used in the example for `brew`; appears optional)
- `label` -- user-facing description shown in the macOS Skills UI
- `os` -- array filter `["darwin" | "linux" | "win32"]`

### Quote / evidence

From https://github.com/openclaw/clawhub/blob/main/docs/skill-format.md (verbatim quote captured via WebFetch 2026-05-08):
> "If your skill needs dependencies installed, declare them in the `install` array"
> "The supported install kinds are: `brew`, `node`, `go`, `uv`."

From https://docs.openclaw.ai/tools/skills (paraphrased; WebFetch 2026-05-08):
> "Optional installer specs used by the macOS Skills UI (brew/node/go/uv/download)."

### Open questions / assumptions

- [ASSUMED] The `id` field is optional (it appears in one of the three examples, omitted in others).
- [UNVERIFIED] No `step`, `command`, `args` fields anywhere; the schema is purely declarative-by-kind, not imperative shell-command.
- [ASSUMED] For our FSB skill, the entry will be:
  ```
  { "kind": "node", "package": "fsb-mcp-server", "bins": ["fsb-mcp-server"], "label": "Install FSB MCP server (npm)" }
  ```
  This matches the canonical `node` shape (`package` + `bins`).

## 2. requires.bins[] accepted enum (SCAFFOLD-02)

**Confidence:** HIGH
**Sources:**
- https://docs.openclaw.ai/tools/skills
- https://openclaw-ai.com/en/docs/tools/skills
- WebSearch 2026-05-08: "OpenClaw 'requires' 'bins' enum allowed values"

### Finding

[OK] `requires.bins[]` is **NOT a closed enum**. It accepts any string that should resolve as an executable on the user's PATH at skill-load time. Documented examples in OpenClaw docs include `uv`, `gemini`, and `npx`. Both `node` and `npx` are valid entries.

Verbatim example (from openclaw-ai.com docs, captured via WebSearch):
> `"requires": { "bins": ["uv"], "env": ["GEMINI_API_KEY"], "config": ["browser.enabled"] }`

For FSB skill: `["node", "npx"]` is well-formed. `npx` ships with Node 18+ but the docs do not state whether OpenClaw treats `node` as implying `npx`; declaring both is the safe path.

### Quote / evidence

From the spec (paraphrased across three doc mirrors):
> "Each binary in the `bins` array must exist on PATH ... OpenClaw checks these at load time and warns you if anything is missing."

### Open questions / assumptions

- [WARN] One known behavior bug exists: GitHub issue openclaw/openclaw#29254 "Skill eligibility checks run on gateway host, not sandbox container -- skills blocked despite binaries existing in sandbox." This means the `bins` PATH check happens on the gateway/host, not inside any sandbox. For our use this is fine (we want the user's local Node), but it is a real-world quirk worth knowing.
- [ASSUMED] Listing `["node", "npx"]` is no more strict than listing `["node"]` because both are real binaries that `npx -y fsb-mcp-server` requires.

## 3. install hook timing (SCAFFOLD-03)

**Confidence:** MEDIUM-LOW
**Sources:**
- https://docs.openclaw.ai/tools/skills
- https://github.com/openclaw/openclaw/issues/23926 ("Feature: skill:pre-install and skill:post-install Hook Events")
- https://github.com/openclaw/openclaw/issues/22459 ("openclaw onboard: skill install fails when pnpm prompts for download confirmation")
- WebSearch 2026-05-08

### Finding

[WARN] OpenClaw documentation does NOT explicitly document when `install[]` runs. The most precise statement found is from docs.openclaw.ai:
> "Optional installer specs used by the macOS Skills UI"

This phrasing strongly implies `install[]` is **driven by the macOS Skills UI install button**, not automatically on every skill load and not silently on first invocation. Corroborating evidence:

1. `install[]` is described as "**optional** installer specs". A skill loads and works fine without it; it does not gate skill availability.
2. GitHub issue #23926 (open as of 2026, feature-request) proposes new `skill:pre-install` / `skill:post-install` hook events. The issue text states the current hook system supports `gateway:startup`, `command:*`, `message:*`, and `agent:bootstrap` but **nothing fires during skill installation** -- meaning installation is currently a UI-driven user action, not an automatic lifecycle event.
3. GitHub issue #22459 confirms `openclaw onboard` triggers skill installs interactively (pnpm prompts can block it), again indicating user-initiated, not background-automatic.
4. Tencent Cloud OpenClaw guide notes: "skill updates do not install automatically. You would need to run `openclaw skills install` again."

[ASSUMED] Putting this together: OpenClaw runs `install[]`:
- when the user clicks "Install" in the macOS Skills UI on the skill detail page, OR
- when the user explicitly runs `openclaw skills install <name>` / `openclaw onboard` and accepts prompts.

It does NOT run `install[]` automatically on every skill load. It does NOT run `install[]` silently on first `/fsb` invocation.

### Quote / evidence

> "Optional installer specs used by the macOS Skills UI (brew/node/go/uv/download)." -- docs.openclaw.ai/tools/skills

> "The current hook system supports gateway:startup, command:*, message:*, and agent:bootstrap -- but nothing fires during skill installation." -- openclaw/openclaw#23926

### Open questions / assumptions

- [UNVERIFIED] Behavior on Linux/Windows (no macOS Skills UI). The most likely behavior: `install[]` is ignored at skill-load on these platforms and the skill must guide the user via its body text + scripts. Our existing milestone plan already does this via `scripts/doctor.mjs` and `scripts/install-other-hosts.mjs`, so we are already covered.
- [ASSUMED] **USAGE.md "3-step install" wording should NOT assume `install[]` ran automatically.** Phrasing should be: "If on macOS, click Install on the ClawHub detail page; otherwise run `npx -y fsb-mcp-server` (the skill handles this for you on first invocation)." The skill body's first action should be `npx -y fsb-mcp-server doctor`, which is idempotent and works regardless of whether `install[]` fired.

### command-arg-mode

[OK] Documented in https://docs.openclaw.ai/tools/skills and corroborated in .planning/research/STACK.md line 46.

Default value: `"raw"` (forwards `{ command, commandName, skillName }` unparsed). Only one documented value as of May 2026. Treat as a one-element enum.

[ASSUMED] For FSB skill: omit the field (use default). Setting it explicitly adds noise without changing behavior. This matches our roadmap-locked decision.

### ClawHub publish QA gates

From https://www.openclawplaybook.ai/blog/clawhub-skill-registry-openclaw-agents/ and clawhub README:

[OK] `clawhub skill publish <path>` runs automated security checks at publish time:
- VirusTotal scan of any binaries/scripts
- ClawScan static analysis for dangerous code patterns
- Metadata-vs-code consistency validation

[OK] **GitHub-account-age policy: account must be at least 7 days old** to authenticate via GitHub OAuth (Convex Auth). Confirmed in clawhub README. This affects Phase 253 only; the user's GitHub account `LakshmanTurlapati` predates this requirement so no issue.

Expect publish to occasionally be slow or to flag `npx -y` invocations as "downloads-from-network-at-runtime"; we should pre-empt by documenting in publish notes that `fsb-mcp-server` is the official Anthropic FSB project's npm package, owned by the same GitHub account doing the publish.

### Schema keys to avoid

[OK] Searched canonical spec (skill-format.md, docs.openclaw.ai, openclaw-ai.com mirror) for `priority` and `must-use`:

- `priority` -- **NOT FOUND** in any official OpenClaw skill schema documentation.
- `must-use` -- **NOT FOUND** in any official OpenClaw skill schema documentation.

[ASSUMED] These are likely confused with concepts from other agent frameworks (e.g., Cursor rules' `alwaysApply` or Continue.dev's rule priority). OpenClaw uses `always: true` (top-level frontmatter, default `false`) for "always-active" behavior; we already decided to omit it. There is no notion of inter-skill priority ordering in the documented spec -- skills are activated by description-matching, not by a priority field.

[OK] FSB skill's decision to avoid both keys is correct; the keys would be silently ignored at best and could confuse a future schema-validating linter at worst.
