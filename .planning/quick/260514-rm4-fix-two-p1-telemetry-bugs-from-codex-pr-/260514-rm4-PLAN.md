<objective>
Fix two P1 telemetry bugs surfaced by Codex on PR #50 with two atomic, reviewable
commits. Bug 1 is a race in `mcp-metrics-recorder.recordDispatch` where the
get->push->set rmw against `chrome.storage.local.fsbUsageData` can drop rows when
two MCP dispatches fire concurrently. Bug 2 is a payload-shape leak in
`telemetry-collector.flush` where the in-memory `attempts` retry counter (set by
`_bumpAttempts`) ends up in the JSON body, which the server's 9-field allowlist
rejects with 400 unknown_field -- causing retried events to burn through the
5-attempt cap and be permanently dropped.

Purpose: restore the BEAT-04 "INSERT OR IGNORE dedup on retry" promise and the
COST-05 "no double count / no dropped row" promise in the live anonymous
telemetry pipeline shipped in v0.9.69.

Output:
  - extension/utils/mcp-metrics-recorder.js: serialised rmw via module-level
    promise-chain lock. Broadcast still fires AFTER each successful set.
  - extension/utils/telemetry-collector.js: a `wireEvents` projection containing
    EXACTLY the 9 ALLOWED_EVENT_KEYS replaces `snapshot` inside the
    `JSON.stringify({events: ...})` body. In-memory queue retains `attempts`.
  - tests/extension-record-dispatch-serializes.test.js: NEW. 20 concurrent
    recordDispatch calls -> 20 rows present (no loss).
  - tests/telemetry-collector-strips-internal-fields.test.js: NEW. Queue
    pre-populated with `attempts: 2` events -> POST body keys == [9-key
    allowlist] sorted; `attempts` NOT present.
  - package.json: two new `&& node tests/...test.js` entries wired into scripts.test.
</objective>
