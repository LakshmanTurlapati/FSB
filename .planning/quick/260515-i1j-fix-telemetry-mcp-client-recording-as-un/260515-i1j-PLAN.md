# Quick Task 260515-i1j — Plan

## Bug

`telemetry_events.mcp_client` is `'unknown'` on 100% of rows in production (61/61 events / 2 install_uuids verified via `flyctl ssh -> sqlite3 /data/fsb-data.db`). Same root cause makes `model='unknown'` on 100% of rows (pricing module gets the same broken `client` value).

## Root cause

`extension/ws/mcp-bridge-client.js` lines 523 and 851 call `dispatchMcpToolRoute({ ..., client: this })` — passing the MCPBridgeClient **instance object** as `client`. The dispatcher (`extension/ws/mcp-tool-dispatcher.js:340-347` and `:424`) forwards `client` straight into `recordDispatch()`. The recorder (`extension/utils/mcp-metrics-recorder.js:273`) does `typeof input.client === 'string' ? input.client : 'unknown'` — the object fails the type check → `'unknown'`.

Meanwhile the MCP server **already sends the canonical normalised label** in `payload.visualSession.client` (set at `mcp/src/tools/manual.ts:180`, normalised against the 13-label allowlist at `mcp/src/tools/visual-session.ts:19`). The extension reads it for visual-session lifecycle (`extension/ws/mcp-bridge-client.js:605-611`) but never threads it into the telemetry path.

## Fix

Add a small helper `extractMcpClientLabel(payload)` at the top of `mcp-tool-dispatcher.js` that returns `payload.visualSession.client` when it's a non-empty string, else `'unknown'`. Use it at both `recordDispatch` call sites in place of the bridge-object `client`.

**Why this level:** The dispatcher is the chokepoint — both action tools (line 340) and message routes (line 424) flow through it. Fixing at the recorder would require passing `payload` to the recorder (wider contract change). Fixing at the bridge-client callers (lines 523, 851) would touch two sites + miss future callers. Dispatcher-level is one helper + two single-line edits.
