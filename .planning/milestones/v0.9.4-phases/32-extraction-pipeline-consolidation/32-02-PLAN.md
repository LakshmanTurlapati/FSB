**Part A: Update MemoryConsolidator (memory-consolidator.js)**

Update MemoryConsolidator to handle task type memories with domain + task similarity instead of pure text similarity. The consolidator performs the actual merge and returns pre-merged data with the UPDATE action.

**Changes to resolve():**
- After finding similar memories via retriever, add special handling when `newMemory.type === 'task'`:
  - Extract domain from `newMemory.metadata.domain` and task from `newMemory.typeData.session.task`
  - Filter `similar` results to only those that are also type='task' AND same domain
  - For matching task memories, compare task descriptions using `_textSimilarity` on the task field (not the full text)
  - If task similarity >= 0.7 on same domain: treat as repeat run of same task -> build merged data and return UPDATE with `mergedData`
  - If task similarity < 0.7 on same domain: ADD (different task, same site)
- For non-task memories, existing behavior unchanged (backward compat)

**Changes to _resolveHighSimilarity():**
- Add task type handling: when both are type='task', build merged data and return UPDATE
  - Return `{ action: 'UPDATE', targetId: existing.id, reason: 'merging repeat task run on {domain}', mergedData: { text, metadata, typeData } }`
  - The `mergedData` contains the MERGED result of old + new:
    - `text`: use the new memory's text (most recent description)
    - `metadata`: spread existing metadata, override with new metadata, set `updatedAt: Date.now()`
    - `typeData.session`: use new memory's session data but set `runCount = (existing.typeData.session.runCount || 1) + 1`, update outcome to new run's outcome
    - `typeData.session.timeline`: concatenate existing timeline + new timeline (full run history across runs)
    - `typeData.session.failures`: concatenate and deduplicate by string equality
    - `typeData.learned.selectors`: merge both arrays, deduplicate by `name` field (keep newer if duplicate)
    - `typeData.learned.siteStructure`: merge, deduplicate by string equality
    - `typeData.learned.patterns`: merge, deduplicate by string equality
    - `typeData.procedures`: merge, deduplicate by `name` field (keep newer if duplicate)

**Changes to _resolveMediumSimilarity():**
- For task type: if same domain and task similarity >= 0.7 from the task description fields, build merged data and return UPDATE (same merge logic)
- Otherwise ADD (different enough to be separate task)

**Add helper _mergeTaskData(existing, newMemory):**
- Extract the merge logic above into a reusable private method
- Used by both _resolveHighSimilarity and _resolveMediumSimilarity for task type
- Also used by consolidateAll for batch merging

**Changes to consolidateAll():**
- In Pass 2 (merge duplicates), group task memories by domain (not domain:type since all are type='task')
- Within each domain group, compare task descriptions (`typeData.session.task`) with `_textSimilarity`
- If task similarity >= 0.7: merge using `_mergeTaskData` -- keep the one with more accesses, update it with merged data via storage.update()

**No changes to _textSimilarity()** -- reuse existing Jaccard index.

**Part B: Update memory-manager.js UPDATE path**

In memory-manager.js `add()`, update the UPDATE handling block (around lines 43-49) to use `mergedData` when the consolidator provides it:

Replace:
```javascript
if (operation.action === 'UPDATE' && operation.targetId) {
  await this._storage.update(operation.targetId, {
    text: memory.text,
    metadata: memory.metadata,
    typeData: memory.typeData
  });
  stored.push(memory);
  continue;
}
```

With:
```javascript
if (operation.action === 'UPDATE' && operation.targetId) {
  // Use pre-merged data from consolidator if available (task memory merges),
  // otherwise fall back to simple overwrite (existing behavior for non-task types)
  const updateData = operation.mergedData || {
    text: memory.text,
    metadata: memory.metadata,
    typeData: memory.typeData
  };
  await this._storage.update(operation.targetId, updateData);
  stored.push(memory);
  continue;
}
```

This is a minimal, backward-compatible change. Non-task UPDATE actions (which don't set mergedData) work exactly as before. Task UPDATE actions use the consolidator's pre-merged data containing combined timelines, selectors, and patterns.
