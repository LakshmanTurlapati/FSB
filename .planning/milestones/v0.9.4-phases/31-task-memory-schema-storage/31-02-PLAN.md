**memory-storage.js changes:**

1. In `_addToIndex(index, memory)`: After existing domain/taskType/tags indexing, add Task Memory-specific index fields. If `memory.type === 'task'` (or more defensively, if `memory.typeData?.session`):
   - Index `outcome`: `index.outcome[memory.typeData.session.outcome]` -- array of memory IDs (create `outcome` bucket if missing on index object)
   - Index `stepCount`: bucket by range -- '0' for 0 steps, '1-5', '6-10', '10+' based on `memory.typeData.session.timeline.length` -- stored in `index.stepCount[bucket]`
   - Gracefully handle missing nested fields (guard with `?.`)

2. In `_removeFromIndex(index, memory)`: Mirror the additions -- remove from outcome and stepCount buckets if they exist.

3. In `getIndex()`: Update default return to include `outcome: {}` and `stepCount: {}` alongside existing domain/taskType/tags. Same for `deleteAll()`.

4. In `getStats()`: The existing code iterates `Object.values(MEMORY_TYPES)` -- since we added TASK to MEMORY_TYPES in Plan 01, this should auto-count. Verify this is the case (it uses a for loop: `for (const type of Object.values(MEMORY_TYPES))`). No change needed if so.

5. In `query()`: Add support for `filters.outcome` and `filters.stepCount` using the same index intersection pattern as domain/taskType/tags.

**memory-retriever.js changes:**

1. In `_keywordScore()`: Extend the `memoryText` array to include Task Memory fields when present:
   ```javascript
   // Task Memory fields
   memory.typeData?.session?.task || '',
   memory.typeData?.session?.outcome || '',
   ...(memory.typeData?.session?.timeline || []).map(s => [s.action, s.target, s.result].join(' ')),
   ...(memory.typeData?.session?.failures || []),
   ...(memory.typeData?.learned?.selectors || []),
   ...(memory.typeData?.learned?.patterns || []),
   ...(memory.typeData?.procedures || []).map(p => p.name)
   ```
   Keep existing episodic fields (typeData.task, typeData.stepsCompleted, typeData.failures) for backward compat.

2. In `_boostScore()`: Add a branch for Task Memory type before the default:
   ```javascript
   if (memory.type === MEMORY_TYPES.TASK) {
     // Task memories get a boost: base 0.15, plus 0.05 for success outcome
     const outcomeBonus = memory.typeData?.session?.outcome === 'success' ? 0.05 : 0;
     score += 0.15 + outcomeBonus;
   }
   ```
   This gives task memories a slight edge over old semantic memories (0.1 base) but doesn't dramatically outrank procedural memories with high success rates. Per CONTEXT.md: "Task Memories get a type boost in retriever scoring."

**memory-consolidator.js changes:**

1. In `resolve()`: The existing code passes `type: newMemory.type` as a filter, so task memories will only match against other task memories -- this is correct behavior.

2. In `_resolveHighSimilarity()`: The text comparison uses `_textSimilarity` on `memory.text` field. Since Task Memory `text` is "task description -- outcome", this works correctly for dedup. No structural changes needed.

3. In `_resolveMediumSimilarity()`: Add a comment noting that task memories with different outcomes for the same task should be kept (they represent different runs). The existing "different types are always additive" logic handles cross-type comparisons. For same-type task memories with medium similarity, the existing "add as complementary" default is correct. No code changes needed unless the same-type branch has special handling -- review and ensure task type follows the ADD path for medium similarity.

   let metaLine;

     const outcomeBadge = sess.outcome
       ? `<span class="outcome-${sess.outcome === 'success' ? 'success' : sess.outcome === 'failure' ? 'failure' : sess.outcome === 'partial' ? 'partial' : 'unknown'}">${(sess.outcome || 'unknown').charAt(0).toUpperCase() + (sess.outcome || 'unknown').slice(1)}</span>`

     const taskDomain = sess.domain || memory.metadata?.domain || 'Unknown';
     let durationStr = '';
     if (sess.duration && sess.duration > 0) {
