- `card-detail-due-date`: Due date section in card detail
- `card-detail-checklist`: Checklist section in card detail

**Guidance text (keyboard-first):**
Document Trello keyboard shortcuts:
- N: New card at bottom of current list
- ? : Show all shortcuts
- / : Focus search
- B: Board menu/switcher
- W: Toggle sidebar
- F: Open filter
- Q: Toggle "My Cards" filter
- Arrow keys: Navigate between cards when card is focused
- Enter: Open selected card
- E: Quick edit selected card
- C: Archive selected card
- L: Open labels on selected card
- D: Open due date on selected card

- M: Add/remove members from card

Trello-specific patterns:
- Cards are in lists, lists are on boards
- Card detail opens as a modal overlay -- wait for [data-testid="card-back"] or [role="dialog"]
- Moving cards between lists: Enter to open card > Move button > select destination list > click Move (keyboard-first alternative to drag)
- Creating cards: Click "Add a card" at bottom of list, or press N with list focused

**Workflows (6-8):**
1. `createCard`: Focus list > N or click "Add a card" > type title > Enter to save
2. `moveCard`: Open card (Enter) > click Move > select destination list > click Move button (keyboard alternative to drag)
3. `editCard`: Navigate to card > E for quick edit OR Enter to open detail > modify fields
4. `createList`: Click "Add another list" > type name > Enter to save
5. `addLabel`: Open card > L to open labels > select or create label
6. `setDueDate`: Open card > D to open date picker > select date > Save
7. `searchCards`: Press / > type query > Enter
8. `navigateBoard`: Arrow keys between cards, Tab between lists, B for board switcher

**Warnings (8-12):**
- Card detail is a modal overlay -- wait for it to render before interacting
- "Add a card" button is per-list -- each list has its own add button at the bottom
- Trello uses react-beautiful-dnd for drag -- synthetic mouse events may not trigger it. Use Move button in card detail as reliable alternative
- Board canvas scrolls horizontally -- lists beyond viewport require horizontal scroll
- Card composer (add card input) disappears on click outside -- type immediately after clicking
- Archived cards are NOT deleted -- they can be restored from archive
- Labels have colors AND text -- clicking a label color without text creates an unnamed label
- Checklist items in card detail use their own Enter key handling

Add a `dragdrop` mechanical action to content/actions.js that performs a coordinated mouse event sequence to simulate drag-and-drop operations.

**Function signature and behavior:**
```javascript
async function executeDragDrop(sourceRef, targetRef, options = {}) {
  // sourceRef: element reference (e.g., "e5") - the element to drag
  // targetRef: element reference (e.g., "e12") - the drop destination
  // options.steps: number of intermediate mousemove events (default: 10)
  // options.holdMs: milliseconds to hold at source before dragging (default: 150)
  // options.stepDelayMs: milliseconds between mousemove steps (default: 20)
  // Returns: { success: boolean, method: string, error?: string }
}
```

**Implementation approach -- try 3 methods in sequence:**

**Method 1: HTML5 DragEvent sequence** (works for native HTML5 drag-and-drop):
```javascript
const dt = new DataTransfer();
source.dispatchEvent(new DragEvent('dragstart', { bubbles: true, cancelable: true, dataTransfer: dt }));
// Small delay for drag state initialization
target.dispatchEvent(new DragEvent('dragenter', { bubbles: true, cancelable: true, dataTransfer: dt }));
target.dispatchEvent(new DragEvent('dragover', { bubbles: true, cancelable: true, dataTransfer: dt }));
target.dispatchEvent(new DragEvent('drop', { bubbles: true, cancelable: true, dataTransfer: dt }));
source.dispatchEvent(new DragEvent('dragend', { bubbles: true, cancelable: true, dataTransfer: dt }));
```

**Method 2: Pointer event sequence** (works for react-beautiful-dnd and similar):
```javascript
const sourceRect = source.getBoundingClientRect();
const targetRect = target.getBoundingClientRect();
const startX = sourceRect.left + sourceRect.width / 2;
const startY = sourceRect.top + sourceRect.height / 2;
const endX = targetRect.left + targetRect.width / 2;
const endY = targetRect.top + targetRect.height / 2;

// Pointerdown on source
source.dispatchEvent(new PointerEvent('pointerdown', { bubbles: true, cancelable: true, clientX: startX, clientY: startY, pointerId: 1 }));
// Hold for holdMs to trigger drag recognition
await sleep(options.holdMs || 150);
// Intermediate moves (react-beautiful-dnd needs these to trigger drag mode)
for (let i = 1; i <= steps; i++) {
  const t = i / steps;
  const x = startX + (endX - startX) * t;
  const y = startY + (endY - startY) * t;
  document.dispatchEvent(new PointerEvent('pointermove', { bubbles: true, cancelable: true, clientX: x, clientY: y, pointerId: 1 }));
  await sleep(options.stepDelayMs || 20);
}
// Drop
source.dispatchEvent(new PointerEvent('pointerup', { bubbles: true, cancelable: true, clientX: endX, clientY: endY, pointerId: 1 }));
```

**Method 3: MouseEvent sequence** (fallback, works for basic drag implementations):
Same as Method 2 but using `MouseEvent` instead of `PointerEvent`, with `mousedown`, `mousemove`, `mouseup`.

**Try methods in order.** After each method, check if the DOM changed (source element moved, or target's children changed). If DOM changed, return `{ success: true, method: 'html5_drag' | 'pointer_events' | 'mouse_events' }`. If none work, return `{ success: false, error: 'All drag methods failed - use keyboard move alternative' }`.

**Integration with existing action dispatch:**
Register this action in the action handler switch/dispatch (find where other mechanical actions like `fillsheet` are registered). The CLI command format should be:
```
dragdrop e5 e12
```
Where e5 is source element ref and e12 is target element ref.

Also register the command in the CLI parser if needed (check how fillsheet is registered).

**Important:** This is a best-effort tool. It will work for some apps (native HTML5 drag) and may fail for others (custom drag libraries). The return value indicates which method succeeded. Guide files should document keyboard alternatives as fallback when dragdrop returns `{ success: false }`.
