# Requirements -- v0.9.60 Multi-Agent Tab Concurrency (MCP 0.8.0)

**Milestone goal:** Let multiple agents drive FSB in parallel, each isolated to its own tab(s), with explicit ownership and a configurable concurrency cap -- and ship `fsb-mcp-server@0.8.0` including the deferred Phase 236 fix.

**Status:** Closeout verified 2026-05-08. Original requirements plus Phase 245 closeout amendments are complete. Phase 246/247 protocol fixes are tracked as gap-closure design contracts against the original OWN/AGENT scope.

---

## v1 Requirements

### AGENT -- Identity & registry

- [x] **AGENT-01**: Each new MCP session/task gets a unique `agent_id` minted by FSB via `crypto.randomUUID()`; callers cannot invent IDs
- [x] **AGENT-02**: Agent registry mirrored to `chrome.storage.session` (write-through); survives MV3 service-worker eviction within a browser session
- [x] **AGENT-03**: On service-worker wake, registry hydrates and reconciles against `chrome.tabs.query({})`, dropping ghost records before servicing requests
- [x] **AGENT-04**: One MCP client may run multiple parallel agents simultaneously, each with its own `agent_id`

### OWN -- Tab ownership enforcement

- [x] **OWN-01**: Every tab opened by an agent binds `tab_id` -> `agent_id` authoritatively in the registry
- [x] **OWN-02**: All MCP tool dispatch flows through a single chokepoint that verifies `(agent_id, tab_id, ownership_token)` before invoking the handler
- [x] **OWN-03**: Cross-agent tab access rejected loudly with typed `TAB_NOT_OWNED` (including the actual owner's `agent_id`)
- [x] **OWN-04**: v0.9.36 visual-session manager rejects cross-agent `startSession` with `tab_owned_by_other_agent` (preserves displacement for same-agent re-entry)
- [x] **OWN-05**: Incognito and cross-window tab IDs rejected at the dispatch boundary

### POOL -- Cap + forced-new-tab pooling

- [x] **POOL-01**: Concurrency cap (default 8, configurable 1-64) enforced via promise-chain mutex; cap-check + insert atomic against in-memory registry
- [x] **POOL-02**: `(N+1)`th agent claim rejects with typed `AGENT_CAP_REACHED { cap: N, active: M }` where `N` is the current configured cap -- fail-loud, no silent queueing
- [x] **POOL-03**: New tabs opened by an owning agent (`chrome.tabs.onCreated` + `openerTabId`) auto-pool under that agent
- [x] **POOL-04**: Pool of multiple tabs stays locked together; closing one tab shrinks the pool; agent only fully released when `pool.size === 0`
- [x] **POOL-05**: Concurrency cap configurable in `options.html` -> Advanced Settings, with min 1 / max 64 / default 8; persisted to `chrome.storage.local`; changes apply at next claim. **Grandfather behavior:** active agents keep their slot when cap is lowered (no forced eviction); new claims gated by the new value
- [x] **POOL-06**: Cap setting validated on save (clamped to 1-64, integer); invalid values rejected with inline error; reset-to-default control available

### LOCK -- Lock release lifecycle

- [x] **LOCK-01**: Lock released when task/session ends via existing `finalizeSession` path
- [x] **LOCK-02**: Lock released when MCP client disconnects, after a `RECONNECT_GRACE_MS` window (default 10s, calibrated during Phase 5 planning) keyed by `connection_id`
- [x] **LOCK-03**: Lock released when user closes the tab (`chrome.tabs.onRemoved` + `chrome.windows.onRemoved`); idempotent + commutative
- [x] **LOCK-04**: No idle timeout -- agents stay locked indefinitely while active

### BG -- Background-tab execution

- [x] **BG-01**: All existing 25+ MCP/autopilot tools audited for `chrome.tabs.update({active: true})` side effects
- [x] **BG-02**: Per-tool `force_foreground` flag in `tool-definitions.cjs` (default false); only foreground-required tools opt in
- [x] **BG-03**: Long `setTimeout`-based waits >1s swapped to `chrome.alarms` for background-tab throttle resilience
- [x] **BG-04**: `webNavigation.onCommitted` detects user-initiated navigation on agent-owned tabs and emits a pause signal

### BACK -- Back-navigation tool

- [x] **BACK-01**: New `back` MCP tool registered, ownership-gated, single-step (no multi-step `back(n)`, no `forward` companion in v0.9.60)
- [x] **BACK-02**: `history.length` precheck before `chrome.tabs.goBack`; returns `NO_BACK_HISTORY` when depth <= 1
- [x] **BACK-03**: Structured result: `{ status: 'ok' | 'no_history' | 'cross_origin' | 'bf_cache' | 'fragment_only', resultingUrl, historyDepth }`
- [x] **BACK-04**: `pageshow`-based settle verification with 2s timeout
- [x] **BACK-05**: Cross-origin content-script re-injection via existing v0.9.11 BF-cache resilience path

### MCP -- Server 0.8.0 + Phase 236

- [x] **MCP-01**: `fsb-mcp-server` bumped to `0.8.0` (`mcp/package.json`, `mcp/server.json`)
- [x] **MCP-02**: `@modelcontextprotocol/sdk` bumped from `^1.27.1` to `^1.29.x`; Zod stays on 3.x
- [x] **MCP-03**: `run_task` returns on completion via `fsbAutomationLifecycleBus` (Phase 236)
- [x] **MCP-04**: 300s ceiling raised to 600s safety net (kept until cleanup-path audit confirms zero dropped lifecycle events)
- [x] **MCP-05**: 30s heartbeat ticks emitted via `_sendProgress` so MCP host clients don't time out long-running tasks
- [x] **MCP-06**: Task lifecycle persisted in `chrome.storage.session` keyed by `task_id` (resumable across SW wake)
- [x] **MCP-07**: MCP tool descriptions updated: `agent_id` required, `tab_id` agent-scoped, cap configurable, ownership enforced -- for `run_task`, all manual tools, `back`
- [x] **MCP-08**: `fsb-mcp-server@0.8.0` prepared for tag-driven npm publish with CHANGELOG and README updates; actual `npm publish` remains user-gated

### UI -- Badge integration & settings

- [x] **UI-01**: v0.9.36 trusted-client badge renderer extended to append short `agent_id` (e.g., "Claude / agent_a3f1") -- closes v0.9.36 deferred badge/glow-collision gap
- [x] **UI-02**: Sidepanel and popup show read-only "owned by Agent X" badge on owned tabs (no enforcement, visibility only)
- [x] **UI-03**: "Concurrency Cap" control rendered in `options.html` Advanced Settings panel (numeric input or slider, range 1-64, default 8, label, helper text explaining trade-offs, current-active counter for context)

### TEST -- Regression + new coverage

- [x] **TEST-01**: Existing MCP / visual-session / dashboard / bridge-client contract tests pass unchanged
- [x] **TEST-02**: New multi-agent regression suite: N parallel agents (where N = configured cap), `(N+1)`th rejected with `AGENT_CAP_REACHED`, all release cleanly on disconnect
- [x] **TEST-03**: SW eviction + wake reconciliation tested (registry rehydrates, ghost records dropped)
- [x] **TEST-04**: 20-concurrent-claim stress test confirms cap invariant under TOCTOU pressure (with cap = 8)
- [x] **TEST-05**: Tab-ID reuse race test (agent A's queued action on closed tab T1 cannot corrupt agent B's reused tab T1)

### CHANGE -- Closeout amendment: post-action change report

- [x] **CHANGE-01**: Every non-read MCP action returns a compact `change_report` field on success unless explicitly opted out
- [x] **CHANGE-02**: `change_report` captures URL/title deltas, opened dialogs, nearby added/removed/changed nodes, input changes, focus shift, mutation count, settle time, and truncation state
- [x] **CHANGE-03**: Read-only tools and noisy opt-out tools omit `change_report`
- [x] **CHANGE-04**: Reports are size-capped with `truncated: true` and a hint to call `read_page` when needed
- [x] **CHANGE-05**: Global Advanced Settings toggle can suppress action change reports

## Out of Scope

- **Cross-window agents** -- agents bound to single Chrome window; cross-window coordination not in v0.9.60
- **Incognito agents** -- explicit reject at dispatch boundary; manifest posture confirms no incognito access
- **Headless server-side workers** -- FSB stays browser-attached; no detached compute
- **Agent-to-agent messaging** -- agents are isolated; coordination must happen at the calling client
- **Per-agent fingerprint / stealth profiles** -- shared Chrome profile is intentional
- **Dirty-form `back` confirmation** -- Chrome's native `beforeunload` already handles this; layering is a known anti-pattern
- **Multi-step `back(n)`** -- single-step only; callers compose if needed
- **Cross-agent `back`** -- only the owning agent can navigate its tabs
- **Per-agent queue with per-agent rate limits** -- one queue per MCP process is sufficient for v0.9.60
- **Persistent agent identity across browser restart** -- `chrome.storage.session` (browser-lifetime); intentional
- **Multi-agent live-preview pane in dashboard** -- single-stream + click-to-switch; multi-stream UI is a separate milestone
- **Auto-promotion of orphaned tabs** -- if a user manually closes the last tab of an agent and the agent disconnects, slot is fully released; orphans are not promoted

## Gap-Closure Design Contracts

Phase 246 and Phase 247 did not add new external requirement IDs. They close protocol gaps against AGENT-04 and OWN-01..05:

- [x] **GAP-246-01**: Read tools, visual-session, action tools, and vault tools resolve target tabs through agent-scoped registry state instead of active foreground tab state
- [x] **GAP-246-02**: `open_tab` defaults to background and does not steal focus unless explicitly requested
- [x] **GAP-247-01**: `open_tab`, zero-owned `navigate`, `switch_tab`, and `list_tabs` are bootstrap-safe from a restricted active tab
- [x] **GAP-247-02**: Cross-owner recovery attempts still reject before navigation or focus transfer
- [x] **GAP-247-03**: MCP error mapping reports Agent scope or Tab ownership instead of generic Page navigation for resolver/ownership failures
