# Phase 14.3: Fix Sheets cell edit mode escape and Name Box navigation sequence - Research

**Researched:** 2026-02-26
**Domain:** Google Sheets prompt engineering -- cell edit mode escape before Name Box navigation
**Confidence:** HIGH

## Summary

This phase fixes a specific failure mode observed in session log `fsb-session-2026-02-26-1772124534725.json`: when the AI types data into a cell and then attempts to navigate to another cell via the Name Box, Google Sheets is still in cell edit mode. The cell reference typed into the Name Box (e.g., "B1") gets appended to the current cell's content instead of triggering navigation, resulting in corrupted data (e.g., "random test datab1" in cell A1). The fix is purely prompt-level -- adding an explicit Escape keystroke before every Name Box navigation instruction across three files.

The formatting directive already solves this for formatting operations with "STEP 0 -- EXIT EDIT MODE: Press Escape to ensure you are not in cell edit mode." The same pattern must be applied to data entry Name Box navigation in the site guide, the TASK_PROMPTS multitab section, and the buildMinimalUpdate continuation reminder.

**Primary recommendation:** Prepend `keyPress Escape` before every "Click the Name Box" instruction in site guide workflows, TASK_PROMPTS multitab Sheets section (inserting a new Step 0), and buildMinimalUpdate continuation reminder. Add a CRITICAL warning to the site guide warnings array.

## Architecture Patterns

### Pattern 1: Escape-Before-NameBox (Mirroring Formatting STEP 0)

**What:** Every instruction that begins with "Click the Name Box" must be preceded by "Press Escape to exit any cell edit mode" as a preparatory step.

**When to use:** Any time the AI transitions from cell editing (typing data) to Name Box navigation. In Google Sheets, typing data puts the cell into edit mode. While in edit mode, clicks and keystrokes are directed to the cell editor, not to other UI elements. Pressing Escape exits edit mode and returns control to the sheet-level selection state.

**Existing implementation (formatting directive, line 470-472):**
```javascript
STEP 0 -- EXIT EDIT MODE:
Press Escape to ensure you are not in cell edit mode.
This is critical -- formatting shortcuts do NOT work in edit mode.
```

**New implementation for data entry Name Box navigation (TASK_PROMPTS multitab, line 235-246):**
```
WRITING TO GOOGLE SHEETS:
- Google Sheets uses a CANVAS-BASED GRID. You CANNOT click individual cells.
- CELL NAVIGATION VS DATA ENTRY -- these are SEPARATE steps:
  Step 0: Press Escape to exit cell edit mode (CRITICAL before Name Box navigation)
  Step 1: Click the Name Box (top-left input showing current cell ref like "A1")
  Step 2: Type the target cell reference (e.g., "A1", "B3") into the Name Box
  Step 3: Press Enter to navigate to that cell
  Step 4: Type the actual data value (keystrokes go to the now-active cell)
  Step 5: Press Tab to move right to next column, or Enter to move down to next row
```

### Pattern 2: Workflow Array Prepend in Site Guide

### Anti-Patterns to Avoid
- **Modifying the formatting directive:** It already has STEP 0 Escape -- leave it alone.
- **Modifying the fillSheetData data entry directive:** That path uses the fillSheetData tool (clipboard paste), not manual Name Box navigation. No Escape needed.
- **Modifying MINIMAL_CONTINUATION_PROMPT constant:** Sheets-specific content belongs in the conditional injection within buildMinimalUpdate, not the generic constant.
- **Adding Escape to workflows that already have it:** `renameSheet` and `formatHeaderRow` already start with "Press Escape" -- do not duplicate.
- **Over-complicating the guidance text wording:** Keep it short and imperative. "Press Escape to exit any cell edit mode" is the established pattern.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Escape injection | Programmatic Escape key injection before Name Box clicks | Prompt-level instruction | The AI decides actions; prompt guidance is the correct lever. Adding code-level Escape injection would bypass AI reasoning and could cause issues in non-edit-mode contexts. |
| Edit mode detection | Content script check for Sheets edit mode state | Prompt instruction to always Escape before Name Box | Sheets edit mode state is internal to the canvas editor and not easily detectable via DOM. The Escape key is safe to press even when not in edit mode (it is a no-op on the sheet-level selection state). |

**Key insight:** Escape is always safe to press. In edit mode, it exits. Not in edit mode, it is a no-op (deselects, which is fine before clicking Name Box). The prompt instruction should state this unconditionally -- no conditional logic needed.

## Common Pitfalls

### Pitfall 1: Forgetting enterHyperlinkFormula
**What goes wrong:** The enterHyperlinkFormula workflow says "Navigate to the target cell via Name Box" as prose, not as a step array reference. If the planner only looks for "Click the Name Box" as a pattern, they miss this.
**Why it happens:** The instruction uses indirect language ("Navigate to the target cell via Name Box") instead of the explicit "Click the Name Box" step.
**How to avoid:** Add an explicit Escape step to the enterHyperlinkFormula workflow: change step 1 to "Press Escape to exit any cell edit mode, then navigate to the target cell via Name Box (click Name Box, type cell ref, press Enter)".
**Warning signs:** Search for "Name Box" in all workflow arrays, not just "Click the Name Box".

### Pitfall 2: Step Renumbering in TASK_PROMPTS
**What goes wrong:** If the planner inserts Step 0 but does not renumber the existing Steps 1-5, the AI sees two "Step 1" entries (the old Step 1 and the new Step 0-was-1).
**Why it happens:** The current steps are numbered 1-5. Inserting Step 0 before Step 1 means the existing steps must be renumbered to 1-5 (Escape is Step 0, Click Name Box becomes Step 1, etc.).
**How to avoid:** The CONTEXT.md explicitly states: "renumber steps to 0-5". Verify the final text has exactly steps 0, 1, 2, 3, 4, 5 with no duplicates.
**Warning signs:** grep for "Step 1" and confirm it appears exactly once.

### Pitfall 3: Guidance text NAME BOX NAVIGATION section needs Escape too
**What goes wrong:** The site guide `guidance` text (not just workflows) has a "NAME BOX NAVIGATION (PRIMARY METHOD):" section (lines 30-36) that lists steps 1-6. If Escape is only added to workflows but not to this guidance prose, the AI may still follow the prose and skip Escape.
**Why it happens:** The guidance text and workflows are separate sections of the site guide. The guidance text is the narrative description; workflows are step arrays. Both are injected into the AI prompt.
**How to avoid:** Add an Escape step (Step 0) to the NAME BOX NAVIGATION numbered list in the guidance text. Renumber the existing steps 1-6 to 1-7 (or insert Step 0 and keep 1-6 as 1-6, making it 0-6).
**Warning signs:** Read the full guidance text after modification and verify every "Click the Name Box" instruction is preceded by an Escape instruction.

### Pitfall 4: appendToExistingSheet has inline Name Box references
**What goes wrong:** The appendToExistingSheet workflow step 1 says "Click the Name Box, type 'A1', press Enter to go to the top". This is a compound instruction in one string. Adding "Press Escape" as a separate array element before it works, but the compound step should also include the Escape note for clarity.
**Why it happens:** Some workflows use compact compound steps rather than one-action-per-step.
**How to avoid:** Prepend "Press Escape to exit any cell edit mode" as a new first step in the array. The existing compound step can stay as-is since the Escape step comes first.

### Pitfall 5: applyAlternatingColors and applyLinkColumnBlueText start with "Click the Name Box"
**What goes wrong:** These formatting-adjacent workflows are in the site guide workflows object (not in the formatting directive). They start with "Click the Name Box" for range selection but do not have Escape.
**Why it happens:** The formatting directive (buildSheetsFormattingDirective) has its own STEP 0 Escape, but these site guide workflows are separate and would be referenced independently.
**How to avoid:** Add Escape as first step in both workflow arrays, matching the CONTEXT.md list of workflows to update.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| No Escape before Name Box nav (data entry) | Escape before Name Box nav (formatting only) | Phase 13 (formatting directive) | Formatting works; data entry still fails when AI is in edit mode |
| 5-step Sheets procedure (Phase 14.2) | 6-step procedure with Escape as Step 0 | Phase 14.3 (this phase) | Eliminates "random test datab1" corruption pattern |

**Root cause analysis from session log:**
- Iteration 3: AI typed "Random Test Data" into cell A1 via formula bar (e41), pressed Enter. Cell committed, but Sheets may remain in edit mode depending on Enter behavior.
- Iteration 4: AI typed "B1" targeting Name Box (e43), but the "B1" went into A1's edit buffer instead. Result: A1 content became "random test datab1".
- Iteration 5: AI pressed Tab on e1 (the cell editor combobox), moving right but still in edit mode context.
- Iteration 6: AI clicked Name Box (e42) explicitly -- this time the click worked because the Tab had committed the cell and exited edit mode.
- Iteration 7: Name Box showed "B1" and data entry succeeded.

The 2-iteration waste (iterations 4-5) would have been prevented by a single Escape press before the Name Box click attempt in iteration 4.

## Open Questions

1. **Escape-first on initial navigation (first cell)**
   - What we know: When the sheet first loads, the AI is NOT in edit mode, so Escape is technically unnecessary before the first Name Box click. However, Escape is harmless (no-op when not in edit mode).
   - What's unclear: Could Escape cause any undesirable side effect on a freshly loaded sheet (e.g., closing a dialog that happened to open)?
   - Recommendation: Always include Escape regardless -- the no-op cost is negligible, and having a conditional rule ("Escape only if you just typed data") is more complex for the AI to follow than an unconditional rule ("always Escape before Name Box").
