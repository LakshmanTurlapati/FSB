  <action>
Create `scripts/parse-research-logs.js` -- a Node.js script (no external dependencies, Node built-ins only: fs, path) that performs these sequential steps:

**Step 1: Scan** -- Read all `fsb-research-*.json` files from `Logs/` directory. Skip files matching:
- `fsb-session-*.json` (session logs, not research)
- Domain `www.google.com` (Google search logs -- but extract career URL from the `text` field of the search combobox element if present, for later use as a URL-only guide)
- Domain `docs.google.com` (not career)
- Domain `www.qatarairways.com` (not career)

For Google search logs (3 files), parse the search box text to extract career URLs for Citi, Google Careers, and AmEx. Generate LOW-confidence URL-only guides for these 3 companies.

**Step 2: Group by company** -- If multiple logs exist for the same domain, group them. When combining, union all selectors from all sessions. For interaction ordering conflicts, the most recent session (by endTime) wins.

**Step 3: Parse each log** -- For each page in the log, extract:
- `interactiveElements[]` -- each has: type, text, id, class, selectors[], elementId
- `internalLinks[]` -- each has: text, url
- `forms[]` -- form elements
- `headings[]` -- page headings
- Page metadata: url, title, depth

Extract from ALL page depths (0, 1, 2, etc.). Even incomplete/partial logs get what's there.

**Step 4: Filter career-relevant elements** -- From the full extraction, identify career-relevant interactive elements using these keyword heuristics on element text, id, class, and aria-labels:
- searchBox: "search", "keyword", "find job", "query"
- locationFilter: "location", "city", "state", "where", "region", "country"
- departmentFilter: "department", "team", "category", "area", "function", "group"
- jobCards: "job", "position", "opening", "listing", "result", "posting", "role", "career"
- jobTitle: "title", "role", "position" (when inside a job card context)
- applyButton: "apply", "submit application", "interested"
- pagination: "next", "page", "previous", "prev", "show more", "load more", "view more"
- resultsContainer: "results", "listings", "openings", "positions"
- cookieDismiss: "cookie", "accept", "consent", "dismiss", "close" (for interaction sequence)

Also extract the career URL from `startUrl` or from internal links that match career path patterns (/careers, /jobs, /job-search, etc.).

**Step 5: Classify selector stability** -- For each selector string:
- STABLE: starts with `#` (id), starts with `[aria-` (aria attributes), starts with `[role=` (role), starts with `[data-` (data attributes), starts with `[name=` (name)
- STABLE (XPath): contains `normalize-space`, `contains(text()`, `contains(.,` (text-based XPath)
- MODERATE: starts with `.` but NOT hashed (semantic class names), type selectors, name attributes
- UNSTABLE: starts with `//` and is positional XPath (no text matching), matches hashed class pattern `/[a-zA-Z]+-[a-f0-9]{5,}|[a-f0-9]{5,}[a-zA-Z]+|css-[a-z0-9]+|sc-[a-z0-9]+/`

Prefer stable selectors first when building the guide's selector lists. Include unstable selectors but flag them (append them after stable alternatives in comma-separated lists).

**Step 6: Detect ATS platform** -- For each company log, detect ATS:
- URL pattern matching: `myworkdayjobs.com` -> workday, `boards.greenhouse.io` or `grnh.se` -> greenhouse, `jobs.lever.co` or `lever.co/` -> lever, `icims.com` or `jobs-*.icims.com` -> icims, `taleo.net` or `oracle*taleo` -> taleo
- DOM signature detection: count occurrences of ATS keywords (workday, greenhouse, lever, icims, taleo, phenom) in the full JSON stringified log. If a keyword appears >10 times, flag that ATS.
- Only return the single best match (or null). Priority: URL pattern match > DOM signature match.
- Store detected ATS in the guide output so Phase 10 knows which ATS base guide to reference.

**Step 7: Compute confidence score** -- For each company:
```
Career-relevant element categories found (0-8): searchBox, locationFilter, departmentFilter, jobCards, jobTitle, applyButton, pagination, resultsContainer
coverage = count of categories with at least one selector

stable_ratio = (STABLE selectors) / (total selectors)  [0.0 to 1.0]

Confidence:
  HIGH:   coverage >= 4 AND stable_ratio >= 0.5
  MEDIUM: coverage >= 2 OR (coverage >= 1 AND stable_ratio >= 0.3)
  LOW:    everything else (coverage < 2 AND (total_elements < 3 OR stable_ratio < 0.3))
```

**Step 8: Generate guide files** -- For each company, write a .js file to `site-guides/career/` using this exact format:

```javascript
registerSiteGuide({
  site: '{Company Name}',
  category: 'Career & Job Search',
  confidence: '{HIGH|MEDIUM|LOW}',
  ats: '{workday|greenhouse|lever|icims|taleo|null}',
  careerUrl: '{direct career URL}',
  patterns: [/{domain pattern}/i],
  guidance: `{COMPANY}-SPECIFIC CAREER INTELLIGENCE:\n...`,
  selectors: {
    searchBox: '{selectors}',
    locationFilter: '{selectors}',
    ...
  },
  workflows: {
    searchJobs: [
      '{ordered interaction steps}'
    ]
  },
  warnings: ['{stability warnings, dynamic loading notes}'],
  toolPreferences: ['navigate', 'type', 'click', 'scroll', 'getText', 'getAttribute', 'waitForElement']
});
```

Machine-optimized, minimal format per user decision: NO comments, NO human-readable explanations in the generated files. The `guidance` field is a concise multi-line string with navigation tips (not prose -- just selector references and step descriptions).

For LOW-confidence companies: generate a minimal guide with just `site`, `category`, `confidence: 'LOW'`, `careerUrl`, `patterns`, and a guidance string saying to use generic ATS fallback. Include whatever selectors exist (even if few).

**File naming convention:** Use the domain name, kebab-cased, extracting the meaningful company portion:
- careers.microsoft.com -> microsoft.js
- www.amazon.jobs -> amazon.js
- jobs.boeing.com -> boeing.js
- www.metacareers.com -> meta.js
- careers.bankofamerica.com -> bankofamerica.js
- usa.visa.com -> visa.js
- www.apple.com -> apple.js
- etc.

2. Write to stdout: the importScripts lines that need to be added to background.js (for human reference -- Plan 03 derives the list via directory scan)

The script should be idempotent -- running it again overwrites existing generated guides with fresh output.

**Important constraints per user decisions:**
- Flexible structure per company (no forced standard interface). If a company has no searchBox selectors, omit that key entirely. Do NOT pad with empty strings.
- Interaction sequence included in workflows.searchJobs: ordered steps like cookie dismiss -> search -> filter -> results
- Selectors and URLs only -- no behavioral metadata (auth walls, AJAX quirks)
- Even all-hashed-selector sites get a guide (something is better than nothing)
- No separate sitemap JSON files -- the .js guide files are the sole output
- Deduplicate elements across pages by selector string (shared nav repeats on every page)
