# Phase 14.2: Fix AI Google Sheets Cell Navigation and Data Entry Confusion - Research

**Researched:** 2026-02-26
**Domain:** AI prompt engineering for Google Sheets manual cell interaction
**Confidence:** HIGH

## Summary

The bug occurs when the AI manually interacts with Google Sheets (non-fillSheetData tasks) and confuses Name Box cell reference typing with cell value typing. The AI types "b1" as a value into the current cell instead of navigating to B1 via the Name Box first, then typing actual data. The root cause is a gap in the AI's prompt guidance: the site guide's `guidance` text describes the Name Box workflow conceptually, but does not provide explicit step-by-step instructions that disambiguate "type a cell reference into the Name Box" from "type a cell value into the active cell." The AI conflates these two distinct operations because they both use the `type` tool but target different elements.

The fix is a prompt-level enhancement. Three specific deficiencies need to be addressed: (1) the `multitab` TASK_PROMPTS "WRITING TO GOOGLE SHEETS" section is too brief (only 3 lines) and lacks the step-by-step disambiguation that the site guide provides, (2) the site guide's `guidance` text itself does not explicitly warn against typing cell references as values, and (3) on continuation iterations using the MINIMAL_CONTINUATION_PROMPT, ALL Sheets-specific guidance is lost because the minimal prompt has zero Sheets-specific content.

**Primary recommendation:** Add explicit "GOOGLE SHEETS CELL NAVIGATION" instructions to the system prompt that clearly separate the two-step process (Step 1: click Name Box, type cell ref, press Enter to NAVIGATE; Step 2: type actual data value into the now-selected cell). Enhance both the site guide guidance text and the multitab task prompt. For continuation iterations, inject a Sheets-specific addendum when the URL matches a Sheets pattern.

## Architecture Patterns

### How the AI Prompt System Works for Google Sheets

The prompt injection follows this flow:

```
1. getGuideForTask(task, currentUrl) -- URL pattern match
   -> Returns google-sheets.js site guide (patterns: docs.google.com/spreadsheets, sheets.google.com)

2. detectTaskType(task, currentUrl, siteGuide)
   -> siteGuide.category = 'Productivity Tools' maps to guideTaskType = 'general'
   -> UNLESS task contains write/enter/fill + sheets keywords -> returns 'multitab'
   -> For manual interaction like "type data in cells", returns 'general'

3. buildSystemPrompt / buildPrompt (first iteration or domain change)
   -> Full system prompt includes:
      a. BATCH_ACTION_INSTRUCTIONS (with Sheets warning from Phase 14.1)
      b. getToolsDocumentation(taskType, siteGuide) -- tools from siteGuide.toolPreferences
      c. _buildTaskGuidance(taskType, siteGuide, currentUrl, task) -- builds full guidance:
         i.  Productivity Tools category guidance (_shared.js)
         ii. SITE-SPECIFIC GUIDANCE (google-sheets.js guidance text)
         iii. KNOWN ELEMENT IDENTIFIERS (selectors)
         iv. COMMON WORKFLOWS (all workflow arrays)
         v.  WARNINGS (all 19 warnings)

4. buildMinimalUpdate (continuation iterations -- NOT first, NOT stuck, NOT domain-changed)
   -> Uses MINIMAL_CONTINUATION_PROMPT as system prompt
   -> NO Sheets guidance, NO site guide, NO category guidance
   -> Only DOM snapshot, last action result, and generic rules
```

### Critical Finding: Continuation Iterations Lose ALL Sheets Context

When `isFirstIteration = false` AND `isStuck = false` AND `isDomainChanged = false`, the system prompt is the MINIMAL_CONTINUATION_PROMPT which contains ZERO Sheets-specific guidance. The AI relies entirely on its conversation history from the first iteration. If the AI's initial reasoning did not establish a strong mental model of the Name Box workflow, continuation iterations will not reinforce it.

However, the multi-turn conversation approach means the AI DOES have its previous system prompt in conversation history. The issue is that the guidance itself is ambiguous enough that even with it in context, the AI conflates the two steps.

### The Name Box vs Cell Value Confusion

The site guide's guidance text says:

```
NAME BOX NAVIGATION (PRIMARY METHOD):
1. Click the Name Box element (#t-name-box) showing current cell reference like "A1"
2. The Name Box text will be selected/highlighted
3. Type the target cell reference (e.g., "A1", "B3", "D10")
4. Press Enter to navigate to that cell
5. The cell is now selected and ready for input
6. Type the cell value -- it will go into the formula bar and cell
```

This is conceptually clear, but the AI sees it as a TEXT description alongside DOM elements like:

```
[e5] input "A1" value="A1"    <-- Name Box
[e12] textbox [editable]       <-- Formula bar / cell input
```

The AI must then translate this into a sequence of actions:
1. `{"tool": "click", "params": {"ref": "e5"}}` -- click Name Box
2. `{"tool": "type", "params": {"ref": "e5", "text": "B1", "pressEnter": true}}` -- type cell ref in Name Box
3. `{"tool": "type", "params": {"text": "My Data"}}` -- type value in now-active cell

The confusion happens because the AI tries to combine steps 2 and 3, or skips step 1 entirely, leading to typing "b1" directly into whatever cell is active (as if "b1" is the data value, not a navigation command).

### What DOM Elements the AI Sees on Google Sheets

The compact snapshot shows elements picked up by the standard DOM query:
- **Name Box** (`#t-name-box`): An `input` element showing current cell ref (e.g., "A1")
- **Formula bar** (`.cell-input`, `#t-formula-bar-input`): A contenteditable div or input
- **Toolbar buttons**: Bold, italic, font size, etc.
- **Menu items**: File, Edit, View, Insert, Format, Data, Tools
- **Sheet tabs**: Sheet1, Sheet2, etc.
- **Canvas grid container** (`#waffle-grid-container`): Not interactive via DOM

The AI does NOT see individual cells. It can only interact with the Name Box input and the formula bar. The Name Box is the gateway to cell selection.

## Root Cause Analysis

### Primary Root Cause: Ambiguous Step Separation in Prompts

The bug has three contributing factors:

**Factor 1: The `multitab` TASK_PROMPTS Sheets section is too terse**

```
WRITING TO GOOGLE SHEETS:
- Google Sheets uses a CANVAS-BASED GRID. You CANNOT click individual cells.
- Use the Name Box (top-left, showing cell reference like "A1") to navigate: click it, type cell reference (e.g., "A1"), press Enter.
- Then type the cell value. Press Tab to move right, Enter to move down.
```

Only 3 lines. No explicit warning about NOT typing cell references as values. No mention that "type cell reference" targets the Name Box element specifically vs "type cell value" which goes to the active cell (no ref needed because typeWithKeys sends keystrokes to whatever is focused).

**Factor 2: Site guide guidance lacks a disambiguation warning**

The google-sheets.js guidance describes the workflow correctly but never says: "WARNING: Do NOT type cell references (like 'B1') into a cell as a value. Cell references should ONLY be typed into the Name Box (#t-name-box) for navigation. If you want to put data in cell B1, first navigate to B1 via the Name Box, THEN type the actual data value."

**Factor 3: The `type` tool on Sheets has no ref -- keyboard emulation goes to active element**

When the AI types on Google Sheets with `canvasEditor = true`, the type handler uses `typeWithKeys` which sends keyboard events to whatever element has focus. If the Name Box has focus, the text goes into the Name Box. If a cell is being edited, the text goes into the cell. The AI must understand this distinction to type the RIGHT text into the RIGHT element at the RIGHT time.

### Secondary Factor: No Sheets-Specific Validation

Unlike the batch action fix (Phase 14.1) which added an execution-level safety check, there is no execution-level check that detects when a cell reference string (like "b1", "c1", "A2") is being typed into a cell rather than the Name Box. This is harder to detect at the execution level because the extension cannot easily distinguish between a user wanting to type the literal text "b1" into a cell vs navigating to cell B1.

## Proposed Fix Strategy

### Fix 1: Enhanced Site Guide Guidance (google-sheets.js)

Add an explicit disambiguation section to the guidance text with a CRITICAL WARNING that separates Name Box navigation from cell value entry. Use emphatic language the AI will weight highly.

### Fix 2: Enhanced Multitab TASK_PROMPTS Sheets Section

Expand the "WRITING TO GOOGLE SHEETS" section from 3 lines to a structured step-by-step procedure that mirrors the site guide but is embedded in the task type prompt (which is injected even without a URL-matched site guide).

### Fix 3: Sheets-Aware Continuation Prompt

When the current URL matches a Google Sheets pattern during `buildMinimalUpdate`, inject a brief Sheets-specific reminder into the minimal update. This ensures continuation iterations preserve the critical Name Box vs cell value distinction. This can be a compact 3-4 line addition appended to the update string.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Detecting cell ref typed as value | Runtime regex check on type params | Prompt-level clear instructions | False positive risk is high ("b1" could be legitimate data); prompt fix addresses root cause |
| Execution-level Name Box state tracking | Custom state machine tracking Name Box focus | Enhanced AI prompt guidance | The AI already receives the DOM snapshot showing which element has focus; it just needs clearer instructions |

**Key insight:** This is a prompt engineering problem, not a code execution problem. The `type` action handler correctly routes text via `typeWithKeys` on Sheets. The problem is the AI choosing the WRONG text to type at each step.

## Common Pitfalls

### Pitfall 1: Over-Engineered Execution-Level Fix
**What goes wrong:** Adding runtime checks for cell reference patterns in typed text creates false positives for legitimate data containing cell-reference-like strings (e.g., "See section B1 of the manual").
**How to avoid:** Keep the fix at the prompt level. The AI is the right layer to make the semantic decision about what text goes where.

### Pitfall 2: Prompt Too Long / Token Bloat
**What goes wrong:** Adding detailed Sheets instructions to multiple prompt locations creates redundancy that pushes system prompt over token budgets.
**How to avoid:** Be surgical. The site guide guidance already has the right info -- it just needs a disambiguation WARNING prepended. The multitab TASK_PROMPTS section needs expansion but should stay under 15 lines. The continuation prompt addendum should be 3-4 lines max.

### Pitfall 3: Not Addressing Continuation Iterations
**What goes wrong:** Fixing only the first-iteration prompt leaves the bug present on iteration 2+ where the minimal prompt lacks all Sheets guidance.
**How to avoid:** The multi-turn conversation carries the first iteration's system prompt in history. However, adding a brief Sheets reminder to the minimal update when on a Sheets URL reinforces the correct behavior. This is the same pattern used for stuck detection and search result navigation.

### Pitfall 4: Breaking fillSheetData Path
**What goes wrong:** Modifications to the Sheets guidance accidentally interfere with the fillSheetData clipboard path used by the career search workflow.
**How to avoid:** The fillSheetData path has its own dedicated directive (`GOOGLE SHEETS DATA ENTRY SESSION`) injected via `context.sheetsData`. The manual interaction fix should NOT modify the sheetsData directive. The fix targets general Sheets interaction (no sheetsData context), not the orchestrated data entry path.

### Pitfall 5: Inconsistent Guidance Between Site Guide and Task Prompt
**What goes wrong:** The site guide says one thing, the task prompt says another, the AI gets confused about which instructions to follow.
**How to avoid:** Keep the language consistent. The site guide provides the detailed reference; the task prompt provides the high-level procedure; the continuation reminder provides the critical rule. All three should use the same terminology: "Name Box" for navigation, "type value" for data entry.

## Open Questions

1. **Should the Productivity Tools category guidance be enhanced too?**
   - What we know: The `_shared.js` category guidance mentions canvas-based apps and Tab/Enter data entry, but does not include Name Box disambiguation
   - What's unclear: Whether adding it at the category level adds value vs the site guide level
   - Recommendation: Keep changes at the site guide level (google-sheets.js) since the disambiguation is Sheets-specific (Docs does not have a Name Box)

2. **Should continuation iterations inject the full site guide or just a compact reminder?**
   - What we know: The multi-turn conversation already has the first iteration's system prompt in history. Adding the full site guide again would be redundant and token-wasteful
   - What's unclear: Whether a 3-4 line reminder is sufficient or if the AI needs more context
   - Recommendation: Start with a compact reminder (similar to the stuck warning pattern). If insufficient, escalate to injecting the guidance section on every iteration when on Sheets.

3. **Does this affect Google Docs canvas interactions?**
   - What we know: Google Docs has its own dedicated multi-paragraph guidance in the multitab TASK_PROMPTS. It uses kix-page-column elements and a fundamentally different interaction model (no Name Box, no cell references)
   - Recommendation: No. This fix is Sheets-specific. Google Docs interactions are not affected.
