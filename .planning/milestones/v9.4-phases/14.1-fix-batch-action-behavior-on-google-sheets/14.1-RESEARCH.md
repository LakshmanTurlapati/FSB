# Phase 14.1: Fix Batch Action Behavior on Google Sheets - Research

**Researched:** 2026-02-24
**Domain:** AI prompt engineering + batch execution engine -- preventing incorrect batch typing on canvas-based grid applications
**Confidence:** HIGH

## Summary

Phase 14 introduced AI-declared batch actions (`batchActions` array) to accelerate multi-action pages. This works excellently for standard DOM-based form fields where each `type` action targets a distinct DOM input element via ref. However, on Google Sheets the grid is **canvas-based** -- individual cells are NOT DOM elements. The AI cannot target cell A1, B1, C1 as separate refs because they don't exist as separate elements in the DOM snapshot. When the AI batches multiple `type` actions for different cells, all values get typed into whichever cell happens to be active, causing concatenation ("999.99random test data sheet" in one cell instead of separate values in separate cells).

The root cause is a semantic mismatch: the AI's batch instructions say "fill multiple form fields" as a valid batch pattern, and the AI incorrectly treats Google Sheets cells as independent form fields. In reality, Sheets data entry requires a **navigation step between each value** -- click Name Box, type cell reference, press Enter, THEN type the value. Each cell entry is a multi-step dependent sequence, not an independent action. The existing Google Sheets site guide (`google-sheets.js`) already documents this Name Box navigation pattern and the canvas-based grid limitation, but the BATCH_ACTION_INSTRUCTIONS constant has no awareness of this constraint.

The fix requires three coordinated changes: (1) add Google Sheets/canvas-app warnings to BATCH_ACTION_INSTRUCTIONS so the AI knows NOT to batch cell-level type actions, (2) add URL-based batch suppression in `executeBatchActions` as a safety net for when the AI ignores the prompt, and (3) reinforce the Tab/Enter sequential entry pattern in the batch context so the AI uses the correct single-action flow for Sheets data entry.

**Primary recommendation:** Add "WHEN NOT TO BATCH" entries specific to Google Sheets and canvas-based applications in BATCH_ACTION_INSTRUCTIONS, plus a URL-pattern guard in executeBatchActions that logs a warning and converts problematic type-heavy batches to single-action flow on Sheets URLs.

### Pattern 1: Prompt-Level Batch Suppression for Canvas Apps

**What:** Add Google Sheets-specific entries to the "WHEN NOT TO BATCH" section of BATCH_ACTION_INSTRUCTIONS so the AI never creates type-action batches for Sheets cells.
**When to use:** Always -- this text is injected into every system prompt.

```javascript
// In ai/ai-integration.js, BATCH_ACTION_INSTRUCTIONS constant
// Add to the "WHEN NOT TO BATCH" section:

const BATCH_ACTION_INSTRUCTIONS = `
=== ACTION BATCHING (PERFORMANCE OPTIMIZATION) ===

When multiple actions target the SAME page and don't depend on each other's results:
- Return them in a "batchActions" array instead of "actions"
- Maximum 5-8 actions per batch
- Actions execute sequentially with stability checks between each
- Navigation-triggering actions (click links, navigate, searchGoogle) MUST be LAST
- If an action fails, remaining actions are skipped
- The existing "actions" array still works for single actions

WHEN TO BATCH:
- Filling multiple form fields: type into field1, type into field2, type into field3
- Selecting multiple checkboxes or radio buttons
- Click + type sequences (click input, then type)
- Multiple data extraction calls (getText on several elements)
- Scroll + extract patterns (scroll down, then getText)

WHEN NOT TO BATCH:
- When the next action depends on the result of the previous one
- When any action might trigger page navigation (put it last or don't batch)
- When you need to verify an action's effect before deciding the next step
- GOOGLE SHEETS / CANVAS-BASED APPS: NEVER batch multiple type actions for different cells. Google Sheets cells are NOT DOM elements -- they are rendered on a canvas. Each cell entry requires Name Box navigation FIRST (click Name Box, type cell ref, press Enter), then type the value. Use single actions with Tab/Enter flow instead: type value -> press Tab (next column) -> type value -> press Tab -> ... -> press Enter (next row). Batching type actions will concatenate all values into the active cell.
- ANY application where typing targets are not independent DOM input elements

Example batch response:
{
  "reasoning": "Three form fields to fill on the same page, no dependencies between them",
  "batchActions": [
    {"tool": "type", "params": {"ref": "e3", "text": "John Doe"}},
    {"tool": "type", "params": {"ref": "e5", "text": "john@example.com"}},
    {"tool": "type", "params": {"ref": "e7", "text": "Software Engineer"}},
    {"tool": "click", "params": {"ref": "e9"}}
  ],
  "taskComplete": false
}`;
```

### Pattern 2: URL-Based Batch Safety Check in executeBatchActions

**What:** At the top of `executeBatchActions`, detect if the current tab is on a Google Sheets URL. If so, and the batch contains 2+ type actions, log a warning and convert to single-action execution (return early with a signal to fall back to the regular action loop).
**When to use:** Every batch execution as a safety net.

```javascript
// In background.js, executeBatchActions() function, after line 5892 (const actions = ...)
// Add URL-based safety check

async function executeBatchActions(batchActions, session, tabId) {
  const actions = batchActions.slice(0, MAX_BATCH_SIZE);

  // SAFETY: Detect Google Sheets URLs and suppress type-heavy batches
  // Google Sheets uses a canvas-based grid where cells are NOT DOM elements.
  // Batching multiple type actions will concatenate values into the active cell
  // instead of typing into separate cells.
  try {
    const tab = await chrome.tabs.get(tabId);
    const currentUrl = tab?.url || session?.lastUrl || '';
    const isSheetsUrl = /docs\.google\.com\/spreadsheets/i.test(currentUrl) ||
                        /sheets\.google\.com/i.test(currentUrl);

    if (isSheetsUrl) {
      const typeActionCount = actions.filter(a => a.tool === 'type').length;
      if (typeActionCount >= 2) {
        automationLogger.warn('Suppressing type-heavy batch on Google Sheets -- cells are canvas-based, not DOM elements', {
          sessionId: session?.sessionId,
          typeActionCount,
          totalActions: actions.length,
          tools: actions.map(a => a.tool)
        });
        // Return a signal that tells the caller to fall back to single-action execution
        return {
          batched: false,
          suppressed: true,
          reason: 'google_sheets_canvas_grid',
          actions: actions, // Pass back for single-action execution
          results: [],
          totalCount: actions.length,
          successCount: 0,
          failedAt: -1,
          duration: 0,
          skippedActions: actions
        };
      }
    }
  } catch (urlCheckError) {
    // URL check is a safety measure -- don't block batch execution if it fails
    automationLogger.debug('URL check for batch suppression failed', { error: urlCheckError.message });
  }

  // ... rest of existing executeBatchActions logic
```

### Pattern 3: Handle Suppressed Batch in startAutomationLoop

**What:** When executeBatchActions returns `suppressed: true`, execute the actions individually through the normal single-action path instead of dropping them.
**When to use:** In the startAutomationLoop batch handling block.

```javascript
// In background.js, startAutomationLoop, after executeBatchActions call (line 8967)

if (aiResponse.batchActions && aiResponse.batchActions.length > 0) {
  const batchResult = await executeBatchActions(aiResponse.batchActions, session, session.tabId);

  // If batch was suppressed (e.g., Google Sheets canvas grid), fall back to single-action
  if (batchResult.suppressed) {
    automationLogger.info('Batch suppressed, falling back to first action only', {
      sessionId,
      reason: batchResult.reason,
      firstAction: aiResponse.batchActions[0]?.tool
    });
    // Execute only the FIRST action from the batch via the normal single-action path
    // The AI will get fresh DOM context and can plan the next action correctly
    aiResponse.actions = [aiResponse.batchActions[0]];
    aiResponse.batchActions = null;
    // Fall through to the else-if block below which handles regular actions
  } else {
    // Normal batch completion logging (existing code)
    automationLogger.info('Batch execution complete', { ... });
    // ...
  }
}
```

### Anti-Patterns to Avoid

- **Blanket batch blocking on Sheets URLs:** Not all batch actions are problematic on Sheets. A batch of `getText` calls for formula bar verification, or a batch of `click` + `keyPress` for formatting, is perfectly safe. Only type-heavy batches targeting cell content cause the concatenation bug. Filter specifically for 2+ type actions.

- **Relying solely on prompt guidance:** The AI may ignore the "WHEN NOT TO BATCH" instructions, especially under pressure to be efficient or with different AI models. The URL-based safety check in executeBatchActions is essential as a fallback. Defense in depth.

- **Breaking the fillSheetData path:** The existing `fillSheetData` tool in background.js (line 7688) uses clipboard paste for bulk data entry and completely bypasses the batch action system. This path works correctly and must not be affected by changes. Phase 14.1 only affects the case where the AI uses the regular automation loop (non-career tasks like "open Google Sheet, fill random data").

- **Blocking single type actions on Sheets:** A single type action on Sheets is valid (typing a value into the currently selected cell). Only batching multiple type actions for different cells is the problem. The safety check threshold should be 2+ type actions, not 1.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Canvas app detection | Custom canvas rendering detection logic | URL pattern matching (docs.google.com/spreadsheets) | The problem is specific to Google Sheets (and potentially Google Docs). URL matching is deterministic and 100% reliable. Canvas detection at the DOM level would be fragile and miss the point. |
| Sheets cell navigation | Custom cell coordinate tracking system | Existing Name Box navigation workflow in google-sheets.js | The site guide already documents the Name Box + Tab/Enter pattern. The AI just needs to be told NOT to batch and to follow the documented workflow instead. |
| Batch action filtering | Complex action dependency analysis engine | Simple type-action count threshold | The problem is specifically about type actions on canvas grids. A count of type actions in the batch plus a URL check is sufficient. No need for a general-purpose dependency analyzer. |

**Key insight:** This is fundamentally a prompt engineering + safety guard problem, not a systems engineering problem. The AI already knows HOW to enter data into Google Sheets correctly (via the site guide). It just doesn't know that batch actions are incompatible with canvas-based grids. The fix is teaching it this constraint and adding a safety net.

## Common Pitfalls

### Pitfall 1: AI Still Batching Type Actions Despite Prompt Warning

**What goes wrong:** The AI reads the "WHEN NOT TO BATCH" warning about Google Sheets but still returns batchActions with multiple type entries because the batch-encouraging examples (filling form fields) look similar to filling spreadsheet cells.
**Why it happens:** AI models weigh positive examples ("WHEN TO BATCH: filling multiple form fields") more heavily than negative warnings, especially when the current page context looks like a "form" with multiple input targets.
**How to avoid:**
1. Make the Google Sheets warning prominent and specific -- not just a bullet point but a capitalized, emphatic statement
2. Add a concrete counter-example showing the WRONG way (batching cells) vs the RIGHT way (single action with Tab flow)
3. The URL-based safety check in executeBatchActions catches this case even when the prompt fails
**Warning signs:** Logs showing "Suppressing type-heavy batch on Google Sheets" -- indicates the prompt didn't prevent the batch and the safety check caught it.

### Pitfall 2: Suppressed Batch Losing All Actions

**What goes wrong:** When the safety check suppresses a batch, the code drops all actions and the iteration produces no result, wasting a full AI round-trip.
**Why it happens:** The suppression logic returns early without executing anything, and the startAutomationLoop doesn't know to retry with single actions.
**How to avoid:**
1. When batch is suppressed, execute the FIRST action from the batch as a single action
2. The AI gets fresh DOM context on the next iteration and can plan the remaining actions correctly
3. The `suppressed: true` return value signals the caller to handle gracefully
**Warning signs:** Iterations with 0 actions executed when the AI clearly intended to act.

### Pitfall 3: Breaking the Existing fillSheetData Clipboard Path

**What goes wrong:** Changes to batch handling interfere with the clipboard-paste based data entry used by career search workflows (Phase 12's fillSheetData tool).
**Why it happens:** fillSheetData is a background-handled tool that uses keyboard emulator directly. If the batch suppression incorrectly catches fillSheetData calls inside a batch.
**How to avoid:**
1. fillSheetData is NOT a type action -- it's a backgroundDataTool. The type-action count check won't catch it.
2. The existing routing (`backgroundDataTools.includes(action.tool)`) handles fillSheetData before any batch-level checks.
3. Verify that fillSheetData still works after changes by testing a career search + Sheets output workflow.
**Warning signs:** Career search workflows failing at the Sheets data entry phase.

### Pitfall 4: Overly Broad Canvas-App Detection

**What goes wrong:** The batch suppression triggers on URLs that happen to match the pattern but are not the Sheets editing interface (e.g., Sheets homepage at sheets.google.com, Sheets embed viewer, etc.).
**Why it happens:** The regex `/docs\.google\.com\/spreadsheets/i` matches both the editing interface and the template gallery/file browser.
**How to avoid:**
1. The batch suppression only fires for type-heavy batches (2+ type actions). On the homepage/gallery, the AI wouldn't be typing cell values anyway.
2. If needed, tighten the regex to match only edit URLs: `/docs\.google\.com\/spreadsheets\/d\//i` (must have a document ID).
3. The suppression is a safety check, not a block -- it logs a warning and falls back gracefully.
**Warning signs:** False positive suppressions on non-edit Sheets pages.

### Pitfall 5: Prompt Getting Too Long

**What goes wrong:** Adding detailed Google Sheets batch warnings to BATCH_ACTION_INSTRUCTIONS inflates the system prompt, increasing token usage and cost.
**Why it happens:** The BATCH_ACTION_INSTRUCTIONS is injected into EVERY system prompt, not just Sheets-specific ones.
**How to avoid:**
1. Keep the warning concise -- one strong paragraph, not a detailed tutorial
2. The Google Sheets site guide already has the full documentation; the batch warning just needs to say "don't batch type actions on canvas apps, use Tab/Enter flow instead"
3. Monitor token usage after deployment to verify impact is minimal (should be <50 additional tokens)
**Warning signs:** Noticeable increase in API costs after deployment.

## Open Questions

1. **Should other canvas-based apps get the same treatment?**
   - What we know: Google Docs also uses canvas rendering for text. Google Slides is canvas-based for slide content.
   - What's unclear: Whether the AI would attempt to batch type actions on Docs/Slides in practice. The current problem is specifically observed on Sheets.
   - Recommendation: **Start with Sheets only.** The URL regex can be expanded later if the same concatenation bug appears on other Google Workspace apps. The prompt warning uses the general term "canvas-based apps" to cover the concept.

2. **Should the batch suppression return ONLY the first action, or ALL actions as single-action sequence?**
   - What we know: Returning all actions as sequential single-actions would require restructuring the action execution loop. Returning only the first action is simpler and lets the AI re-plan with fresh DOM state.
   - What's unclear: Whether losing the remaining planned actions wastes significant time (the AI has to re-derive them).
   - Recommendation: **Return only the first action.** On Sheets, each cell entry requires verification via formula bar anyway. The AI needs fresh DOM state between cells. Re-planning is the correct behavior.

3. **Should the safety check examine action params for ref duplication?**
   - What we know: On Sheets, all type actions typically target the same element (the formula bar input, which is the only text input on the page). If 2+ type actions share the same ref, that's a strong signal of the canvas-concatenation bug.
   - What's unclear: Whether the AI always uses the same ref for Sheets cell typing. It might use different refs if it sees multiple input elements in the DOM snapshot.
   - Recommendation: **Use type-action count as the primary heuristic, not ref analysis.** Simpler, more robust, and the false positive rate on non-Sheets pages is zero (the URL check gates it).
