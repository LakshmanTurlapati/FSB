Implementation:
1. Extract the "at [company list]" portion from the end of the task string using `/\bat\s+(.+?)$/i`
2. Split on comma-and, comma, or standalone "and": `.split(/,\s*(?:and\s+)?|\s+and\s+/i)`
3. Trim and filter empty strings
4. Filter out non-company words using the same nonCompanyWords list as extractCompanyFromTask (find, search, look, get, show, list, software, engineering, remote, senior, junior, jobs, careers, openings, positions, roles, etc.)

6. Return the array if length >= 2, otherwise return null (single-company tasks fall through to the existing extractCompanyFromTask path)

3. Add a new `handleBackgroundAction(action, session)` async function (separate from handleMultiTabAction) with switch cases:

   - `storeJobData`: Receives `{company: string, jobs: array}` in action.params. Reads 'fsbJobAccumulator' from chrome.storage.local.   <files>background.js</files>
  <action>
Add three utility functions to background.js (place near the handleBackgroundAction function):

**1. normalizeApplyUrl(url):**
- Parse with `new URL(url)` in try/catch
- Remove common tracking params: utm_source, utm_medium, utm_campaign, utm_content, utm_term, source, ref, src, returnUrl, from
- Remove trailing slashes from pathname
- Return normalized URL string
- On parse failure, return the raw url string

**2. deduplicateJobs(jobs):**
- Per locked decision: duplicate = same apply link (URL match), silently drop duplicates
- Create a Map keyed by normalizedApplyUrl
- Jobs without apply links or with "not available" apply links are NEVER duplicates -- always keep them
- For jobs with apply links, first occurrence wins (drop subsequent duplicates)
- Return the deduplicated array

**3. checkAccumulatorRelevance(existingAccumulator, newTaskString):**
- Per locked decision: new search with old data is context-dependent. If old data is relevant (same role keywords), keep it. If different role, clear.
- Extract role keywords from newTaskString (words before "at" in the task string, excluding common verbs like find/search/look/get)
- Compare against existingAccumulator.searchQuery using simple keyword overlap
- If overlap >= 50% of keywords, return 'keep' (old data is relevant)
- If overlap < 50%, return 'clear' (different search, clear accumulator)
- If no existing accumulator or no searchQuery, return 'clear'

This function is called at the start of a new multi-site search to decide whether to clear or extend the existing accumulator.
