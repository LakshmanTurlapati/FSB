<task type="auto">
  <name>Task 2: Reset session startTime on phase transitions to prevent premature timeout</name>
  <files>background.js</files>
  <action>
The session timeout check at line 7353 uses `const MAX_SESSION_DURATION = 5 * 60 * 1000` and compares against `session.startTime`. The startTime is set once at session creation and never updated. A multi-site pipeline (2 companies x ~2min each + Sheets entry ~3min + formatting ~1min) easily exceeds 5 minutes.

**Fix approach:** Reset `session.startTime` at each major phase transition so the 5-minute clock restarts. This is simpler and more robust than computing a dynamic budget (which would require predicting how long each phase takes).

**In launchNextCompanySearch (same block as Task 1, right after the clearConversationHistory call):**
```javascript
// Reset session timer so the 5-minute timeout restarts for this company (Gap 3 fix)
session.startTime = Date.now();
```

**In startSheetsDataEntry (same block as Task 1, right after the clearConversationHistory call):**
```javascript
// Reset session timer for Sheets entry phase (Gap 3 fix)

```

**In startSheetsFormatting (same block as Task 1, right after the clearConversationHistory call):**
```javascript
// Reset session timer for formatting phase (Gap 3 fix)
session.startTime = Date.now();
```

This gives each phase its own 5-minute budget. A 3-company search + Sheets + formatting gets: 5min + 5min + 5min + 5min + 5min = 25 minutes total budget, which is more than enough while still protecting against truly stuck sessions.

IMPORTANT: Do NOT change the MAX_SESSION_DURATION constant itself. The 5-minute per-phase budget is appropriate -- the bug is that the timer was never reset, not that 5 minutes is too short for a single phase.
