<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP server process -> Extension SW | First and only sendAndWait of agent:register from this plan's surface. Payload is `{}` so no untrusted data crosses the boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-238-01 | Spoofing | AgentScope.ensure() recipient | accept | The bridge layer authenticates the WebSocket pairing at handshake (out of P238 scope). AgentScope sends only an empty payload; nothing to spoof here. Phase 240 owns end-to-end agent identity enforcement. |
| T-238-02 | Tampering | MCPMessageType union widening | accept | Type-only change. Runtime accepts strings opaquely; widening cannot enable a malicious caller because the type is compile-time only. |
| T-238-03 | DoS | Repeated agent:register calls if `this.pending` clearing leaks | mitigate | The cached-promise pattern (D-03) guarantees one in-flight register; concurrent first callers share the promise. Test 2 actively asserts exactly one register call across 5 concurrent ensure() invocations. The `finally { this.pending = null; }` clear AFTER `this.agentId` is set means subsequent calls hit the fast path before the clear. |
| T-238-04 | Information disclosure | console.error logging of agent_id | accept | The minted id is logged with a SHORT prefix (12 chars or registry-supplied agentIdShort), never the full id. Going to stderr — process-local only, not shipped to remote logs. The full id is the secret-equivalent for ownership in P240; truncation in P238 logs is the safe posture. |
| T-238-05 | Elevation of privilege | AgentScope grants no capability in P238 | accept | Phase 240 is the mitigation owner. P238 control is "structural setup only, no trust boundary yet" — AgentScope holds an id but the extension explicitly ignores the field. |

**Phase 240 enforcement note:** Every threat in this register that defers to P240 does so because P238 is purely additive plumbing. Phase 240 will introduce dispatch-side ownership_token verification, cross-agent reject, and incognito reject — that is where the agent_id becomes a security-relevant identifier.
</threat_model>
