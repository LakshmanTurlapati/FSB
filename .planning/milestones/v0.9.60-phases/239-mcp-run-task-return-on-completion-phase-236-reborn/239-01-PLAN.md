<open_question_resolution>
**RESEARCH.md Open Question 1 (bridge.ts pendingRequests survival on disconnect) is RESOLVED by this planner:**

`mcp/src/bridge.ts:131-136` (`disconnect()` method) explicitly **rejects all pendingRequests on disconnect** with `pending.reject(new Error('Bridge disconnected'))` and clears the map. This means:

- The extension CANNOT synthesize a post-reconnect `mcp:result` to settle a pre-disconnect sendAndWait — the server-side Promise has already rejected.
- **SW-wake settle logic MUST live on the server side** (Plan 03's responsibility): the server catches the `Bridge disconnected` rejection at the `bridge.sendAndWait` call site in `autopilot.ts`, and resolves the run_task tool with `{success: ?, sw_evicted: true, partial_state: <from chrome.storage.session via a new bridge message that asks the extension to look up the snapshot post-reconnect>, last_heartbeat_at}`.
- This decision is locked here so Plan 03 can build on it without re-litigating.

This Open Question resolution is what motivates the cross-plan dependency: Plan 03 depends on Plan 02's `mcp-task-store.js` because the server-side reconciliation handler asks the extension for the persisted snapshot on reconnect.
</open_question_resolution>
