<objective>
Fix the recovery-tool protocol bug exposed when the active tab is `chrome://newtab/` and the MCP agent owns zero tabs.

The implementation must let bootstrap-safe tools (`open_tab`, `navigate`, `switch_tab`, `list_tabs`) recover from a restricted active tab without weakening ownership enforcement for normal action/read tools. It must also make MCP error reporting distinguish ownership/resolver/protocol failures from true page-navigation failures.
</objective>

<interfaces>
Current broken path:
```javascript
// extension/ws/mcp-bridge-client.js
async _handleExecuteAction(payload) {
  const resolved = await globalThis.resolveAgentTabOrError(agentId, params, this);
  if (resolved.success === false) return resolved;
  // ...
}
```

For `open_tab`, this is wrong because the tool creates the first owned tab. A zero-tab agent gets `NO_OWNED_TAB` before `handleOpenTabRoute` can run.

For `navigate` and `switch_tab`, this is incomplete because a restricted active page can still be navigated/switched via `chrome.tabs` APIs. The tool must guard ownership before mutation/focus transfer, but it must not require content-script attachability.

Existing gate shape:
```javascript
function checkOwnershipGate({ tool, params, payload }) {
  const tabId = _resolveTabIdForGate(tool, params, payload);
  if (tabId === null) return null;
  if (!reg.isOwnedBy(tabId, agentId, ownershipToken)) {
    return { success: false, code: 'TAB_NOT_OWNED', ownerAgentId, requestedTabId: tabId, requestingAgentId: agentId };
  }
}
```

MCP mapper gap:
```typescript
const errorCode = typeof fsbResult?.errorCode === 'string' ? fsbResult.errorCode : '';
if (FSB_ERROR_MESSAGES[errorCode]) return errorCode;
// fsbResult.code is ignored, so code-only protocol errors fall to Page navigation.
```

<risks>
- The most important safety risk is accidental tab hijack. `navigate` and `switch_tab` must only claim unowned tabs or same-agent tabs; other-agent tabs must reject before mutation or focus transfer.
- The highest regression risk is broadening the resolver bypass too far. Only the explicit bootstrap-safe tools get special handling.
- `open_tab` currently receives change-report wrapping only because `_handleExecuteAction` invents a pre-action tab via the resolver. Removing that pre-action target is correct; any report should be creator-aware or omitted without blocking success.
</risks>

Phase 247 is done when a zero-owned registered agent can recover from `chrome://newtab/` using `open_tab`, `navigate`, or `switch_tab`, while content tools still require an owned/attachable target and protocol errors no longer display as generic `Page navigation`.
