<objective>
Wire the Plan 01 registry primitives into Chrome's tab lifecycle and the MCP bridge so the phase's user-observable behaviors actually fire:

1. **Forced-tab pooling (POOL-03):** A new chrome.tabs.onCreated listener in background.js inspects `openerTabId`. If the opener is owned by an agent A, the new tab is bound to A via `bindTab(A, newTabId, { forced: true })`. New tabs without openers are intentionally left unowned.
2. **Bridge connection_id minting (LOCK-02):** `_ws.onopen` mints a fresh connection_id (randomUUID); cancels any in-flight staged release for the previous connection_id. `_ws.onclose` stages release for all agents stamped with the current connection_id, with 10s grace. Persisted across SW eviction via Plan 01's `stagedReleases` envelope field.
3. **AGENT_CAP_REACHED surfacing (POOL-01 closure):** `handleAgentRegisterRoute` branches on the new error-return shape from `registerAgent` and surfaces typed `{ success: false, code: 'AGENT_CAP_REACHED', cap, active }` upstream. Plan 01 added the registry-side; this plan adds the dispatcher-side branch.
4. **connection_id threading (LOCK-02 wire):** The MCP server's `agent-scope.ts` captures `connectionId` from the `agent:register` response (alongside the existing `ownershipTokens`). Outbound `bridge.sendAndWait` payloads thread it as a sibling of `agentId` and `ownershipToken`. Extension's `handleAgentRegisterRoute` calls `reg.stampConnectionId(agentId, connectionId)` so on bridge onclose the registry can find which agents to stage.

Purpose: Wave 2 wiring. Plan 01 provided the contract; Plan 02 makes it fire on actual events. Parallel with Plan 03 (settings UI) — no file overlap (Plan 03 touches extension/ui/control_panel.html + extension/ui/control_panel.js / options.js).
