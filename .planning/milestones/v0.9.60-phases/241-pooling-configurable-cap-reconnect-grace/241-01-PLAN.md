<objective>
Extend the Phase 237/240 AgentRegistry in-place with the cap-enforcement and pool-lifecycle primitives that Plan 02 (bridge wiring) and Plan 03 (settings UI) depend on. This plan ships ZERO observable behavior on its own outside the registry — it is the foundation. After this plan, the registry can:

1. Reject the (N+1)th distinct agent claim with typed `AGENT_CAP_REACHED { code, cap, active }` (POOL-02).
2. Read/write the cap from `chrome.storage.local` under `fsbAgentCap`, hydrate fresh on SW wake, and apply changes via `chrome.storage.onChanged` (POOL-05).
3. Release the owning agent record only when its tab pool drains to zero (POOL-04, LOCK-03 closure for the registry side).
4. Look up an agent by tabId synchronously (`findAgentByTabId`) — needed by Plan 02's `chrome.tabs.onCreated` handler.
5. Stage / cancel / fire-on-expiry / hydrate-recover staged releases keyed by `connection_id` (LOCK-02 registry side; bridge wiring lives in Plan 02).
6. Stamp a `connectionId` field on agent records (so Plan 02's bridge wiring can route close events).

Purpose: Foundation for the whole phase. Plan 02 calls these new APIs (`stageReleaseByConnectionId`, `cancelStagedRelease`, `findAgentByTabId`, `stampConnectionId`); Plan 03 writes the cap that this plan reads. Wave 1 isolates this work so the bridge / UI plans run in parallel against a stable contract.

Output:
- Extended `extension/utils/agent-registry.js` (additive — Phase 237/240 contracts unchanged).
- Three new plain-Node assert test files: `tests/agent-cap.test.js`, `tests/agent-cap-storage.test.js`, `tests/agent-pool-shrink.test.js`.
- All Wave 0 test scaffolds for these three already exist (Validation §Wave 0 Requirements) — this plan fills them in with assertions.

Open Questions resolved in this plan:
- **Q2 (stagedReleases storage shape):** RESOLVED. Persisted shape is `{ [connectionId]: { deadline: <ms epoch>, agentIds: [...snapshot at stage time...] } }`. The `agentIds` array is captured at stage time so agents claimed AFTER the disconnect (e.g., a fresh bridge connection's agents) are NOT swept up by an older grace timer. `timeoutId` is intentionally NOT persisted (DOM timer ids do not survive SW eviction). Plan 02's hydrate test exercises this snapshot semantics.
- **Q3 (setCap diagnostic when M > newCap):** RESOLVED YES. `setCap` emits one-shot LOG-04 `agent-cap-lowered-grandfathered { previousCap, newCap, activeAtChange }` for operator observability. No eviction; purely diagnostic.

Decisions implemented (per CONTEXT.md):
- D-02 (cap counts agents not tabs) — registerAgent's cap-check uses `_agents.size`, not pool tab count.
- D-03 (AGENT_CAP_REACHED typed error) — return shape `{ error: 'AGENT_CAP_REACHED', code: 'AGENT_CAP_REACHED', cap, active }`.
- D-04 (LOG-04 emission on cap rejection) — event family `agent-cap-reached`.
- D-05 (cap stored in chrome.storage.local under `fsbAgentCap`).
- D-06 (grandfathered on lower-cap; no eviction).
- D-09 (LOG-04 emission `agent-grace-expired { agentId, connectionId, poolSize }` on stage expiry).
- D-10 (per-tab onRemoved drives pool shrink — extends releaseTab to delete the agent record when pool.size === 0).
- D-13 (no idle timeout — explicit negative; no setInterval / setTimeout / chrome.alarms scheduling for "idle reaping" anywhere in this plan).
